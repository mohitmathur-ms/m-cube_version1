Stage 2 — Strategy → Order (submit)

Inside on_bar, our _check_entries builds and submits via the project's single helper _submit_order:

# core/managed_strategy.py:724-739
def _submit_order(self, side: OrderSide) -> None:
    kwargs = dict(
        instrument_id=self.config.instrument_id,
        order_side=side,
        quantity=self.instrument.make_qty(self.config.trade_size),
        time_in_force=TimeInForce.GTC,
    )
    if self._pending_entry_reason:
        kwargs["tags"] = [self._pending_entry_reason]
    order = self.order_factory.market(**kwargs)
    self.submit_order(order)
    self._pending_entry_reason = None

Important: submit_order(order) does not block on a fill. It hands the command to the engine and returns. The fill confirmation arrives later via the order-lifecycle events in Stage 3.

Calls the project does NOT make: order_factory.limit, .stop_market, .stop_limit, .bracket, submit_order_list, modify_order, cancel_order (individual). Only market orders + GTC.

Happy path for a market order: on_order_initialized → on_order_submitted → on_order_accepted → on_order_filled ★. Of these four, the project hooks only the last one. The order's lifetime in the book between accept and fill is essentially zero for market orders.

Inside on_order_filled the project does (managed_strategy.py:657):

def on_order_filled(self, event) -> None:
    if self._expecting_close_fill:           # this fill is closing an existing position
        self._expecting_close_fill = False
        return
    self.entry_price = float(event.last_px)  # ← fill confirmation: price
    self.highest_profit = 0.0
    self.sl_wait_count = 0
    self._was_trailed = False
    self._reentry_blocked_until_ns = 0
    self._entry_filled_at_ns = self._current_bar_ts_ns
    self._move_sl_fired_this_position = False
    is_buy = event.order_side == OrderSide.BUY
    self.position_side = "LONG" if is_buy else "SHORT"

Stage 4 — Position lifecycle

A fill not only triggers on_order_filled — Nautilus's PortfolioFacade also rolls the fill into a Position object and fires position events. These run after on_order_filled for the same fill.

HandlerEvent classFires whenWe override?
on_position_openedPositionOpenedNet position transitioned flat → non-flat. A new Position object now exists.✗ Not used
on_position_changedPositionChangedAn existing position's quantity changed (scale-in / partial close).✗ Not used
on_position_closedPositionClosedNet position transitioned back to flat. Final realized PnL is on this event.✗ Not used
on_position_eventPositionEvent (base)Catch-all for the three above.⚠ Catch-all, not used
Why we don't override these: the project derives all position state from on_order_filled directly — tracking position_side in a private field (managed_strategy.py:683). The dedicated position events would be a cleaner place to do it (especially for realized PnL), but the current single-strategy-per-engine design makes the inference safe.

Practical takeaways

  1. The single "fill confirmation" handler is on_order_filled(OrderFilled event). The project already overrides it ([managed_strategy.py:657]) and uses event.last_px as the canonical entry price.
  2. For richer ack tracking (e.g. "log every transition" / "alert on rejected orders"), override the catch-all on_order_event(OrderEvent event) once — it fires for every order event in Stage 3. Saves defining 14 separate hooks.
  3. For realized PnL, override on_position_closed(PositionClosed event) instead of recomputing PnL from fills + entry_price. The event payload carries the realized PnL directly.
  4. Failure paths are currently silent. on_order_denied (RiskEngine refused) and on_order_rejected (venue refused) aren't overridden. With RiskEngineConfig(bypass=True) at backtest_runner.py:474 these are unlikely in backtest, but if they fire the strategy never knows — position_side stays None while the user wonders why no trades happen.
