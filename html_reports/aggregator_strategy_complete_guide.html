1. Overview & motivation

We have 1-minute FX bars. Most strategies want 5-minute bars instead, so something has to combine every five 1-min bars into one 5-min bar. Nautilus has a built-in way to do this; you can also write your own. We did both — to compare them side-by-side and prove they agree (or find where they don't).
Technically: The two notebooks demonstrate two ways to produce a 5-MINUTE bar stream from a 1-MINUTE-EXTERNAL source: (a) Nautilus's TimeBarAggregator driven by a composite BarType with aggregation source INTERNAL; (b) a user-space FiveMinAggregator class that buffers Bar objects, computes OHLCV plus derived features (range, ATR(14), EMA(9/21), RSI(14), VWAP, etc.), and emits a custom dataclass. The first notebook only writes CSVs; the second wires both into a real Strategy with bracket SL/TP and produces the standard Nautilus reports (order_fills, positions, account).

2. The data pipeline

Our raw data is one folder per trading day, each with two CSVs — an ASK side (the price someone will sell to you) and a BID side (the price someone will buy from you). Each row is one 1-minute bar. We load all days for the month into a DataFrame, then hand it to Nautilus, which wraps each row into a Bar object.
Technically: Source CSVs live under C:/Users/HP/Desktop/MS/Dataset/FX_yyyy/Fx/EUROUSD/2024/01/<DD>/<DD>.<MM>.<YYYY>_<SIDE>_OHLCV.csv with timestamps in the format DD.MM.YYYY HH:MM:SS.fff GMT+HHMM (Indian Standard Time = UTC+05:30 in our case). After pd.to_datetime(..., utc=True) the index is UTC; Nautilus requires UTC nanoseconds internally (BarDataWrangler.process calls as_utc_index(data)). For display, the notebook converts back to Asia/Kolkata via tz_convert(OUTPUT_TZ) when writing CSV output.

Loading the full month

aggregator_strategy_demo.ipynb · cell-04
def load_month(data_dir: Path, side: str) -> pd.DataFrame:
    """Concatenate every `*_<side>_OHLCV.csv` under `data_dir/<DD>/` into one UTC frame."""
    frames: list[pd.DataFrame] = []
    for day_dir in sorted(data_dir.iterdir()):
        if not day_dir.is_dir():
            continue
        for csv_path in sorted(day_dir.glob(f"*_{side}_OHLCV.csv")):
            frames.append(pd.read_csv(csv_path))
    df = pd.concat(frames, ignore_index=True)
    df["timestamp"] = pd.to_datetime(
        df["timestamp"], format="%d.%m.%Y %H:%M:%S.%f GMT%z", utc=True
    )
    df = df.set_index("timestamp").sort_index()
    df = df[~df.index.duplicated(keep="last")]  # drop day-boundary overlaps
    return df[["open", "high", "low", "close", "volume"]].astype(float)


df_1min_ask = load_month(DATA_DIR, side="ASK")
df_1min_bid = load_month(DATA_DIR, side="BID")
BID + ASK is mandatory for FX, not optional. Nautilus's simulated FX matching engine needs both quote sides to determine a fillable price for any market order. With only ASK loaded, every market entry is rejected with 'no market for EURUSD.FOREX_MS' and the bracket's SL/TP children cascade-reject. The project enforces this everywhere — see core/backtest_runner.py:_pair_bid_ask_bar_type. Strategies still subscribe to just one side (we pick ASK); the other side is only there for the matching engine.

3. Aggregation: two approaches

3.1 Nautilus INTERNAL aggregator

Tell Nautilus the bar type you want is "5-minute bars built internally from 1-minute external bars." Nautilus reads your wish, spawns its own aggregator behind the scenes, and feeds your strategy the 5-min bars as if they were the original input. You write no aggregation code.
Technically: A composite BarType string has the form <id>-<step>-<agg>-<price>-INTERNAL@<step>-<agg>-EXTERNAL. The right-hand side declares the source bar spec (only step + aggregation + source — no price type) and the left-hand side declares the output. When a strategy calls self.subscribe_bars(composite_bar_type), the DataEngine recognises the INTERNAL@…-EXTERNAL shape, instantiates a TimeBarAggregator with a TestClock, sets it to historical mode, and routes the EXTERNAL source bars through it. The aggregator's _pre_process_historical_events advances the test clock to each incoming bar's ts_init and fires any pending timer events inline before applying the bar; the _build_bar path emits the aggregated bar via the message bus back to the strategy's on_bar.

Composite bar-type syntax

StringMeaning
EURUSD.FOREX_MS-5-MINUTE-ASK-INTERNAL@1-MINUTE-EXTERNALTarget: 5-min ASK bars, built internally by Nautilus. Source: 1-min EXTERNAL bars (price type inferred from context).
EURUSD.FOREX_MS-1-MINUTE-ASK-EXTERNALPlain 1-min ASK bars. Strategy receives them as-is.
Gotcha: the right-hand side of @ does NOT include the price type. ...-INTERNAL@1-MINUTE-ASK-EXTERNAL raises ValueError: invalid token: '1-MINUTE' at position 5. Correct form is ...-INTERNAL@1-MINUTE-EXTERNAL.

3.2 Custom FiveMinAggregator

Same idea, but you write the aggregator yourself in plain Python. Each 1-min bar comes in, you stash it in a list. When you notice you've moved into the next 5-minute window, you take the list you've been building, compute open/high/low/close/volume from it (plus extra features like ATR, EMA, RSI), and hand that out as one 5-min bar.
Technically: A pure-Python class with two state buckets: (1) per-window state (_buffer, _window_close) and (2) cross-window state (_prev_close, _ema_s, _ema_l, Wilder-style _avg_gain/_avg_loss/_atr, cumulative VWAP _cum_pv/_cum_v). The bucketing rule is bucket_close = ts.ceil(f"{N}min") — this snaps a timestamp upward to the next N-minute mark, with the special property that timestamps already on the boundary stay put. That implements the left-open (T-N, T] convention and matches Nautilus's default interval_type="left-open". All indicator updates use one-line recurrence forms (new = old + α·(sample − old)) so each emit is O(1) regardless of how many past bars exist.

The whole class

aggregator_strategy_demo.ipynb · cell-08
class FiveMinAggregator:
    """Buffer 1-min bars, emit a `Bar5MinFeatures` at every N-min boundary."""

    def __init__(self, target_minutes=5, ema_short_period=9, ema_long_period=21,
                 rsi_period=14, atr_period=14):
        self.target_minutes = target_minutes
        self._alpha_ema_s = 2.0 / (ema_short_period + 1)
        self._alpha_ema_l = 2.0 / (ema_long_period + 1)
        self._alpha_rsi   = 1.0 / rsi_period
        self._alpha_atr   = 1.0 / atr_period

        # per-window state
        self._buffer: list[Bar] = []
        self._window_close: pd.Timestamp | None = None

        # cross-window state (drives derived features)
        self._prev_close = None
        self._ema_s = None; self._ema_l = None
        self._avg_gain = 0.0; self._avg_loss = 0.0
        self._atr = None
        self._cum_pv = 0.0; self._cum_v = 0.0

    def on_bar(self, bar: Bar) -> Bar5MinFeatures | None:
        ts = pd.Timestamp(bar.ts_event, unit="ns", tz="UTC")
        bucket_close = ts.ceil(f"{self.target_minutes}min")
        emitted = None
        if self._window_close is None:
            self._window_close = bucket_close
        elif bucket_close != self._window_close:
            emitted = self._close_window()          # close prior window
            self._window_close = bucket_close
        self._buffer.append(bar)
        return emitted

    def flush(self) -> Bar5MinFeatures | None:
        """End-of-stream: emit any trailing partial window."""
        if self._buffer:
            return self._close_window()
        return None

Why flush() exists. The aggregator only emits when a new bucket arrives. If the data stream ends mid-window (e.g. data ends at 18:29 but the window closes at 18:30), the 4 partial bars sit in the buffer forever — no future bar to trigger emission. flush() is the explicit "stream is done, emit whatever's there" call. Our strategies call it from on_stop but discard the result (we don't trade on partial windows).

4. The Bar5MinFeatures dataclass

A plain Python record class — like a struct — that holds one 5-minute bar plus all the derived features in one place. The aggregator builds one of these every time a window closes; the strategy or a CSV writer can pull individual fields off it.
Technically: A @dataclass with 20 fields. field_names() returns the column order for csv.DictWriter; to_row() calls dataclasses.asdict for serialisation. Note this is not a Nautilus Data subclass — so it cannot be routed via publish_data / on_data without first inheriting from nautilus_trader.core.data.Data and exposing ts_event/ts_init properties (see §11).

5. Same vs different between the two aggregators

The two aggregators produce the same numbers — same OHLCV bar by bar. But they hand those bars to the strategy at slightly different moments in time. Nautilus delivers each 5-min bar the moment its closing 1-min bar arrives; our custom one delivers it one minute later, when the next 1-min bar shows up. That one-minute lag is invisible if you only look at the bars, but it changes when orders get placed.

The 1-minute delivery lag is the only material difference, and it only matters when the strategy submits orders or otherwise acts at bar-arrival time. For pure data export (the first notebook) you see identical CSV content for every overlapping row. For trading (the second notebook), this lag cascades through order-fill prices, SL/TP triggers, and position-state alignment — see §10.

6. The trading strategy

On every 5-minute bar we look at two moving averages — a fast one (9-period) and a slow one (21-period). When the fast one crosses above the slow one, we buy; below, we sell. With every entry we automatically attach a stop-loss (close out if price drops 20 pips against us) and a take-profit (close out if price moves 40 pips in our favour). One trade at a time — if we're still in a position, we ignore new crosses.
Technically: Two ExponentialMovingAverage indicators are updated per 5-min bar (auto-registered for the internal strategy via register_indicator_for_bars; manually via update_raw(close) for the custom strategy since the engine doesn't know about the user-space aggregator). Cross detection uses the sign of fast_ema.value - slow_ema.value compared against a stored _prev_fast_above. On a cross, if self.portfolio.is_flat(instrument_id) the strategy submits a bracket: an OrderList containing a market MARKET entry, a STOP_MARKET stop-loss child, and a LIMIT take-profit child, linked under contingency type OUO ("one-updates-other": when one child fills, the other is cancelled).

The bracket helper

def submit_bracket(strategy, instrument, side, entry_price):
    if side == OrderSide.BUY:
        sl = instrument.make_price(entry_price - SL_PIPS * PIP)   # 20 pips below
        tp = instrument.make_price(entry_price + TP_PIPS * PIP)   # 40 pips above
    else:
        sl = instrument.make_price(entry_price + SL_PIPS * PIP)
        tp = instrument.make_price(entry_price - TP_PIPS * PIP)
    order_list = strategy.order_factory.bracket(
        instrument_id=instrument.id,
        order_side=side,
        quantity=instrument.make_qty(TRADE_SIZE),
        sl_trigger_price=sl,
        tp_price=tp,
        time_in_force=TimeInForce.GTC,
    )
    strategy.submit_order_list(order_list)

The internal-aggregation strategy

aggregator_strategy_demo.ipynb · cell-12 (excerpt)
class InternalEmaCrossStrategy(Strategy):
    def on_start(self):
        self.instrument = self.cache.instrument(self.config.instrument_id)
        self.register_indicator_for_bars(self.config.bar_type, self.fast_ema)
        self.register_indicator_for_bars(self.config.bar_type, self.slow_ema)
        self.subscribe_bars(self.config.bar_type)        # 5-min INTERNAL@1-min EXTERNAL

    def on_bar(self, bar: Bar):
        if not self.indicators_initialized(): return
        fast_above = self.fast_ema.value > self.slow_ema.value
        if self._prev_fast_above is None:
            self._prev_fast_above = fast_above; return
        crossed_up   = fast_above and not self._prev_fast_above
        crossed_down = (not fast_above) and self._prev_fast_above
        self._prev_fast_above = fast_above
        if not (crossed_up or crossed_down): return
        if not self.portfolio.is_flat(self.config.instrument_id):
            return                                       # let active bracket finish
        side = OrderSide.BUY if crossed_up else OrderSide.SELL
        submit_bracket(self, self.instrument, side, float(bar.close))

7. Order routing pipeline

When the strategy says "buy", the order doesn't go straight to the matching engine. It goes through a chain: first a routing layer, then a risk checker, then a venue adapter, then finally the simulated exchange where the actual fill happens. Each layer has one job — and importantly, every step is the same in live trading; only the final adapter swaps from "simulated exchange" to "real broker".
Technically: self.submit_order(order) (or submit_order_list) puts a SubmitOrder command onto the in-process message bus. The ExecutionEngine consumes it, hands it to the RiskEngine (which we configure with bypass=True so it forwards without checking), which hands it to the BacktestExecClient, which hands it to the SimulatedExchange's MatchingEngine. The matching engine maintains a working-order book; market orders fill on the next bar, limit/stop orders sit queued until their condition is met. Fill events flow back via the same bus to the strategy's on_order_filled / on_position_opened handlers.

Live vs backtest: only the last box changes. Same strategy code, same submission API, same upstream pipeline — in live trading BacktestExecClient + SimulatedExchange are replaced by a live execution client and the real venue. That's the whole reason for the layered design.

8. The two clocks

Two completely different rhythms are running in parallel during a backtest. The strategy "thinks" every 5 minutes — only at boundary bars. The matching engine "thinks" every 1 minute — checking if any working order (especially SL/TP) should fire. So when the strategy is asleep between two 5-minute wake-ups, the matching engine is still wide awake and can quietly close your position before you ever see another bar.
Technically: The strategy's on_bar fires on the cadence of whatever BarType it subscribed to (5-min in our case). The matching engine inside SimulatedExchange consumes every record in add_data() — which, for our FX setup, is the full 1-min BID+ASK stream. Each 1-min bar triggers the matching engine to (a) fill any pending market orders at the bar's open, (b) check each working stop/limit order's trigger against the bar's [low, high] range and fire it if touched. This happens whether the strategy is currently awake or not.

5-min decision clock22:00, 22:05, 22:10, …Strategy's on_bar, EMA cross detection, submit_bracket calls
1-min execution clock22:00, 22:01, 22:02, 22:03, …Matching engine's per-bar checks: entry fills, SL/TP triggers, position-close events

9. Order lifecycle

A bracket order has three pieces — an entry, a stop-loss, and a take-profit. The entry is a "buy at next price" market order that always fills on the very next bar, no conditions. Once it fills, the SL and TP become active and wait. The matching engine then checks every following 1-minute bar's price range against the SL and TP levels; the first one that gets touched fires immediately on that bar, and cancels the other one.
Technically: An OrderList from order_factory.bracket(...) contains three orders linked under ContingencyType.OUO. The entry (OrderType.MARKET by default) fills on the next bar at that bar's open price — unconditional because market orders have no price predicate. Once the entry fills, the SL (OrderType.STOP_MARKET) and TP (OrderType.LIMIT) children transition from dormant to working. On every subsequent 1-min bar, the matching engine checks: does [low, high] contain the SL trigger price? Does it contain the TP price? If either is touched, the matching engine fills it on that same bar at the trigger/limit price (not the next bar — that's the difference from market orders), then auto-cancels the sibling via the OUO contingency.

Three timestamps to keep distinct. Strategy submits at T; market entry fills at T+1's open and that is when the position opens; SL or TP closes the position whenever its trigger is first touched (could be any later 1-min bar).
Why is_flat is checked at the top of on_bar, not after submit_order. submit_order is non-blocking — it returns instantly, before any fill. So if you called is_flat immediately after submitting, it would still report the pre-submission state. The portfolio state only updates when the fill event flows back through the message bus, which happens on the next bar at the earliest. So the right pattern is: at the start of each on_bar, ask "what's my state right now?" — the answer reflects everything that filled before this bar.

10. Worked example: 39 vs 40 brackets

We ran the exact same strategy on the exact same data through both aggregators. The internal path made 39 trades; the custom path made 40. The aggregated 5-min bars themselves were identical — same open/high/low/close. The reason for the extra trade is much more subtle: it's that 1-minute delay in when the custom aggregator hands each bar to the strategy. That tiny lag changes the price at which entries fill, which changes where SL/TP get placed, which changes when positions close, which changes whether the strategy is in or out of a trade at some later cross. Across 40 entries over a month, the net imbalance lands at +1 for custom.
Technically: The custom aggregator's flush() result is discarded in on_stop (we don't run cross-detection on the trailing partial), so the trailing partial is NOT the source — the fills count 80 = 40 × 2 proves every bracket completed normally with both entry and exit fills. The cause is the cascade: delivery lag → different entry fill bar → different fill price → different absolute SL/TP price levels → SL/TP triggers fire on different 1-min bars → position-close time differs → is_flat state differs at some future cross detection → one path takes a trade the other skips. Over many crosses the timing perturbations don't cancel perfectly; this run happens to net +1 for custom.

11. publish_data / on_data — when (and when not) to use them

Nautilus has a "message bus" — components talk to each other by publishing typed messages and subscribing to types they care about. Bars and quotes already use this internally; you get them via on_bar and on_quote_tick. If you have custom data (something Nautilus doesn't natively know about — a market-regime label, an alternative data feed, a custom aggregated bar with extra features), you publish it via publish_data and other strategies subscribe to receive it via on_data.
Technically: Actor.publish_data(data_type, data) publishes a Data subclass instance on the message bus under a custom-data topic. Actor.subscribe_data(DataType(cls), instrument_id=…) registers self.handle_data for that topic, which forwards to on_data. Bars, quotes, trades, order book events all have their own typed channels (subscribe_bars + on_bar, etc.) — don't shoehorn them through publish_data. For plain scalar values use publish_signal(name, value, ts_event) + subscribe_signal(name) + on_signal(signal) — it auto-generates a Signal* data class for you.

When to use

ScenarioWhy
Cross-strategy / cross-actor sharingA RegimeDetectorActor publishes MarketRegime snapshots; multiple strategies subscribe. Single producer, many consumers.
Custom data clients / adaptersAn adapter pulls funding rates, news events, on-chain metrics — emits them as a custom Data type for strategies to consume.
Replaying alt-data in a backtestengine.add_data(custom_data_list) interleaves custom Data records with bars/ticks in time order. Subscribed strategies receive them via on_data.

When NOT to use

ScenarioUse instead
Inside one strategy with no other consumerJust call a private helper method (e.g. self._on_5min_bar(b5)) — no bus overhead, no Data subclass needed
Built-in market data (bars, quotes, trades, book events)Use the typed channel (subscribe_bars/on_bar, etc.)
Plain scalar values (int / float / str)Use publish_signal/subscribe_signal — auto-generates the Data class for you

How it would fit our notebook

In CustomAggEmaCrossStrategy, the strategy both produces and consumes Bar5MinFeatures — same actor, single consumer. So we just return the dataclass from on_bar and act on it directly. No bus traffic needed.

But if you later wanted a second strategy in the same engine (say a portfolio-overlay watching multiple FX pairs' RSI) to consume the same 5-min features, the refactor would be:

  1. Make Bar5MinFeatures subclass nautilus_trader.core.data.Data with ts_event / ts_init properties.
  2. After _close_window() in the aggregator, call self.publish_data(DataType(Bar5MinFeatures), b5).
  3. The second strategy adds self.subscribe_data(DataType(Bar5MinFeatures), instrument_id=...) in on_start and implements on_data(self, data: Bar5MinFeatures).

13. Entry control — order types, time-in-force, time-based triggers

In our notebook we used the simplest possible entry: "buy at next market price." But Nautilus supports far more. You can vary three independent dimensions: what kind of order (market, limit, stop, etc.), how long it lives (good-til-cancelled, good-til-date, immediate-or-cancel, etc.), and when you actually submit it (right now, after a 10-second delay, at 09:30 every morning, etc.). And the bracket factory can take any of those non-market entries with SL+TP still attached.
Technically: Entry orders are Order objects produced by OrderFactory methods on the strategy (self.order_factory.market(...), .limit(...), .stop_market(...), etc.). All accept a TimeInForce enum and an optional expire_time: datetime. Time-based scheduling uses the strategy's per-instance Clock (self.clock.set_time_alert(name, alert_time, callback) or set_timer(name, interval, callback)); callbacks fire in the same engine event loop. order_factory.bracket(...) accepts entry_order_type=OrderType.LIMIT (or STOP_MARKET, etc.) plus entry_price / entry_trigger_price, letting you combine non-market entries with SL+TP children under the same OUO contingency.

13.1 Order type — controls at what price the entry fills

# LIMIT entry — wait for price to come to us

13.3 Time-delayed entry via the strategy clock

"Enter 10 seconds after the signal" or "submit a buy at 09:30 every morning" doesn't fit any single order type — you implement it by detecting the signal in on_bar, then asking the clock to fire a callback at a future time. The callback does the actual submit_order.
example — delay entry by 10 seconds
def on_bar(self, bar):
    # detect signal here, but don't submit yet
    if cross_detected and self.portfolio.is_flat(...):
        self._last_close = float(bar.close)
        self.clock.set_time_alert(
            name="delayed_entry",
            alert_time=self.clock.utc_now() + pd.Timedelta(seconds=10),
            callback=self._submit_delayed_entry,
        )

def _submit_delayed_entry(self, event):
    # Fires when the alert time arrives
    submit_bracket(self, self.instrument, OrderSide.BUY, self._last_close)

Other clock methods you'll use:

  • set_timer(name, interval, callback) — recurring tick (e.g. evaluate every 30 seconds)
  • cancel_timer(name) / cancel_time_alert(name) — kill a pending schedule
  • set_time_alert_ns(..., alert_time_ns=...) — same idea but using nanosecond timestamps
Backtest caveat. In a backtest the clock advances in lockstep with the data, so a 10-second delay against 1-min bar data has no effect — the next bar is 60 seconds later, not 10. Sub-bar delays only matter when you replay quote/trade ticks (granular enough to advance the clock between scheduled callback time and the next data event) or in live trading. With 1-min bar data, the smallest meaningful schedule is the next bar.

13.4 Multi-bar / conditional entry

"Enter only after 3 consecutive bars above a level" or "wait for confirmation from a higher timeframe" — no special API for this; just keep a counter or flag in __init__ and update it across bars, gating submit_bracket on the predicate.
example — require 3 consecutive bars above a level
def __init__(self, config):
    ...
    self._consecutive_above = 0

def on_bar(self, bar):
    if float(bar.close) > THRESHOLD:
        self._consecutive_above += 1
    else:
        self._consecutive_above = 0

    if self._consecutive_above >= 3 and self.portfolio.is_flat(...):
        submit_bracket(self, self.instrument, OrderSide.BUY, float(bar.close))

14. Exit control — beyond SL/TP

Brackets give you automatic SL/TP exits, which covers most cases. But you can supplement or replace them with: closing manually on a signal, reversing direction on an opposite signal, scheduling a daily square-off at a fixed time, exiting on an indicator extreme, trailing the stop as price moves in your favour, moving the stop to break-even once a profit threshold is hit, or simply closing after N bars regardless of price.
Technically: Position closes flow through the same engine pipeline as entries — they're just orders. self.close_position(position) and self.close_all_positions(instrument_id) internally generate opposite-side market orders sized to net the position to zero. Trailing offsets are available natively via the bracket factory (sl_trailing_offset, sl_trailing_offset_type); more complex exit logic (move-to-cost, bar-count, trailing with custom step) is implemented strategy-side by cancelling and replacing the working SL order each bar. The project's core/managed_strategy.py implements production-grade versions of trailing-SL, move-to-cost, and timed square-off — refer there for full code.

14.1 Manual close

closing on any condition
def on_bar(self, bar):
    if some_exit_condition_met:
        self.close_all_positions(self.config.instrument_id)
        # OR for a specific position only:
        # for pos in self.cache.positions_open(instrument_id=self.config.instrument_id):
        #     self.close_position(pos)

Internally these submit opposite-side market orders sized to bring the net position back to zero. In NETTING mode they net cleanly to flat.

14.2 Reverse on opposite signal

This is the pattern the project's existing strategies/ema_cross.py uses — when the EMA cross flips, close the current position and immediately open a fresh one in the opposite direction.
strategies/ema_cross.py — reversal pattern
# Currently long; fast EMA just crossed below slow
if self.fast_ema.value < self.slow_ema.value:
    if self.portfolio.is_net_long(self.config.instrument_id):
        self.close_all_positions(self.config.instrument_id)   # close the long
        self._submit_order(OrderSide.SELL)                    # open a fresh short

In NETTING mode you can also just submit a SELL of qty = 2 × position_qty to reverse in one order, but the two-step form is clearer.

14.3 Time-based exit ("square off at HH:MM")

"Close everything at 15:15 every day, no questions asked." Useful for intraday strategies that don't want to carry overnight risk.
daily square-off via clock alert
def on_start(self):
    ...
    # Set a daily alert at 15:15 local time to flatten everything
    self.clock.set_time_alert(
        name="daily_squareoff",
        alert_time=self._next_squareoff_time(),
        callback=self._on_squareoff,
    )

def _on_squareoff(self, event):
    self.cancel_all_orders(self.config.instrument_id)
    self.close_all_positions(self.config.instrument_id)
    # Re-arm for tomorrow
    self.clock.set_time_alert("daily_squareoff", self._next_squareoff_time(), self._on_squareoff)

The project already implements this — see core/managed_strategy.py, the squareoff_minute / squareoff_tz config fields.

14.4 Indicator-based exit

Close when a separate indicator gives an exit signal — e.g. RSI saying "overbought" closes a long even if the SL/TP haven't been hit yet.
example — RSI extreme exit
def on_bar(self, bar):
    if not self.portfolio.is_flat(self.config.instrument_id):
        # In-position: check separate exit conditions
        if self.rsi.value > 70 and self.portfolio.is_net_long(...):
            self.close_all_positions(self.config.instrument_id)
        elif self.rsi.value < 30 and self.portfolio.is_net_short(...):
            self.close_all_positions(self.config.instrument_id)
        return
    # else: flat → look for entries

14.5 Trailing stop — move SL with price as it moves in your favour

Lock in profits as price moves. Nautilus has built-in support — just pass a trailing offset to the bracket factory. You can also do it manually by cancelling and replacing the SL order each bar.
built-in via the bracket factory
order_list = self.order_factory.bracket(
    instrument_id=...,
    order_side=OrderSide.BUY,
    quantity=...,
    sl_trailing_offset=Decimal("20"),                         # 20 pips
    sl_trailing_offset_type=TrailingOffsetType.PRICE,         # or BASIS_POINTS, TICKS
    tp_price=self.instrument.make_price(1.1040),
)

Manual variant — cancel the current SL when price has moved N pips in your favour, then submit a new SL at the new level. Production version with trailing_sl_step / trailing_sl_offset lives in core/managed_strategy.py.

14.6 Move-to-cost / break-even stop

After the position is in profit by some threshold, move the SL up to the entry price so the worst case becomes "flat" instead of a loss. Many traders consider this essential risk management.
example — move SL to entry after threshold
def on_bar(self, bar):
    if not self.portfolio.is_flat(self.config.instrument_id):
        positions = list(self.cache.positions_open(instrument_id=self.config.instrument_id))
        if not positions: return
        pos = positions[0]
        unrealised_pnl_pips = (float(bar.close) - float(pos.avg_px_open)) / PIP
        if unrealised_pnl_pips > MOVE_SL_THRESHOLD_PIPS and not self._moved_to_cost:
            # Cancel old SL, place new one at entry price
            for working in self.cache.orders_working(instrument_id=...):
                if working.order_type == OrderType.STOP_MARKET:
                    self.cancel_order(working)
            new_sl = self.order_factory.stop_market(
                ...,
                trigger_price=self.instrument.make_price(float(pos.avg_px_open)),
                time_in_force=TimeInForce.GTC,
                reduce_only=True,
            )
            self.submit_order(new_sl)
            self._moved_to_cost = True

Production version is in core/managed_strategy.py under the move_sl_enabled / move_sl_safety_sec / move_sl_action config block.

14.7 Bar-count / time-in-position exit

Close after N bars regardless of price. Useful when you want a hard time cap on how long capital is committed to any one trade.
example — close after 20 bars in position
def on_bar(self, bar):
    if not self.portfolio.is_flat(...):
        self._bars_in_position += 1
        if self._bars_in_position >= 20:                # exit after 20 5-min bars
            self.close_all_positions(self.config.instrument_id)
            self._bars_in_position = 0
        return

14.8 Combining bracket SL/TP with early exit

You can keep the bracket's SL/TP as your safety net and exit early on a different signal. Just cancel the bracket children and close the position when the early-exit fires.
example — early exit on opposite EMA cross
def on_bar(self, bar):
    # Already in a bracket position — check for an EARLY exit signal
    if not self.portfolio.is_flat(...):
        if self.fast_ema.value < self.slow_ema.value and self.portfolio.is_net_long(...):
            # Cross back down — exit BEFORE SL/TP fires
            self.cancel_all_orders(...)        # cancel pending SL + TP children
            self.close_all_positions(...)
        return
    # else: flat → look for entries (existing cross detection)

Custom aggregator output has one extra row at the end compared to the Nautilus output The data stream ends mid-window (last 4 of 5 bars present).
