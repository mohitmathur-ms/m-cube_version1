Stoploss & Target — Implementation Report

A friendly, end-to-end walk-through of every way you can configure stoploss (SL) and target (TGT) in this system — both at the leg level (per-strategy slot) and the portfolio level (whole book). Includes JSON keys, formulas, code entry points and copy-paste recipes.

1. Introduction

What is a stoploss? A pre-set price (or PnL amount) at which an open position is automatically closed to limit losses. What is a target? The opposite — a pre-set level at which the position is closed to lock in profits.

This system supports SL and Target at two independent levels:

Leg Level per-slot
core/managed_strategy.py
Each strategy slot in a portfolio has its own SL/TGT. Evaluated in real time on every bar. Acts only on that single slot. Think of it as the local stop for one player.
Portfolio Level aggregate
core/backtest_runner.py
Looks at the combined PnL of all slots together. When the whole book hits a loss/profit number, all slots can be cut at once. Think of it as the "team" risk limit.
Analogy: imagine each slot is a player on a team. Leg-level SL/TGT is each player's personal stop ("I'll quit if I lose $50"). Portfolio-level SL/TGT is the coach's rule for the whole team ("if the team is down $500 collectively, everyone goes home"). Both can be active at the same time — whichever fires first wins.

SL 3. Leg-Level Stoploss 4 types

Leg stoploss is set per slot via the stop_loss_type field. Exactly four values are accepted (see core/models.py:24):

3.1 None type 1/4
"stop_loss_type": "none"
No SL is checked. The position only exits via target, squareoff time, or manual close. This is the default value.
3.2 Percentage type 2/4
"stop_loss_type": "percentage"
SL is a % away from entry. stop_loss_value: 0.5 means 0.5% below entry for a long, 0.5% above for a short. Computed at fill time, never moves.
3.3 Points type 3/4
"stop_loss_type": "points"
SL is a fixed price distance from entry. stop_loss_value: 0.0050 on a long entry of 1.1000 → SL at 1.0950. Useful for FX (raw pip distance).
3.4 Trailing type 4/4
"stop_loss_type": "trailing"
Starts as a % SL, then ratchets tighter as profit grows. Two extra fields — trailing_sl_step (profit needed to ratchet) and trailing_sl_offset (how much to tighten by).

Formulas (from _compute_sl_price(), managed_strategy.py:710)

# percentage
long_sl = entry_price * (1 - stop_loss_value / 100)
short_sl = entry_price * (1 + stop_loss_value / 100)
 
# points
long_sl = entry_price - stop_loss_value
short_sl = entry_price + stop_loss_value
 
# trailing — ratchets every `trailing_sl_step` % of profit
on every bar:
if profit_pct >= last_trail_level + trailing_sl_step:
current_sl = current_sl + trailing_sl_offset # long: tighten upward
last_trail_level = profit_pct

Example — 0.5% percentage SL on a long EUR/USD trade

// portfolios/my_portfolio.json — slot.exit_config
{
  "stop_loss_type": "percentage",
  "stop_loss_value": 0.5,
  "target_type":    "none",
  "target_value":   0.0
}
// Entry @ 1.1000 long → SL @ 1.10000 * (1 - 0.005) = 1.09450
Code references: SL field defined in core/models.py:24 · SL fill price computed in core/managed_strategy.py:686-694 · SL hit detection (with optional sl_wait_bars) in managed_strategy.py:495-511 · trailing-SL ratchet at managed_strategy.py:482-492.

TGT 4. Leg-Level Target 3 types

Leg target is set via target_type in exit_config. Three values are accepted (see core/models.py:30):

4.1 None type 1/3
"target_type": "none"
No profit target is checked. Position is held until SL, squareoff, or manual close. Default value.
4.2 Percentage type 2/3
"target_type": "percentage"
Target is a % away from entry. target_value: 0.5 means 0.5% above entry for a long, 0.5% below for a short.
4.3 Points type 3/3
"target_type": "points"
Target is a fixed price distance from entry. target_value: 0.0050 on a long at 1.1000 → target at 1.1050.

Formulas (from on_order_filled(), managed_strategy.py:697-706)

# percentage
long_tp = entry_price * (1 + target_value / 100)
short_tp = entry_price * (1 - target_value / 100)
 
# points
long_tp = entry_price + target_value
short_tp = entry_price - target_value

4.4 Bonus: Target Locking (a SL-style trailing protection)

Two optional fields turn a winning leg into a guaranteed-minimum-profit leg by raising the SL once a profit threshold is reached:

FieldMeaning
target_lock_triggerProfit % at which the lock activates (e.g., 0.4 → 0.4% in profit).
target_lock_minimumThe minimum % profit that will now be guaranteed by raising the SL (e.g., 0.2 → SL is moved so the worst case is +0.2% profit).
Where: implemented in managed_strategy.py:463-471. Once triggered, this acts as a one-shot SL upgrade — the SL is raised, the original target_type still controls the upper exit.
Code references: target field at core/models.py:30 · TP fill price at core/managed_strategy.py:697-706 · TP hit detection at managed_strategy.py:514-522.

5. Auxiliary Leg Exit Features

These are not separate "types" of SL/Target, but they shape how SL and Target behave at the leg level:

5.1 SL Wait Bars (false-trigger filter)

Field sl_wait_bars on ExitConfig. SL only fires if the SL condition holds for N consecutive bars. If the price wicks back inside the SL within those N bars, the trigger is cancelled. Useful to filter out single-bar wicks. Default 0 = fire immediately.

5.2 Move SL to Cost (break-even protection)

Configured at the portfolio level (because it can react to events on other slots) but applied per leg. After the position has been profitable for move_sl_safety_sec seconds, the SL is raised to the entry price — so the worst case becomes break-even.

FieldMeaning
move_sl_enabledMaster switch.
move_sl_safety_secHow long the leg must stay in profit before SL is moved.
move_sl_action"Move Only for Profitable Legs" (default) — only profitable legs are protected.
move_sl_trail_afterIf true, SL keeps trailing further as profit grows.
move_sl_no_buy_legsSkip the move-to-cost logic for long positions.
move_sl_hit_on_leg_slCross-slot trigger: when any other slot's SL hits, run move-to-cost on remaining legs.
move_sl_hit_on_leg_targetSame but triggered when any slot's target hits.
Code: core/managed_strategy.py:438-457 · validation in core/backtest_runner.py:851-885.

5.3 Squareoff Time (time-based forced exit)

A clock-based exit. Set squareoff_time (HH:MM) and optional squareoff_tz (IANA timezone). Resolution priority:

  1. Leg-level ExitConfig.squareoff_time — most specific, wins if set.
  2. Slot-level StrategySlotConfig.squareoff_time — falls back here.
  3. Portfolio-level PortfolioConfig.squareoff_time — last resort.
Code: evaluated in core/managed_strategy.py:253-272 at the very top of on_bar().

5.4 Exit Actions — what happens when SL/TGT fires

close default
"on_sl_action": "close"
Just closes the position. The slot remains armed — re-entries depend on the strategy's normal entry logic.
re_execute
"on_sl_action": "re_execute"
Closes the position AND immediately re-opens it in the same direction (subject to max_re_executions cap and delay_between_legs_sec).
reverse
"on_sl_action": "reverse"
Closes and immediately opens an equal-and-opposite position. Often used as an "if I'm wrong, flip" tactic.

5.5 Action Filters (Other Settings tab)

Two filter fields on PortfolioConfig decide whether the action above fires when the SL/TGT hit was the fixed level vs a trailed level:

FieldValuesBehavior
on_sl_action_onOnSL_N_Trailing_Both · OnSL_Only · OnSL_Trailing_OnlyFire on any SL hit / only fixed-SL hits / only trailed-SL hits.
on_target_action_onOnTarget_N_Trailing_Both · OnTarget_Only · OnTarget_Trailing_OnlySame shape, for targets.
P
Portfolio Level — whole-book risk limits

SL 6. Portfolio-Level Stoploss 5 types

Set via pf_sl_type on PortfolioConfig. The five accepted strings are listed below — but only one is universal: on FX/crypto books all options-only types silently downgrade to "Combined Loss" with a warning logged from _resolve_pf_stoploss() (backtest_runner.py:728-793).

6.1 Combined Loss FX + Options
"pf_sl_type": "Combined Loss"
Universal type. Fires when summed PnL across all slots ≤ −pf_sl_value. Implementation: backtest_runner.py:991.
6.2 Combined Premium Options-only
"pf_sl_type": "Combined Premium"
Net premium received/paid across legs of a multi-leg options structure crosses a threshold. On FX it is downgraded to Combined Loss with a warning (backtest_runner.py:742-745).
6.3 Absolute Combined Premium Options-only
"pf_sl_type": "Absolute Combined Premium"
Like Combined Premium but uses absolute value — ignores the sign of the net premium.
6.4 Underlying Movement Options-only
"pf_sl_type": "Underlying Movement"
Triggers based on how far the underlying spot has moved vs entry — independent of leg PnL.
6.5 Loss and Underlying Range Options-only
"pf_sl_type": "Loss and Underlying Range"
Compound trigger — fires only when both a PnL loss threshold AND a spot-range threshold are breached.

6.6 Bonus: Trailing Portfolio SL

Three fields on PortfolioConfig ratchet the SL tighter as the book gains:

FieldMeaning
pf_sl_trail_enabledMaster switch for trailing portfolio SL.
pf_sl_trail_everyProfit step (raw PnL) required before the SL ratchets. e.g., 100 = each $100 of new high-water mark triggers a tighten.
pf_sl_trail_byHow much (raw PnL) the SL tightens per ratchet step.
Code: trailing-SL ratchet at core/backtest_runner.py:978-988.

6.7 Portfolio SL action options

SqOff default
"pf_sl_action": "SqOff"
Square off all open legs at the moment the portfolio SL fires. Optional filter flags below let you cut only-loss-legs or only-profit-legs.
ReExecute
"pf_sl_action": "ReExecute"
Square off and replay strategy entries from the clip point onward (subject to pf_sl_reexecute_count; 0 = unlimited).

6.8 All Portfolio SL fields (full schema)

{
  "pf_sl_enabled": false,
  "pf_sl_type": "Combined Loss",
  "pf_sl_value": 0.0,                    // raw PnL amount (positive)
  "pf_sl_action": "SqOff",                 // SqOff | ReExecute
  "pf_sl_delay_sec": 0,                   // confirmation delay before firing
  "pf_sl_reexecute_count": 0,             // 0 = unlimited
  "pf_sl_sqoff_only_loss_legs": false,
  "pf_sl_sqoff_only_profit_legs": false,    // mutually exclusive with above
  "pf_sl_trail_enabled": false,
  "pf_sl_trail_every": 0.0,
  "pf_sl_trail_by": 0.0
}

TGT 7. Portfolio-Level Target 4 types

Set via pf_tgt_type on PortfolioConfig. The four types mirror the SL list, minus the compound "Loss and Underlying Range" combo. Same FX-vs-Options downgrade rules apply.

7.1 Combined Profit FX + Options
"pf_tgt_type": "Combined Profit"
Universal type. Fires when summed PnL across all slots ≥ pf_tgt_value. Implementation: backtest_runner.py:1020.
7.2 Combined Premium Options-only
"pf_tgt_type": "Combined Premium"
Mirror of the SL counterpart, but for profit side.
7.3 Absolute Combined Premium Options-only
"pf_tgt_type": "Absolute Combined Premium"
Absolute-value variant — fires regardless of sign.
7.4 Underlying Movement Options-only
"pf_tgt_type": "Underlying Movement"
Fires when underlying spot has moved a configured distance from entry, regardless of PnL.

7.5 Bonus: Trailing Portfolio Target / Profit Lock

Once profit reaches pf_tgt_trail_when_profit_reach, the system starts locking a minimum profit (the pf_tgt_trail_lock_min_profit) and trails the SL upward as further gains accrue.

FieldMeaning
pf_tgt_trail_enabledMaster switch.
pf_tgt_trail_when_profit_reachActivation threshold (raw PnL). Below this, no locking.
pf_tgt_trail_lock_min_profitMinimum profit (raw PnL) you guarantee from this point on.
pf_tgt_trail_everyEach PnL step that further raises the locked floor.
pf_tgt_trail_byHow much the locked floor rises per step.
Code: core/backtest_runner.py:996-1014.

8. Implementation Guide — How to Wire It Up

Five copy-paste recipes covering the most common scenarios. Edit the file portfolios/<your_portfolio>.json, save, and re-run a backtest. No code changes are needed.

Recipe 8.1 — Add a 0.5% leg-level SL with no target

Edit the slot's exit_config:

{
  // inside slots[i].exit_config
  "stop_loss_type": "percentage",
  "stop_loss_value": 0.5,
  "target_type":    "none",
  "target_value":   0.0,
  "on_sl_action":   "close"
}
Effect: for every entry on this slot, position is closed when price moves 0.5% against entry. No profit target — the only way out is SL or squareoff.

Recipe 8.2 — Add a trailing leg-level SL (initial 0.3%, ratchet 0.05%, tighten 0.02%)

{
  "stop_loss_type": "trailing",
  "stop_loss_value": 0.3,         // initial SL distance %
  "trailing_sl_step": 0.05,      // every 0.05% of profit, ratchet
  "trailing_sl_offset": 0.02,    // tighten SL by 0.02% per ratchet
  "target_type": "none",
  "on_sl_action": "close"
}
Effect: SL starts 0.3% from entry. As soon as the trade is +0.05% in profit, SL tightens by 0.02%. As more profit accrues, SL keeps stepping closer to current price — locking in gains.

Recipe 8.3 — Portfolio Combined-Loss SL of $500 with SqOff

Edit the top level of portfolios/<name>.json (NOT inside exit_config):

{
  // top-level fields on PortfolioConfig
  "pf_sl_enabled": true,
  "pf_sl_type": "Combined Loss",
  "pf_sl_value": 500.0,
  "pf_sl_action": "SqOff",
  "pf_sl_delay_sec": 0
}
Effect: after the backtest finishes, the merged equity curve is walked tick-by-tick. The first moment combined PnL ≤ -$500, all open trades are clipped (closed) at that timestamp; trades after the clip are dropped from the report.

Recipe 8.4 — Portfolio Combined-Profit target $1000 with profit lock

{
  "pf_tgt_enabled": true,
  "pf_tgt_type": "Combined Profit",
  "pf_tgt_value": 1000.0,
  "pf_tgt_action": "SqOff",
  "pf_tgt_trail_enabled": true,
  "pf_tgt_trail_when_profit_reach": 600.0,    // arm at +$600
  "pf_tgt_trail_lock_min_profit": 300.0,        // guarantee +$300
  "pf_tgt_trail_every": 100.0,                  // every +$100 more profit...
  "pf_tgt_trail_by": 50.0                         // ...raise the floor by $50
}
Effect: hard target is $1000. But once the book is up $600, a profit-lock arms guaranteeing at least $300; for every additional $100 of gain, the locked floor steps up by $50. Either the target or the floor will close out the book.

Recipe 8.5 — Combine leg SL with portfolio SL

Both can be active simultaneously — they don't conflict. Each slot has its own leg SL, the portfolio has an aggregate SL.

// each slot
slot.exit_config = {
  "stop_loss_type": "percentage",
  "stop_loss_value": 0.5,
  "on_sl_action": "close"
}
// portfolio top level
portfolio.pf_sl_enabled = true
portfolio.pf_sl_type    = "Combined Loss"
portfolio.pf_sl_value   = 500.0
portfolio.pf_sl_action  = "SqOff"
Precedence: Leg SL fires in real time during the backtest. Portfolio SL is applied post-hoc. If the portfolio-level clip happens before some leg SL would have fired, post-clip leg trades are dropped from the report. Net: whichever exit is "earlier" in time wins, with portfolio clipping taking precedence in the merged report.

10. Glossary & FAQ

Glossary

TermMeaning
LegOne individual open position. Inside this codebase, leg-level state is tracked inside a slot's strategy instance.
SlotA configured strategy entry on a portfolio (one row in slots[]). Each slot has its own SL/TGT via exit_config.
PortfolioThe collection of all slots, plus shared run settings (capital, dates, RBO, portfolio SL/TGT).
FXForeign exchange instruments. The "universal" SL/TGT types are the only ones that work here — options-only types are downgraded with a warning.
TrailingAn SL/floor that ratchets in your favor as the trade moves into profit, but never moves against you.
ClipThe portfolio-level act of cutting all trades from a given timestamp onward in the merged equity curve.

FAQ

Why does Options have more SL/Target types than FX?

Multi-leg options books have additional risk dimensions (net premium, underlying spot movement, range breaches) that simply don't exist for spot FX. The codebase keeps the option-specific values in the schema for forward compatibility but downgrades them to "Combined Loss" / "Combined Profit" with a warning when the portfolio runs FX/crypto data — see core/backtest_runner.py:742-745.

What if my leg SL fires before the portfolio SL would have?

Leg SL fires in real time during the backtest run. Portfolio SL is applied after the backtest, by walking the merged equity curve. If your leg SL fires at 10:32:00 and the portfolio-level SL would have fired at 10:31:30, the post-hoc clip drops every trade after 10:31:30 — including trades the leg SL had already closed. Net effect: the earlier of the two timestamps wins.

Where are SL/TGT hits logged?

The exit reason string is constructed in _handle_exit() at core/managed_strategy.py:570-575 and emitted with the close event. Reports built from the merged trades file include the reason verbatim, so SL/TGT/squareoff/clip hits are all distinguishable in the audit trail.

Can I have a target without a stoploss (or vice versa)?

Yes. Both stop_loss_type and target_type default to "none". Set whichever you need. They are evaluated independently each bar.

Does re_execute on a leg respect the portfolio's no-re-entry rules?

Yes. Re-execute fires through _handle_exit() but is gated by delay_between_legs_sec on the portfolio and by max_re_executions on the leg. The ReExecute tab also has a no_reexec_sl_cost flag that suppresses re-execution after a Move-SL-to-Cost upgrade — see 5. Logics/ReExecute_Logics.html for the full spec.
