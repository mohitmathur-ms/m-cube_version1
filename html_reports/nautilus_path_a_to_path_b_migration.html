1. Why two paths exist

NautilusTrader exposes two backtest APIs in Python (mirroring the Rust ones in run_rust_backtest.md):

  • Path A  BacktestEngine — low-level. You construct the engine, call add_venue() / add_instrument() / add_data() / add_strategy() by hand, then engine.run(). Bars live in the engine's in-memory buffer.
  • Path B  BacktestNode — high-level. You describe the run with config dataclasses (BacktestVenueConfig, BacktestDataConfig, BacktestRunConfig) and the node reads from a ParquetDataCatalog on disk, optionally streaming in chunks of N rows.

Both paths run through the same matching engine internals — BacktestNode just builds a BacktestEngine for each BacktestRunConfig under the hood (venv/Lib/site-packages/nautilus_trader/backtest/node.py:104, self._engines: dict[str, BacktestEngine] = {}). The difference is purely in the wiring layer.

Today this project uses Path A. Verified by greping the codebase: BacktestNode is not imported in any Python source — only referenced inside static HTML doc files. BacktestEngine is constructed in three sites inside core/backtest_runner.py, all of which run the manual 5-step Path A flow.

2. What this project does today (Path A)

Three engine sites, all in core/backtest_runner.py, all following the same 5-step pattern. The Parquet catalog is used as storage only: bars are read into Python lists with catalog.bars(...) and then handed to the engine via engine.add_data(bars).

_run_single_slot() — per-slot portfolio worker [line 355]

Same 5-step pattern, run inside a ProcessPoolExecutor worker. Differences:

  • Wraps strategy in ManagedExitStrategy when SL/TP/trailing or squareoff_time is configured (line 490-499).
  • Adds phase-timing wrappers (_phase) and worker telemetry (RSS, cache hits) to the result dict.
  • Engine wrapped in try/finally to guarantee engine.dispose() on BaseException (line 567-572).

Storage layer — ParquetDataCatalog already in place

This is the lucky part. Path B's prerequisite is “data lives in a Parquet catalog on disk” — and the project already writes its data there via save_to_catalog() at core/nautilus_loader.py:88-123:

catalog = ParquetDataCatalog(catalog_path)
catalog.write_data([instrument])    # write_instruments-equivalent
catalog.write_data(bars)            # write_to_parquet-equivalent

So the storage side requires no work at all. Both paths read the same Parquet files; only the in-memory wiring layer changes.

3. Path B Python API — verified field shapes

All field names and types below come from the installed package: venv/Lib/site-packages/nautilus_trader/backtest/config.py and .../backtest/node.py. Nothing here is invented from generic tutorial material.

BacktestNode — the orchestrator [node.py:76]

class BacktestNode:
    def __init__(self, configs: list[BacktestRunConfig]) -> None: ...
    def build(self) -> None: ...                  # constructs internal engines from configs
    def run(self) -> list[BacktestResult]: ...     # implicit build() if not already built
    def get_engine(self, run_config_id: str) -> BacktestEngine | None: ...
    def get_engines(self) -> list[BacktestEngine]: ...
    def dispose(self): ...
The key trick for this project: custom strategies (loaded via importlib from custom_strategies_dir) cannot be expressed as ImportableStrategyConfig — they're not picklable by string path. So you must call node.build() explicitly, fetch the engine via node.get_engine(run_id), attach the strategy with engine.add_strategy(strategy), and only then call node.run().

BacktestVenueConfig [config.py:50]

Same conceptual fields as the existing engine.add_venue(...) call — the only meaningful change is starting_balances goes from list[Money] to list[str] because configs need to be msgspec-serializable. Format: "1000000 USD".

BacktestRunConfig [config.py:378]

class BacktestRunConfig(NautilusConfig, frozen=True):
    venues: list[BacktestVenueConfig]
    data:   list[BacktestDataConfig]              # list: multiple data configs per run is supported
    engine: BacktestEngineConfig | None = None
    chunk_size: int | None = None             # the streaming knob
    raise_exception:        bool = False
    dispose_on_completion:  bool = True
    start: str | int | None = None
    end:   str | int | None = None
    data_clients: dict[str, type[LiveDataClientConfig]] | None = None
Streaming gate: chunk_size=None means “load everything at once” (behavioral parity with Path A). chunk_size=100_000 (or any positive int) enables row-chunked streaming via DataBackendSession(chunk_size=...). Migrate with None first, tune later.

5. Concrete code diff: 3 sites to migrate

Recommended approach: parallel implementation, gated by _USE_BACKTEST_NODE=1. Don't rip out Path A. The project already uses this pattern for the shared-engine path (_USE_GROUPING at line 1021) — mirror it.

1Add a shared run-config builder

One helper that all three migration sites can call. Put it near _pair_bid_ask_bar_type in backtest_runner.py.

from nautilus_trader.backtest.config import (
    BacktestVenueConfig, BacktestDataConfig, BacktestRunConfig,
)

def _build_run_config(
    catalog_path: str,
    instrument_id,
    bar_type_strs: list[str],          # primary + paired + extras
    venue,
    starting_capital: float,
    start_date: str | None,
    end_date:   str | None,
    trader_id: str = "BACKTESTER-001",
    chunk_size: int | None = None,    # None = parity with Path A
    oms_type: str = "NETTING",
    run_id: str = "run-001",
) -> BacktestRunConfig:
    venue_cfg = BacktestVenueConfig(
        name=str(venue),
        oms_type=oms_type,
        account_type="MARGIN",
        starting_balances=[f"{starting_capital} USD"],
        base_currency="USD",
        default_leverage=1.0,
    )
    data_cfgs = [
        BacktestDataConfig(
            catalog_path=catalog_path,
            data_cls="nautilus_trader.model.data.Bar",
            instrument_id=str(instrument_id),
            bar_types=bar_type_strs,
            start_time=start_date,
            end_time=end_date,
        )
    ]
    engine_cfg = BacktestEngineConfig(
        trader_id=TraderId(trader_id),
        logging=LoggingConfig(bypass_logging=True),
        risk_engine=RiskEngineConfig(bypass=True),
        run_analysis=False,
    )
    return BacktestRunConfig(
        venues=[venue_cfg],
        data=data_cfgs,
        engine=engine_cfg,
        chunk_size=chunk_size,
    )

6. Behavioral changes to be aware of

Worker-local LRU cache becomes inert. _cached_catalog_bars() at line 119 caches catalog.bars(...) results between slots in the same worker process — saves ~500 ms of parquet read per cache hit. Path B reads via DataBackendSession (Arrow) on every run, so this win disappears for repeat runs of the same (bar_type, range) in one worker. Mostly relevant when running the same portfolio many times back-to-back; for normal one-shot portfolio runs the impact is negligible.
Custom strategies must be attached after node.build(). The project loads strategies via importlib from custom_strategies_dir (core/custom_strategy_loader.py), so they cannot round-trip through ImportableStrategyConfig(strategy_path="pkg.mod:Cls", config=...). The imperative attach pattern (build() → get_engine() → add_strategy() → run()) is the only viable flow.
Money → string for starting balances. BacktestVenueConfig.starting_balances: list[str] — format "1000000 USD". Don't pass Money(...) objects.
Don't double-dispose. BacktestRunConfig.dispose_on_completion=True (default) means node.run() already disposes engines on completion. Drop the try/finally: engine.dispose() block in the new code paths — or set dispose_on_completion=False if you need the engine handle alive after node.run() returns (e.g. for additional cache reads).

7. Things that stay the same

  • HTTP entry points — server.py:969 calls run_backtest(); server.py:1174 calls run_portfolio_backtest(). Both function signatures are preserved; the env flag flips the implementation.
  • ProcessPoolExecutor parallelism — one BacktestNode per worker, single run config each. Same blast radius, same scheduling, same _USE_GROUPING behavior.
  • FX BID/ASK auto-pairing — _pair_bid_ask_bar_type() stays as-is. Its output is the input to BacktestDataConfig.bar_types.
  • Parquet catalog writer — core/nautilus_loader.py needs zero changes. It already writes the format Path B reads.
  • _extract_results() and downstream report code — all six engine APIs it touches (engine.trader.generate_*, engine.kernel.cache.*) are available on engines retrieved via node.get_engine(run_id).
  • FX rate resolver — FxRateResolver.from_adapter_config(...) reads from the catalog directly, independent of which engine API drove the run.
  • Worker SIGINT handling — _worker_init_ignore_sigint stays. node.run() blocks the same way engine.run() does.
  • Runtime history / LPT scheduling — pure metadata, untouched.

8. Verification plan

  1. Pick a representative portfolio — ideally 2–3 slots covering both crypto-LAST (no pairing) and FX-MID (BID/ASK auto-pair exercised). Use an existing one from portfolios/.
  2. Run with _USE_BACKTEST_NODE unset (current Path A). Capture account_report, fills_report, positions_report for each slot to JSON.
  3. Run with _USE_BACKTEST_NODE=1, chunk_size=None. Diff against step 2 — expect bit-exact match on all three reports. Same matching engine internals, same data, just a different wiring layer. Any divergence here is a bug in the migration.
  4. Run with _USE_BACKTEST_NODE=1, chunk_size=100_000. Diff again — expect a match modulo floating-point noise (chunked accumulation can shift the last few sig-figs of running PnL).
  5. Sanity-check via the browser — hit /api/backtest/run-stream and /api/backtest/run-backtest-api with the flag set. The HTTP response shape must be identical.
  6. Spot-check shared-engine path — flip _USE_GROUPING=1 alongside _USE_BACKTEST_NODE=1 and confirm grouped runs still produce the same per-slot results.
  7. Profile — with _PROFILE_PHASES=1, compare phase_times across paths. Expected: similar engine_run; bars_load is replaced by node-internal data load, time should be comparable; cache hits drop to zero (expected).

Once steps 3 and 5 produce parity for one portfolio, expand to the regression set — 5–10 portfolios across asset classes — before flipping the default.

9. Trade-offs — should you actually migrate?

Pros of Path B

  • Memory ceiling. chunk_size caps peak in-memory bars regardless of date range. Year of 1-minute FX bars is ~75 MB per side; loading 5 years of MID + ASK + BID is ~1 GB per slot. Streaming makes that flat.
  • Configs are serializable. BacktestRunConfig is a frozen msgspec dataclass — you can dump/load it, log it, replay it, store it next to the result file. Easier to reproduce a run from the config alone.
  • Less manual wiring. Venue, instrument loading, and data attachment are all derived from the config — less surface to bug-fix.

Cons / things to weigh

  • The current code is already memory-conscious. del bars after add_data at line 254 and line 469; worker-local LRU. If you're not hitting RAM pressure today, the streaming win is theoretical.
  • Cache regression for repeated runs. See section 6 — the LRU cache that currently saves ~500 ms per repeated (bar_type, range) read inside one worker process becomes inert.
  • Migration cost is real. Three engine sites + result-extraction wiring + verification across portfolios. Probably 1–2 days of focused work plus a soak window before flipping the default.
  • The shared-engine path needs care. The _run_slot_group migration is the most fiddly piece — works fine via imperative add_strategy after node.build(), but is the highest-risk change.
Recommendation: migrate only if (a) you're hitting memory ceilings on long-range backtests, or (b) you specifically want serializable run configs for reproducibility/replay. Otherwise the existing Path A code is functionally equivalent, well-instrumented, and already production-tested in this codebase. If you do migrate, do it gated — _USE_BACKTEST_NODE=1 — validate parity on real portfolios, then flip the default and remove the dead code in a follow-up.
