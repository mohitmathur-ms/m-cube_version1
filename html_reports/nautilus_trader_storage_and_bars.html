Where Is Data Stored? (Three Independent Tiers)

Tier 1 In-Memory Cache

Speed: Nanoseconds

Capacity: Limited (10K items)

File: cache/cache.pyx

Latest quotes, trades, bar history deques. Fastest access, used during runtime.

Tier 2 Parquet Catalog

Speed: Milliseconds

Capacity: Unlimited

File: persistence/catalog/parquet.py

All market data as .parquet files. Used for backtesting and analysis. Supports cloud.

Tier 3 Redis Optional

Speed: Sub-millisecond

Capacity: Medium

File: cache/database.pyx

Orders, positions, accounts ONLY. For crash recovery. NOT for market data.

No traditional database for market data! NautilusTrader uses Parquet files (columnar, compressed, fast reads) — not SQL, not MongoDB, not InfluxDB. Redis is only for trading state and is optional.

How Data Merges (The Write Path)

1 Ingestion → In-Memory Cache

_handle_quote_tick(tick):
    cache.add_quote_tick(tick)    ← adds to in-memory deque (max 10,000 items)
    msgbus.publish(topic, tick)   ← routes to all subscribers

The Cache holds:

  • Latest values: dict[InstrumentId, QuoteTick] — only the most recent
  • History deques: dict[InstrumentId, deque[QuoteTick]] — last N items (configurable via tick_capacity / bar_capacity)

2 Ingestion → Parquet Catalog (if configured)

# In data/engine.pyx
self._update_catalog(data, data_cls, identifier)
    → catalog.write_data([data])
        → ArrowSerializer.serialize_batch()
        → pq.write_table() ← new .parquet file

The catalog enforces:

  • No overlapping timestamps — each file covers a unique time range
  • Adjacent file merging — extend_file_name() renames files to merge adjacent time ranges without rewriting data

BarBuilder: The OHLCV Accumulator

Step 1: BarBuilder.update() — Each tick updates OHLCV

cpdef void update(self, Price price, Quantity size, uint64_t ts_init):
    if self._open is None:          # First tick → set OPEN
        self._open = price
        self._high = price
        self._low = price
    elif price > self._high:         # Update HIGH
        self._high = price
    elif price < self._low:          # Update LOW
        self._low = price

    self._close = price              # Always update CLOSE to latest price
    self.volume += size              # Accumulate VOLUME
    self.count += 1                  # Count ticks in this bar

Step 2: BarBuilder.build() — Emit the completed bar

cpdef Bar build(self, uint64_t ts_event, uint64_t ts_init):
    # Edge case: no ticks received → use last bar's close
    if self._open is None:
        self._open = self._last_close
        self._high = self._last_close
        self._low = self._last_close
        self._close = self._last_close

    bar = Bar(
        bar_type=self._bar_type,
        open=self._open,
        high=self._high,
        low=self._low,
        close=self._close,
        volume=self.volume,
        ts_event=ts_event,
        ts_init=ts_init,
    )
    self._last_close = self._close   # Remember for next bar's edge case
    self.reset()                      # Clear OHLCV state for next bar
    return bar

TimeBarAggregator (1-minute, 5-minute bars, etc.)

Line: 1390 in aggregation.pyx

The most commonly used aggregator. Uses an internal Clock timer to fire at regular intervals.

Configuration options

ParameterDefaultWhat it does
interval_type"left-open""left-open" = (start, end], "right-open" = [start, end)
timestamp_on_closeTrueBar's ts_event = close time (True) or open time (False)
skip_first_non_full_barFalseSkip emitting a bar if aggregation starts mid-interval
build_with_no_updatesTrueIf True, emit bar even if no ticks arrived (uses last close)

Walkthrough: 1-minute bars

1. Strategy subscribes:
   subscribe_bars(BarType("ETHUSDT.BINANCE-1-MINUTE-LAST"))

2. DataEngine creates TimeBarAggregator:
   Sets internal timer via Clock to fire every 60 seconds
   interval_type="left-open" → tick AT the boundary goes to NEXT bar

3. Ticks arrive continuously:
   10:00:00.100  TradeTick(price=3500.50, size=1.5)  → builder.update()
   10:00:00.250  TradeTick(price=3501.00, size=0.8)  → builder.update()
   10:00:00.500  TradeTick(price=3499.75, size=2.0)  → builder.update()
   ... hundreds more ticks ...
   10:00:59.990  TradeTick(price=3502.25, size=1.0)  → builder.update()

4. Timer fires at 10:01:00.000:
   → builder.build(ts_event=10:01:00) returns:
     Bar(open=3500.50, high=3505.00, low=3498.00, close=3502.25, volume=1543.7)
   → handler(bar) → published to message bus
   → builder.reset() → ready for next minute

5. If NO ticks arrive in next minute:
   build_with_no_updates=True (default)
   → Emits bar with open=high=low=close=last_close, volume=0

VolumeBarAggregator (Smart Tick Splitting)

Line: 577 in aggregation.pyx

The cleverest aggregator — it splits a single tick across bar boundaries when volume overflows.

The algorithm

cdef void _apply_update(self, Price price, Quantity size, uint64_t ts_init):
    raw_size_update = size
    raw_step = bar_type.spec.step  # e.g., 100 units

    while raw_size_update > 0:     # While there is size to apply
        if builder.volume + raw_size_update < raw_step:
            # Fits entirely in current bar
            builder.update(price, raw_size_update, ts_init)
            break

        # Fill current bar to threshold
        raw_size_diff = raw_step - builder.volume
        builder.update(price, raw_size_diff, ts_init)
        build_now_and_send()  ← emit completed bar

        # Remaining size carries over to next bar
        raw_size_update -= raw_size_diff

Visual example: step=100, current volume=80

Tick arrives: price=3500, size=50

  Current bar volume: 80 / 100
  ├── 20 units fill current bar to 100 → BAR EMITTED!
  └── 30 units start the next bar (volume=30/100)

  One tick created two bar states — no data lost!
Why this matters: In high-volume markets, a single large trade (e.g., 500 BTC) could span multiple volume bars. The aggregator correctly splits it, ensuring each bar contains exactly the specified volume threshold.

Other Bar Types

TickBarAggregator (line 400)

Simplest aggregator. Counts ticks and emits when count == step.

cdef void _apply_update(self, Price price, Quantity size, uint64_t ts_init):
    self._builder.update(price, size, ts_init)

    if self._builder.count == self.bar_type.spec.step:   # e.g., every 100 ticks
        self._build_now_and_send()

TickImbalanceBarAggregator (line 447)

Tracks the imbalance between buy and sell ticks. Emits when |imbalance| reaches threshold.

cpdef void handle_trade_tick(self, TradeTick tick):
    self._apply_update(tick.price, tick.size, tick.ts_init)

    if tick.aggressor_side == BUYER:
        self._imbalance += 1
    else:
        self._imbalance -= 1

    if abs(self._imbalance) >= self.bar_type.spec.step:
        self._build_now_and_send()
        self._imbalance = 0  # reset

TickRunsBarAggregator (line 509)

Counts consecutive ticks on the same side. Resets run counter when aggressor side changes.

cpdef void handle_trade_tick(self, TradeTick tick):
    if side != self._current_run_side:
        self._current_run_side = side
        self._run_count = 0        # new side, reset run
        self._builder.reset()

    self._apply_update(tick.price, tick.size, tick.ts_init)
    self._run_count += 1

    if self._run_count >= self.bar_type.spec.step:
        self._build_now_and_send()
        self._run_count = 0

RenkoBarAggregator

Emits a bar when price moves a fixed amount (the "brick size") from the last bar's close. Useful for filtering noise and focusing on significant price movements.

Key Takeaways

1. No traditional database for market data. Parquet files (columnar, compressed, fast reads) — not SQL, MongoDB, or InfluxDB.
2. Redis is only for trading state (orders, positions, accounts) and is optional. Without Redis config, everything runs in-memory only.
3. Three storage tiers exist simultaneously:
• In-memory Cache — fastest, limited (deques of 10K)
• Parquet Catalog — disk, unlimited, for backtesting
• Redis — optional, for crash recovery
4. Tick-to-bar conversion happens via BarAggregator subclasses. The BarBuilder accumulates OHLCV; each aggregator decides when to emit (time, tick count, volume, value, imbalance, or runs).
5. Volume/Value bars intelligently split ticks across bar boundaries — a single large tick can complete one bar and start the next. No data is ever lost or double-counted.
