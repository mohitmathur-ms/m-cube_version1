Path A vs Path B — In Plain English

1. What is a backtest?

A backtest is a simulation. You take a trading strategy — say, "buy when the 10-day moving average crosses above the 20-day moving average" — and you replay old market data through it. The simulator pretends each historical price tick is happening live, tells the strategy about it, lets the strategy place fake orders, and tracks what would have happened.

At the end you get a P&L curve, a list of trades, and a sense of whether the strategy would have made or lost money if it had been running back then.

Our project runs backtests using a Python library called NautilusTrader. Nautilus offers two different ways to set up a backtest. We call them Path A and Path B.

2. The two paths — an analogy

Imagine you want to cook dinner.

There are two ways to do it.

Path A — cook from scratch. You pick out every ingredient yourself, measure them, set up the stove, decide the order of operations, and hand-stir at every step. You're in full control of every detail.

Path B — order a meal kit. You fill out a form ("I want pasta with chicken, for 2 people, ready by 7pm"), hand it to the meal-kit company, and they handle the assembly. The food comes out the same, but you spent your effort describing what you wanted instead of doing each step yourself.

Both paths produce dinner. The difference is how much manual work you do versus how much the system does for you based on a description.

Backtests work the same way:

  • Path A You build the engine, load the data, attach the strategy, and press “run” — one step at a time, by hand. This is what our project does today.
  • Path B You fill out three small forms describing what you want, hand them to a "node," and it handles the assembly. The result is the same; the wiring is automatic.

3. Path A — what we use today

Today, every backtest in our project follows the same five-step recipe. It's done by hand, in code, in three places: run_backtest(), _run_single_slot(), and _run_slot_group() — all in core/backtest_runner.py.

The five steps

1
Build the engine. The "engine" is the simulator itself. Think of it as the empty stove. We set its basic options (logging, risk checks) and turn it on.
2
Add the venue. A "venue" is the simulated exchange — e.g., Binance for crypto, FOREX_MS for forex. We tell the engine the account type, starting cash, and leverage.
3
Load the data. We open our parquet files, read the historical bars (price candles) into memory as a Python list, and hand the whole list to the engine.
4
Attach the strategy. We construct the strategy object (e.g., EMA crossover with periods 10 and 20) and tell the engine to use it.
5
Run. The engine walks through the bars in time order, calls the strategy on each one, simulates fills for any orders the strategy places, and records the results.

What the code looks like (simplified)

engine = BacktestEngine(config=engine_config)        # step 1

engine.add_venue(                                     # step 2
    venue=Venue("FOREX_MS"),
    starting_balances=[Money(1_000_000, USD)],
    ...
)

bars = catalog.bars(...)                              # step 3 (load)
engine.add_data(bars)                                 # step 3 (hand to engine)

strategy = EMACross(strategy_config)                  # step 4 (build)
engine.add_strategy(strategy)                         # step 4 (attach)

engine.run()                                          # step 5

Five method calls. Everything happens in our Python code.

Key thing about Path A: all the bars are loaded into memory at once, before the run starts. For a 5-year EURUSD 1-minute backtest that's about 1.8 million bars sitting in RAM the entire run.

Where this lives in our code

  • core/backtest_runner.py:141 — run_backtest() handles single one-off backtests.
  • core/backtest_runner.py:355 — _run_single_slot() runs one strategy slot inside a portfolio backtest.
  • core/backtest_runner.py:718 — _run_slot_group() runs multiple strategies in one shared engine (when _USE_GROUPING=1).

Each of these three functions does the same five steps with slight variations. Today they're three near-copies of the same recipe.

4. Path B — what we'd switch to

Path B uses a class called BacktestNode instead of BacktestEngine. Same library, same package — we already have it installed. The class lives at venv/Lib/site-packages/nautilus_trader/backtest/node.py. We don't write it; we just import it.

Instead of calling five methods on an engine, we fill out three "forms" (called configs) that describe what we want, hand them to the node, and the node does the wiring for us.

The three forms (configs)

FormWhat you put on itReplaces…
BacktestVenueConfig Venue name, account type, starting cash, leverage. The engine.add_venue(...) call from Path A step 2.
BacktestDataConfig Path to the parquet folder, which bar types to load, date range. The manual catalog.bars(...) + engine.add_data(...) from step 3.
BacktestEngineConfig Trader ID, logging settings, risk-check settings. The argument we already pass to BacktestEngine(config=...) in step 1 — but now embedded inside the run config.

You then bundle all three into one BacktestRunConfig and hand it to BacktestNode.

What the code looks like (simplified)

venue_cfg = BacktestVenueConfig(                      # form 1
    name="FOREX_MS",
    starting_balances=["1000000 USD"],
    ...
)

data_cfg = BacktestDataConfig(                        # form 2
    catalog_path="./catalog",
    bar_types=["EURUSD-1-MINUTE-MID-EXTERNAL", ...],
    start_time="2024-01-01",
    end_time="2024-12-31",
)

engine_cfg = BacktestEngineConfig(...)                # form 3

run_cfg = BacktestRunConfig(                          # bundle
    venues=[venue_cfg],
    data=[data_cfg],
    engine=engine_cfg,
    chunk_size=100_000,                          # optional: stream 100k bars at a time
)

node = BacktestNode(configs=[run_cfg])                # hand to node
node.build()                                          # node assembles the engine
engine = node.get_engine(run_cfg.id)                  # grab the engine it built
engine.add_strategy(strategy)                         # attach strategy (same as today)
node.run()                                            # run!
The key new feature: chunk_size. With chunk_size=100_000, the node loads only 100,000 bars into memory at a time. It processes them, drops them, loads the next 100,000, and so on. Peak memory stays flat regardless of how long the backtest is.

Why we still need node.build() and engine.add_strategy()

In a "fully config-driven" world, you'd describe the strategy on the form too — and the node would build and attach it for you. But our project loads custom strategies from user-supplied directories at runtime, and sometimes wraps them in ManagedExitStrategy for stop-loss/take-profit logic. That kind of dynamic construction can't be expressed on a form.

So we use a hybrid: forms for the engine/venue/data setup, and a one-line manual attach for the strategy. The strategy attach is exactly the same line as today: engine.add_strategy(strategy).

6. What would change in our code

Three categories: things to add, things to keep as-is, and things we'd eventually delete.

Things to add (new code, ~150 lines total)

  • One shared helper — _build_run_config(). Builds the three configs from our existing arguments. About 25 lines. Defined once, used by all three call sites.
  • Three new functions — run_backtest_node(), _run_single_slot_node(), _run_slot_group_node(). Path B versions of our current functions. About 40-60 lines each.
  • One env-flag check at the top of each public function: 
    if os.environ.get("_USE_BACKTEST_NODE") == "1":
    return run_backtest_node(...)
# otherwise fall through to existing Path A code
    This lets us toggle Path B on per-run without removing Path A. We can flip the flag, validate results, then flip back if anything looks off.
  • Four new imports at the top of backtest_runner.py: BacktestNode, BacktestVenueConfig, BacktestDataConfig, BacktestRunConfig.

Things to keep exactly as-is

  • The HTTP server — server.py calls run_backtest() and run_portfolio_backtest(). Those public function names don't change; only their internals do.
  • The parquet writer — core/nautilus_loader.py. It already writes data in the format Path B reads.
  • Result extraction — _extract_results() reads from a BacktestEngine object. Path B gives you back a BacktestEngine object via node.get_engine(...), so this code keeps working unchanged.
  • The strategy registry — how we load custom strategies, build their configs, and wrap them in ManagedExitStrategy. None of that changes.
  • The parallel-worker pool — ProcessPoolExecutor still runs slots in parallel. Each worker just builds a BacktestNode instead of a BacktestEngine.
  • FX BID/ASK auto-pairing — the _pair_bid_ask_bar_type() helper still gets called. Its output now feeds BacktestDataConfig.bar_types instead of a Python list passed to add_data().

Things to remove (eventually, after Path B becomes default)

  • The current Path A function bodies in run_backtest() / _run_single_slot() / _run_slot_group().
  • The _cached_catalog_bars() LRU cache helper — it caches in-memory bar lists, which Path B doesn't need.
  • The engine.dispose() calls and try/finally blocks — Path B disposes engines automatically.

None of these get removed up-front. We keep both paths side-by-side until the new one is fully validated, then clean up in a follow-up.

7. What stays exactly the same

From a user's perspective, nothing visible changes. The HTTP API responds the same way. The result HTML reports look the same. The strategies run the same way. P&L numbers should be identical (or differ only by tiny floating-point noise if streaming is enabled).

  • Strategy code is unchanged.
  • Parquet files on disk are unchanged.
  • HTTP endpoints are unchanged.
  • Result reports are unchanged.
  • Parallel-run scheduling is unchanged.
  • FX rate conversion is unchanged.

The migration is purely an internal wiring change. Outside observers can't tell the difference.

8. Should we actually switch?

Reasons to switch

  • Memory ceiling on long backtests. A 5-year minute-bar EURUSD backtest with BID + ASK loaded uses around 1 GB of RAM. With chunk_size set, peak RAM stays around 15-50 MB regardless of date range. If you're hitting RAM limits, this is the fix.
  • Reproducibility. The configs are plain serializable objects. You can save a backtest's config to JSON, store it next to the result, and rerun the exact same backtest months later from the saved config alone.
  • Cleaner code. Three call sites that today each repeat the same setup will share one helper. Less drift, fewer bugs like the one we just fixed in _run_slot_group() where one copy diverged from the others.

Reasons not to switch (yet)

  • The current code already works. It's well-instrumented, tested in production, and memory-conscious (del bars after add_data). If you're not feeling pain today, the win is mostly aesthetic.
  • The cache helper goes away. The worker-local LRU cache that today saves ~500 ms on repeated runs of the same date range disappears. This matters in CI scenarios where the same portfolio is run many times back-to-back.
  • Real engineering cost. About 1-2 days of focused work for the migration plus a soak period validating Path A and Path B produce identical results across our portfolio set. Not free.
Honest recommendation: If we're hitting memory limits on long backtests, or if reproducibility (saving and rerunning exact configs) becomes important, switch — but do it gated behind _USE_BACKTEST_NODE=1 so we can validate side-by-side before flipping the default. If we're not feeling either pain point, leave it. The existing code is fine.

9. Glossary

Bar
One price candle. Has open, high, low, close, volume, and a timestamp. The basic unit a strategy reacts to.
BarType
A string identifier like "EURUSD.FOREX_MS-1-MINUTE-MID-EXTERNAL". Encodes the instrument, the time interval, and which side (BID/ASK/MID/LAST).
Catalog (Parquet catalog)
A folder on disk where bars are stored as Parquet files, organized by bar type. Our project writes to it via nautilus_loader.py; both Path A and Path B read from it.
Engine (BacktestEngine)
The actual simulator. Walks through bars in time order, calls the strategy on each one, simulates fills, tracks P&L. Both paths use it internally; the difference is whether we construct it directly (Path A) or have a node construct it for us (Path B).
Node (BacktestNode)
A wrapper class introduced in Path B. Reads config objects, builds engines from them, runs them, and gives us back the engine handle for result extraction. We don't write the node; we import it from Nautilus.
Strategy
The trading logic itself — e.g., "buy when EMA(10) crosses above EMA(20)". Receives bars via callbacks, places orders. Doesn't care which path is used to run it.
Venue
The simulated exchange — e.g., "FOREX_MS" for forex, "BINANCE" for crypto. Determines fee structure, fill rules, account type.
chunk_size
A Path B-only setting. Controls how many bars the node holds in memory at once. None = load all at once (same as Path A). A number like 100_000 = stream that many bars at a time.
OMS type
"Order Management System" type. Two options: NETTING merges all orders on one instrument into a single position. HEDGING tracks each strategy's position independently. We use HEDGING when running multiple strategies in one shared engine.
Matching engine
The part of the engine that decides whether and at what price an order fills. For FX, it needs both BID and ASK bars loaded so it can fill at realistic spread prices.
