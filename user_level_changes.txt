# Plan: Multi-user identity layer (Option B)

## Context

The system today has zero notion of users — one set of portfolios, one sizing config, no identity at any layer. The user wants multiple traders to "play along" on different instruments from separate machines, with a central admin that:

- defines each user (alias, user_id),
- assigns a per-user **multiplier** that scales effective trade size,
- restricts each user to an **allowed_instruments** list.

Users picked **identity-only, no security**: every request carries an `X-User-Id` header — no passwords, no sessions. This is appropriate for trusted internal teams; anyone on the network can spoof, and we'll document that. Total scope: ~1–2 days, contained.

The end goal: each user runs the same Flask app from their own machine, hits a shared backend, sees only their own portfolios, has their effective trade size scaled by their personal multiplier (on top of the admin's per-instrument policy), and is rejected when they try to add an instrument outside their allowlist. The admin manages all users from the existing admin panel (port 5001).

## Design overview

### Identity flow

```
Browser (alice's machine)              Server
─────────────────────────              ──────────────────────────
On first load:                         GET /api/users/list →
  GET /api/users/list                    [{user_id, alias}, ...]
  Show picker → user clicks "alice"
  localStorage["user_id"] = "alice"

On every API call:
  fetch(url, { headers: {              Validate header against
    "X-User-Id": "alice"                config/users.json
  }})                                   →  401 if missing/unknown
                                        →  Inject user object into
                                            request context
                                        →  Scope state by user_id
```

### Effective trade size — final formula

```
qty = lots × lot_size × resolved_trade_size × user_multiplier
                         │                     │
                         └─ slot override or  └─ from users.json,
                            admin default,       defaults to 1.0
                            clamped by admin
                            max_size LAST
```

**Cap order matters**: apply user_multiplier first, then admin `max_size` clamp. This makes the admin cap a true hard ceiling — a malicious or mistaken multiplier can't bust through.

## Data model

### `config/users.json` (new, admin-managed)

```json
{
  "_meta": {
    "description": "Identity-only user registry. The X-User-Id header on each API call must match a user_id here. No auth/security — for trusted environments.",
    "schema_version": 1
  },
  "users": [
    {
      "user_id": "alice",
      "alias": "Alice — FX Desk",
      "multiplier": 1.0,
      "allowed_instruments": ["EURUSD", "GBPUSD", "USDJPY"]
    },
    {
      "user_id": "bob",
      "alias": "Bob — Crypto Desk",
      "multiplier": 0.5,
      "allowed_instruments": null
    }
  ]
}
```

- `user_id`: URL-safe `[a-z0-9_-]{1,32}` (used as path component)
- `multiplier`: `> 0`. Defaults to `1.0` if omitted or invalid
- `allowed_instruments`: `null` = all instruments; `list[str]` = whitelist of bare symbols (`"EURUSD"`, not full bar_type strings)

### State partitioning

| Resource | Today | After |
|---|---|---|
| `portfolios/*.json` | Flat | `portfolios/<user_id>/*.json` |
| `reports/*.csv` | Flat | `reports/<user_id>/*.csv` |
| `config/instrument_sizing.json` | Shared | **Stay shared** (admin policy) |
| `catalog/` (market data) | Shared | **Stay shared** (read-only data) |
| `custom_strategies/` | Shared | **Stay shared** (admin-uploaded) |
| `custom_adapters/`, `adapters_config/`, `data_formats/` | Shared | **Stay shared** (admin-managed) |
| `.runtime_history.json` | Shared | **Stay shared** (system-wide perf telemetry) |

Migration: one-shot script moves existing flat `portfolios/*.json` → `portfolios/_default/*.json` and seeds a `_default` user, so legacy data is reachable on day one without admin setup.

## Migration

On server startup (`server.py` near `PORTFOLIOS_DIR` definition, line 1053):

1. If `portfolios/` contains `*.json` files at root level (legacy flat layout):
   - Ensure `config/users.json` exists with at least a `_default` user (create if missing)
   - Move `portfolios/*.json` → `portfolios/_default/*.json`
   - Log the migration once
2. If `portfolios/` is already partitioned (subdirs with json files), skip
3. Idempotent — safe to run multiple times

Frontend: existing browsers with empty localStorage will be sent to the picker on first load and pick `_default` (or whatever the admin renames it to).

## Verification

End-to-end test plan once implementation is done:

### 1. Resolver math (`core/instrument_sizing.py`)
```python
# alice has multiplier=2.0; EURUSD admin max_size=5.0
slot = StrategySlotConfig(bar_type_str="EURUSD...", trade_size=2.0)
assert effective_trade_size(slot, user_id="alice") == 4.0   # 2.0 × 2.0
slot.trade_size = 10.0
assert effective_trade_size(slot, user_id="alice") == 5.0   # 10×2=20 → clamped to 5
```

### 2. Header validation (Flask test client)
- Hit `GET /api/portfolios/list` with no header → 401
- With unknown user_id → 401
- With known user_id → 200, returns only that user's portfolios

### 3. Per-user isolation
- alice POSTs portfolio "P1"
- bob's `GET /api/portfolios/list` does NOT include "P1"
- bob can also create his own "P1" without conflict (different directory)

### 4. Allowlist enforcement
- bob's allowed_instruments = `["BTCUSD"]`
- bob POSTs portfolio with EURUSD slot → 403 with descriptive error

### 5. Admin CRUD round-trip
- Admin POST `/api/users` with new user "carol" → 200
- Main server `GET /api/users/list` includes "carol"
- carol can immediately authenticate and create portfolios

### 6. Migration
- Stash a flat `portfolios/test_legacy.json` before startup
- Start server
- Verify file moved to `portfolios/_default/test_legacy.json`
- `GET /api/portfolios/list` as `_default` user returns it

### 7. Manual browser test
- Open two browsers, set localStorage `user_id` to alice and bob
- Verify each sees only their own portfolios in the UI
- Switch alice's user picker → portfolios list re-fetches and updates

### 8. Documentation note (no code)
- Add a clear warning in `config/users.json._meta` that this is identity-only, not security; anyone on the network can spoof headers.

## Out of scope (deliberate)

- Real auth (passwords/sessions) — explicitly chosen against. Can be layered on later by replacing the header check with `flask_login.current_user.id`; everything downstream stays the same.
- Per-user `max_size` overrides — explicitly excluded by user from feature picks.
- Per-user `starting_capital` cap — same.
- Audit logging of who-did-what — not requested. The `X-User-Id` header is logged by Flask access logs anyway.
- Sharing portfolios between users — not requested. If needed later, add a `shared_with: list[user_id]` field on the portfolio.

## Risk callouts

1. **Spoofability**: explicit and accepted. Document loudly in `users.json._meta` and on the picker UI.
2. **localStorage clearing** silently re-sends user to picker — fine, recoverable.
3. **Cap order**: applying admin `max_size` AFTER user multiplier is a deliberate choice (admin authority is final). Document in `resolve_trade_size` docstring.
4. **Allowlist symbol matching**: must match the same way as `instrument_sizing._symbol_key` (split on `.`, uppercase). Reuse that helper, don't reinvent.
5. **`_default` user collision**: if admin renames `_default` to something else after migration, existing portfolios become orphaned. Mitigation: don't allow renaming `user_id`; only `alias` is editable.
