M_Cube Crypto Dashboard

A beginner-friendly guide to understanding the entire project, inside and out.

What Does This App Do?

In simple words, this app lets you:

  1. Import historical cryptocurrency price data (like Bitcoin, Ethereum, etc.) from CSV files on your computer
  2. Explore that data with interactive charts (candlestick charts, price tables, return analysis)
  3. Test trading strategies — "what if I had followed this trading rule for the last 5 years?" (this is called backtesting)
  4. Analyze the results — see how much profit/loss each strategy would have made, visualize performance with charts
Imagine a flight simulator for trading. Instead of risking real money, you feed the app historical price data and test your trading ideas against the past. The app shows you exactly what would have happened — how many trades you'd have made, how much you'd have won or lost, and when things went wrong.

How the App is Built (The Big Picture)

The app has two main halves that talk to each other:

Frontend (What You See)

This is the part that runs in your web browser. It's built with:

  • HTML — the structure of the pages (buttons, inputs, layout)
  • CSS — the styling (dark theme, colors, spacing)
  • JavaScript — the logic that makes things interactive (click a button, see a chart)
  • Plotly.js — a library that draws the interactive charts
Backend (What Does the Work)

This is the part that runs on your computer as a Python program. It handles:

  • Flask — a Python web framework that serves the website and handles requests
  • NautilusTrader — a professional trading simulation engine
  • Pandas — a Python library for processing data tables
  • File system — reading CSVs, saving reports, managing the data catalog
The frontend is like a restaurant menu — it shows you options and takes your order. The backend is like the kitchen — it does the actual cooking (loading data, running backtests) and sends the results back to your table.

How to Run the App (Step by Step)

Install Python Dependencies

Open a terminal and run:

pip install flask flask-cors pandas nautilus_trader

This installs the Python libraries the backend needs.

Place Your CSV Data Files

Put your crypto CSV files in a folder on your computer. The default location the app looks for is:
C:\Users\ADMIN\Desktop\id_name_all_symbols\id_name_all_symbols
But you can change this in the app. CSV files should be named like: 1_BTC_Bitcoin.csv

Start the Server

python crypto_data_app_using_html/server.py

You'll see a message saying the server is running on http://localhost:5000

Open Your Browser

Go to http://localhost:5000 in Chrome, Firefox, or Edge. You'll see the dashboard!

Glossary of Terms

If you see a word you don't know, look it up here. Terms are organized by category.

Trading & Finance Terms

TermPlain English ExplanationExample
OHLCV Stands for Open, High, Low, Close, Volume. These are the 5 numbers that describe a single day of trading. Open = price at start of day, High = highest price, Low = lowest, Close = price at end of day, Volume = how much was traded. Bitcoin on Jan 1: Open=$42,000, High=$43,500, Low=$41,800, Close=$43,100, Volume=50,000 BTC
Candlestick Chart A chart where each "candle" shows one day's OHLCV. The body shows open-to-close, the "wicks" show high and low. Green = price went up, Red = price went down. A tall green candle means the price rose a lot that day
Backtest Testing a trading strategy on historical data to see how it would have performed. Like rewinding time and asking "what if?" "If I bought BTC every time the RSI went below 30 since 2015, how much would I have made?"
Strategy A set of rules that decide when to buy and when to sell. No emotions, just math. "Buy when the 10-day average crosses above the 20-day average"
EMA (Exponential Moving Average) An average of recent prices that gives more weight to recent days. A "fast" EMA (short period) reacts quickly to price changes; a "slow" EMA (long period) is smoother. 10-day EMA reacts faster than 50-day EMA
RSI (Relative Strength Index) A number from 0-100 that measures if something is "overbought" (too expensive, might drop) or "oversold" (too cheap, might rise). Typically: above 70 = overbought, below 30 = oversold. RSI at 25 suggests the asset might be a good buy
Bollinger Bands A channel around the price made of an average line plus upper and lower bands. When price touches the lower band, it might bounce up. When it hits the upper band, it might drop. Price hitting the lower band = potential buy signal
P&L (Profit & Loss) How much money a trade or strategy made or lost. Positive = profit, negative = loss. P&L of +$5,000 means you made five thousand dollars
Drawdown The drop from a peak balance to a trough. Shows the worst "paper loss" you'd have experienced. Max drawdown is the biggest drop. Balance went from $110K to $95K = drawdown of -13.6%
Win Rate What percentage of your trades made money. 24 wins out of 42 trades = 57.1% win rate
Equity Curve A line chart showing how your account balance changed over time. Goes up when you profit, down when you lose. A steadily rising equity curve = consistently profitable strategy
Tearsheet A one-page summary of a strategy's performance with key metrics and charts. Like a report card for your trading strategy. Shows return %, drawdown, win rate, and trade analysis all in one view
Trade Size How many units (e.g., how many Bitcoin) to buy or sell per trade. Trade size of 1 = buy/sell 1 BTC at a time
Starting Capital How much fake money you start with in the simulation. $100,000 starting capital is the default

Technical / Programming Terms

TermPlain English Explanation
API (Application Programming Interface) A set of URLs that the frontend uses to ask the backend for data. When you click "Load Data", the browser sends a message to the backend saying "please load this CSV" — that message is an API call.
REST API A specific style of API that uses standard HTTP methods (GET = retrieve data, POST = send data). All our API endpoints follow this pattern.
Endpoint A specific URL that the backend listens on. For example, /api/data/bars is an endpoint that returns price data.
JSON A text format for sending structured data between frontend and backend. Looks like: {"name": "Bitcoin", "price": 83000}
Flask A lightweight Python web framework. It lets you write Python functions that respond to web requests. Think of it as the "waiter" that connects the browser to our Python code.
SPA (Single Page Application) A website that loads one HTML page and then swaps content dynamically using JavaScript, instead of loading a new page each time you click something. Our app is an SPA.
DOM (Document Object Model) The browser's internal representation of the HTML page. When JavaScript "updates the DOM", it's changing what you see on screen.
NautilusTrader A professional-grade Python library for running trading simulations (backtests). It handles order execution, position tracking, and performance calculations.
Parquet Catalog A high-performance storage format. Instead of reading CSV files every time, we convert the data once into Parquet format (stored in the catalog/ folder) which is much faster to read later.
DataFrame A table of data in Python (from the pandas library). Think of it like a spreadsheet in code — rows and columns you can filter, sort, and analyze.
CORS (Cross-Origin Resource Sharing) A security feature in browsers. It controls whether a website can talk to a different server. We enable it so the frontend can talk to the Flask backend.
CSS Variables Reusable color/size values defined once and used everywhere. For example, --accent: #00d4aa defines our teal accent color. Change it once, and the whole app updates.
Plotly.js A JavaScript charting library loaded from a CDN (internet link). It draws all the interactive charts — candlesticks, histograms, pie charts, etc.
Bar (in NautilusTrader) One unit of price data. In our case, one Bar = one day of OHLCV data. The string BTCUSD.CRYPTO-1-DAY-LAST-EXTERNAL means "Bitcoin/USD, 1-day bars, last price, from an external source".

What Happens When You Click Something?

Every user action follows this sequence. Here's an example: you click "Load Data" for Bitcoin:

You click the "BTC - Bitcoin" button

This triggers a JavaScript function in load_data.js

JavaScript sends a request to the backend

Specifically, it sends a POST request to /api/csv/load with the Bitcoin CSV entry details

Flask receives the request

The csv_load() function in server.py handles it

Flask calls the core modules

load_csv_and_store() reads the CSV, creates NautilusTrader objects, and saves to the catalog

Flask sends back results as JSON

Something like: {"symbol": "BTCUSD", "num_bars": 4200, "latest_close": 83000}

JavaScript updates the page

It creates metric cards, data tables, and success messages — all without reloading the page

How Page Navigation Works

This is a Single Page Application (SPA). That means:

  • Only one HTML file (index.html) is ever loaded
  • When you click a sidebar link, JavaScript replaces the content inside the main area
  • The page never fully reloads — it just swaps what's visible

Here's the code that makes this work (in app.js):

// When you click a sidebar link:
navigate(page) {
    const main = document.getElementById("main-content");  // Find the content area
    switch (page) {
        case "dashboard":   Dashboard.render(main); break;  // Replace content with Dashboard
        case "load_data":   LoadData.render(main); break;   // Replace content with Load Data
        case "view_data":   ViewData.render(main); break;   // ...and so on
        case "backtest":    Backtest.render(main); break;
        case "tearsheet":   Tearsheet.render(main); break;
    }
}

How Data Persists Between Pages

When you run a backtest and then go to the Tearsheet page, how does Tearsheet know your results?

The answer is shared state stored in App.state:

// In backtest.js - after backtest completes:
App.state.backtestResults = data.results;   // Save results to shared state
App.state.backtestConfig = { bar_type, strategies };

// In tearsheet.js - when the page loads:
const results = App.state.backtestResults;  // Read from shared state
Important for beginners: This state only lives in your browser's memory. If you refresh the page (F5), the state is lost and you'll need to run the backtest again. It does NOT save to disk automatically.

All API Endpoints (What the Server Can Do)

Each endpoint is like a specific service the server offers. The browser calls these URLs to get things done:

GET /api/catalog/status

What it does: Checks if the data catalog exists and what data is in it.

When it's called: When Dashboard or Load Data pages load.

What it returns:

{ "exists": true, "data_types": ["Bar(BTCUSD.CRYPTO-1-DAY-LAST-EXTERNAL)", ...] }

GET /api/csv/scan

What it does: Looks through a folder on your computer and finds all CSV files that contain crypto data.

When it's called: When you go to Load Data page (automatically) or click "Scan Folder".

What it returns:

{ "entries": [{"id": 1, "symbol": "BTC", "name": "Bitcoin", "path": "...", "label": "BTC - Bitcoin"}], "count": 42 }

POST /api/csv/load

What it does: Takes CSV entries you selected and loads them into the NautilusTrader catalog. This is the heavy lifting — it reads the CSV, creates trading instruments, and saves to Parquet format.

When it's called: When you click "Load Selected into Catalog" or a quick-select button.

What you send:

{ "entries": [{"id": 1, "symbol": "BTC", "name": "Bitcoin", "path": "..."}] }

What it returns:

{ "results": [{"symbol": "BTCUSD", "num_bars": 4200, "date_start": "2013-04-28", "date_end": "2026-04-06", "latest_close": 83000.50, "sample_data": [...]}], "errors": [] }

GET /api/data/bar_types

What it does: Returns a list of all instruments you've loaded (e.g., "BTCUSD", "ETHUSD").

What it returns:

{ "bar_types": ["BTCUSD.CRYPTO-1-DAY-LAST-EXTERNAL", "ETHUSD.CRYPTO-1-DAY-LAST-EXTERNAL"] }

GET /api/data/bars

What it does: Returns all the OHLCV price data for a specific instrument.

What it returns:

{ "data": [{"timestamp": "2013-04-28T00:00:00+00:00", "open": 135.3, "high": 135.98, "low": 132.1, "close": 134.21, "volume": 100000.0}, ...], "count": 4200 }

GET /api/strategies

What it does: Returns all available trading strategies (both built-in and any custom ones you uploaded) with their parameter definitions.

What it returns:

{ "strategies": {"EMA Cross": {"description": "...", "params": {"fast_ema_period": {"label": "Fast EMA", "default": 10, "min": 2, "max": 500}}}}, "warnings": [] }

POST /api/custom_strategies/upload

What it does: Lets you upload your own trading strategy as a Python file. It validates the file in a separate process (to avoid crashes), checks for required exports, and saves it.

Safety measure: Validation runs in a subprocess — if the uploaded code has errors, it crashes the subprocess, not the main server.

POST /api/backtest/run

What it does: The most complex endpoint. It runs one or more trading strategies on your data and returns detailed results.

What you send:

{
    "bar_type": "BTCUSD.CRYPTO-1-DAY-LAST-EXTERNAL",
    "starting_capital": 100000,
    "strategies": {
        "EMA Cross": { "params": {"fast_ema_period": 10, "slow_ema_period": 20}, "trade_size": 1 }
    }
}

What it does internally (step by step):

  1. Merges built-in and custom strategies into one registry
  2. For each strategy: runs the backtest simulation via NautilusTrader
  3. Collects results: P&L, trades, fills, positions, account balance
  4. Generates an HTML report and saves it to reports/
  5. Saves individual CSV reports per strategy (positions, fills, account, orders, logs)
  6. Returns everything as JSON to the browser

strategies.py — "The Strategy Library"

Job: Contains 4 pre-built trading strategies. Each strategy is a set of rules that tells the backtester when to buy and sell.

StrategyHow It Works (Beginner Explanation)Default Parameters
EMA Cross Tracks two moving averages — a fast one (recent trend) and a slow one (longer trend). Buys when the fast average crosses above the slow (uptrend starting). Sells when it crosses below (downtrend starting). Fast: 10 days, Slow: 20 days
RSI Mean Reversion Uses the RSI indicator (0-100 scale). Buys when RSI drops below 30 (oversold = "too cheap"). Sells when RSI goes above 70 (overbought = "too expensive"). Bets that prices will return to normal. Period: 14, Overbought: 70, Oversold: 30
Bollinger Bands Creates a channel around the price using standard deviation. Buys when price touches the lower band (price is unusually low). Sells when it touches the upper band (price is unusually high). Period: 20, Std Dev: 2.0
4 Moving Averages Uses 4 moving averages of different speeds. Buys when all 4 are in order (fastest on top = strong uptrend). Sells when the alignment breaks. Periods: 10, 20, 50, 200 (Use EMA: Yes)

All strategies have these in common:

  • They use Market Orders (buy/sell at current price, no waiting)
  • They require a trade_size parameter (how many units per trade)
  • On stop, they cancel all open orders and close all positions

// Daily returns (percentage change day-to-day):

How Key Metrics Are Calculated

// Equity curve: starting capital + cumulative sum of P&L per trade
equity[0] = starting_capital + pnl[0]
equity[1] = starting_capital + pnl[0] + pnl[1]
... and so on

// Drawdown: how far equity has fallen from its peak
peak = highest equity seen so far
drawdown[i] = ((equity[i] - peak) / peak) * 100

// Trade statistics:
avgWin    = total profit from winning trades / number of wins
avgLoss   = total loss from losing trades / number of losses
bestTrade = largest single P&L value
worstTrade = smallest (most negative) single P&L value
maxDrawdown = the most negative drawdown value

Optimization Suggestions

Ways to make the app faster, safer, and better — sorted by importance

How to read this section:
PERFORMANCE = Make it faster
SECURITY = Make it safer
UX = Make it easier to use
ARCHITECTURE = Make the code better organized
MAINTAINABILITY = Make it easier to change in the future

High Priority (Fix These First)

PERFORMANCE 1. All Price Data Loaded at Once

The problem: When you view data, the server sends ALL bars (potentially thousands) in one giant JSON response. For years of daily data, this can be slow.

Why it matters: Users on slower connections will wait a long time. The browser might lag with huge datasets.

How to fix: Add pagination (load 500 bars at a time) or server-side date filtering so the browser only gets what it needs.

SECURITY 2. Debug Mode is On

The problem: debug=True in server.py shows detailed error pages with your code. If anyone on your network accesses the server, they can see your source code and even run Python commands.

Why it matters: This is the #1 security risk in the entire app.

How to fix: Change to debug=False for production, or use an environment variable to control it.

SECURITY 3. Server Accepts Connections from Anywhere

The problem: host="0.0.0.0" + CORS(app) with no restrictions means anyone on your network can access the app and make API calls.

How to fix: Change host to "127.0.0.1" (localhost only) and restrict CORS origins.

PERFORMANCE 4. Backtests Block the Server

The problem: When a backtest is running, the server can't handle any other requests. If you run 5 strategies, the server is completely blocked until all 5 finish.

How to fix: Run backtests in a background thread/process and let the frontend poll for progress.

SECURITY 5. Custom Strategy Files Can Run Any Code

The problem: When you upload a .py file, it eventually gets imported and executed. A malicious file could delete files, steal data, etc.

How to fix: Add code scanning (block dangerous imports like os, subprocess), run strategies in a sandbox, or use Docker containers.

Medium Priority (Nice to Have)

UX 6. Results Lost on Page Refresh

The problem: If you refresh the browser, all backtest results are gone. You have to re-run the backtest.

How to fix: Save results to sessionStorage (browser storage) so they survive a refresh.

PERFORMANCE 7. Catalog Reloaded Every API Call

The problem: Every time the server needs data, it reopens the Parquet catalog from disk. This is wasteful when the data hasn't changed.

How to fix: Cache the catalog in memory and only reload when new data is loaded.

UX 8. Progress Bar Isn't Real

The problem: The backtest progress bar shows 30% while waiting, then jumps to 100%. It doesn't reflect actual progress.

How to fix: Run each strategy individually and update progress after each one completes.

ARCHITECTURE 9. Sidebar Catalog Path Does Nothing

The problem: There's a "Catalog Storage Path" input in the sidebar, but no JavaScript code reads it. It's decorative.

How to fix: Either connect it to API calls or remove it to avoid confusing users.

PERFORMANCE 10. HTML Report Sent Inside JSON

The problem: The entire HTML report (can be hundreds of KB) is sent as part of the backtest response JSON, even though it's only needed when the user clicks "Download Report".

How to fix: Save the report on the server and return just a download URL.

UX 11. No Bookmarkable URLs

The problem: All pages share the same URL (localhost:5000). You can't bookmark the backtest page or share a link to a specific page.

How to fix: Add hash-based routing (e.g., localhost:5000/#/backtest).

Low Priority (Polish)

MAINTAINABILITY 12. Hardcoded CSV Path

The default CSV folder path only works on the developer's computer. Use a config file or environment variable instead.

MAINTAINABILITY 13. Global JavaScript Objects

All JS modules are global variables. This can cause naming conflicts. Migrating to ES modules (import/export) would be cleaner.

UX 14-16. Minor UX Improvements
  • Add loading indicators when charts are re-rendering
  • Cache API responses to avoid redundant calls
  • Show error messages in chart areas when Plotly fails
SECURITY 17. XSS Risk in Dynamic HTML

Strategy names and error messages are inserted with innerHTML without sanitization. A crafted name could inject malicious scripts. Fix by escaping HTML or using textContent.

UX 18-19. Mobile & Markdown
  • Sidebar doesn't collapse on mobile — add a hamburger menu
  • The markdown parser for strategy guidelines is fragile — use a proper library like marked.js

Frequently Asked Questions

Common questions a new developer might have when first reading this codebase

General Questions

Q: Where do I start if I want to understand the code?
Start with server.py. Read the route functions top to bottom. Each @app.route is one thing the server can do. Then read the JavaScript file for the page you're interested in (e.g., backtest.js for the backtest page).
Q: What's the minimum I need to run this app?
Python 3.8+, the packages (flask, flask-cors, pandas, nautilus_trader), and at least one CSV data file. Run python server.py and open localhost:5000.
Q: Why is it called a "Single Page Application" if it has 5 pages?
Because the browser only loads one HTML file (index.html). The "pages" are really just different chunks of HTML that JavaScript swaps into the main content area. The URL never changes.
Q: Why does the app use NautilusTrader instead of a simpler backtesting library?
NautilusTrader is a professional-grade engine that handles realistic order execution, position tracking, slippage, and fees. It produces results closer to real trading than a simple spreadsheet backtest would.

Data Questions

Q: What format do the CSV files need to be in?
Files should be named {id}_{SYMBOL}_{Name}.csv (e.g., 1_BTC_Bitcoin.csv) and contain columns for date, open, high, low, close, and volume.
Q: What is the "catalog" and why do we need it?
The catalog is a folder (./catalog) that stores your data in Parquet format. CSV files are slow to read every time, so we convert them once into Parquet (a fast binary format) during the "Load Data" step. After that, reading data for charts and backtests is much faster.
Q: Can I delete the catalog folder and recreate it?
Yes! The catalog is derived from your CSV files. If you delete ./catalog, just go to "Load Data" and re-import your CSVs.
Q: What does "BTCUSD.CRYPTO-1-DAY-LAST-EXTERNAL" mean?
It's NautilusTrader's identifier for this data: BTCUSD (Bitcoin vs US Dollar), CRYPTO (asset class), 1-DAY (daily bars), LAST (closing price type), EXTERNAL (data from an external source, not a live exchange).

Backtest Questions

Q: Do the backtest results account for trading fees?
Yes. The instrument factory sets maker and taker fees at 0.1% each (configured in instrument_factory.py). Every simulated buy/sell includes this fee.
Q: What does "trade size" mean? Should I change it?
Trade size is how many units of the cryptocurrency the strategy buys/sells per trade. Default is 1 (= 1 BTC, 1 ETH, etc.). For expensive coins like BTC, 1 unit = 1 full Bitcoin, which costs ~$83,000. Make sure your starting capital can cover it!
Q: Why did my backtest show 0 trades?
The strategy's conditions were never triggered by the data. Try adjusting parameters (e.g., widen the RSI thresholds) or try a different time period.
Q: Where are the backtest reports saved?
In the ./reports folder. You'll find both CSV files (positions, fills, account, orders, logs) and an HTML report per backtest run.

Custom Strategy Questions

Q: How do I create a custom strategy?
  1. Download the strategy template from the Backtest page
  2. Edit it to implement your trading logic
  3. Make sure you export: STRATEGY_NAME, STRATEGY_CLASS, CONFIG_CLASS, DESCRIPTION, PARAMS
  4. Upload the .py file via the Backtest page
Q: Why does the upload use a "subprocess" for validation?
NautilusTrader uses Rust bindings under the hood. If a bad strategy file causes a crash, it would kill the Flask server. By running validation in a subprocess, only the subprocess crashes — the server stays alive.

Code Questions

Q: How do I add a new page to the app?
  1. Create a new JS file (e.g., static/js/my_page.js) with a MyPage object that has a render(container) method
  2. Add a <script> tag in index.html
  3. Add a nav item in index.html's sidebar with data-page="my_page"
  4. Add the case in app.js's navigate() switch statement
Q: How do I add a new API endpoint?
  1. Add a new function in server.py with the @app.route("/api/your/endpoint") decorator
  2. Implement the logic (call core modules as needed)
  3. Return JSON with jsonify()
  4. Call it from JavaScript with App.api("/api/your/endpoint")
Q: How do I change the color theme?
Edit the CSS variables at the top of static/css/style.css. For example, changing --accent: #00d4aa to --accent: #ff6b6b would make the entire accent color red instead of teal.
Q: Why are there no frameworks (React, Vue, etc.)?
The app uses vanilla JavaScript intentionally. This keeps the dependency count low, makes it easier to understand for beginners, and avoids build tooling complexity. Each JS file is a plain object with methods — no JSX, no virtual DOM, no bundler needed.
