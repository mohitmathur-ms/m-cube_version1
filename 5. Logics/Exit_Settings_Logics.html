Exit Order Type

Dropdown MARKET Active

Defines the order type used when exiting (squaring off) positions. The dropdown has three options — MARKET, Limit, and SL_Limit — however only MARKET is currently active in the backtest engine. The other two options are blocked at the portfolio save step and are reserved for future implementation.

MARKET Active

When MARKET is selected the engine exits each leg at the current directional market price of that leg's option at the moment the exit event fires. No price condition is imposed — the exit is assumed to fill immediately at the prevailing price.

Directional Fill Logic

The engine uses the bid/ask split to simulate realistic fills, depending on whether the leg was originally a SELL or BUY:

Leg Direction Exit Action Price Used Fallback Chain
SELL Buy back (close short) ask_close ask_close → close → last_mtm_price → entry_price
BUY Sell out (close long) bid_close bid_close → close → last_mtm_price → entry_price
Why bid/ask split? A SELL leg (short option) must be bought back — you pay the ask. A BUY leg (long option) is sold out — you receive the bid. This models realistic slippage versus using a single mid or last-traded price.

VWAP Enhancement (Portfolio SqOff only)

In process_portfolio_sqoff() (triggered by SL / Target hits), the engine applies an additional VWAP-based fill improvement:

# Applied only in process_portfolio_sqoff — not in time-based _close_active_legs SELL leg (buying back): vwap = ask_vwap ref = ask_close if vwap > ref: fill_price = vwap # worse fill (paying more) — conservative else: fill_price = ref BUY leg (selling out): vwap = bid_vwap ref = bid_close if vwap > ref: fill_price = vwap # better fill (receiving more) — conservative else: fill_price = ref
Rationale: When a SL or Target is hit mid-candle, the VWAP of that candle's ask/bid stream better represents the average fill price than just the candle close. The formula uses whichever is more conservative (higher ask, higher bid) so the backtest does not over-optimise exit fills.

PnL Calculation at Exit

# For each leg exited at MARKET: if SELL leg: pnl = (entry_price − exit_price) × QTY × lots × multiplier if BUY leg: pnl = (exit_price − entry_price) × QTY × lots × multiplier

Limit Not Enabled in Backtest

🚧
Not yet implemented in the backtest engine.
Selecting Limit in the UI will prevent the portfolio from being saved (the save button triggers a validation check and blocks portfolios with exit_order_type set to Limit or SL_Limit). The option is visible in the dropdown but highlighted in red to indicate it is restricted. No engine logic exists for this order type yet.

Planned behaviour (future): Exit legs by placing a limit order at a specified price offset from the current market. The order would remain open until filled or until a time/price threshold forces a MARKET fallback.

SL_Limit Not Enabled in Backtest

🚧
Not yet implemented in the backtest engine.
Same save-time block applies as for Limit above. The option is visible in the UI dropdown but restricted (red highlight). No engine logic exists for this order type yet.

Planned behaviour (future): A stop-limit exit — a stop price triggers the order, which then fills at a limit price. Intended for slippage-controlled exits on fast-moving instruments.

Exit Sell Legs First

Checkbox Plugin 1

When checked, the engine exits all SELL legs before BUY legs during any batch square-off event. This controls the order in which legs are closed — it does not skip or filter any leg; all active legs are always exited.

StateExit Order
Checked (default) SELL legs closed first, then BUY legs
Unchecked Legs closed in their original insertion order (Python dict order)
Why SELL legs first? In a typical short-premium strategy (sell a straddle, sell a strangle), the SELL legs carry the most risk exposure. Closing them first reduces the open delta/gamma risk of the portfolio as quickly as possible. BUY legs (hedges, long spreads) are typically there to cap loss and can safely wait a tick.

Scope — Where Sorting is Applied

The sort is applied at every batch exit point in the engine — events where all (or most) active legs are closed in the same processing cycle:

FunctionTriggered BySort Applied
_close_active_legs() Time SqOff, End Time exit, EOD cleanup ✅ Yes
process_portfolio_sqoff() Portfolio SL hit, Portfolio Target hit, Force SqOff ✅ Yes
process_active_legs() (monitoring) Per-leg individual SL / Target checks ❌ No — individual leg exits, ordering irrelevant

Engine Implementation

# Applied at the top of each batch-exit function # State key: state["exit_settings"]["exit_sell_first"] (bool, default True) _sell_first = state.get("exit_settings", {}).get("exit_sell_first", True) _legs_iter = ( sorted(state["legs"].items(), key=lambda x: (0 if x[1]["conf"]["BUY/SELL"] == "SELL" else 1)) if _sell_first else list(state["legs"].items()) ) for lid, leg in _legs_iter: # ... close leg, calculate PnL, write to orderbook
Sort key explained: key=lambda x: (0 if BUY/SELL == "SELL" else 1) — SELL legs get sort priority 0 (first), BUY legs get 1 (last). Within each group, relative order is preserved from the original dict (stable sort).

Example — 4-Leg Portfolio

✅ Exit Sell Legs First = True (default)
Legs configured as:
Leg1: BUY CE · Leg2: SELL CE
Leg3: BUY PE · Leg4: SELL PE

Exit order:
1. Leg2 (SELL CE) closed
2. Leg4 (SELL PE) closed
3. Leg1 (BUY CE) closed
4. Leg3 (BUY PE) closed
🔵 Exit Sell Legs First = False
Same 4 legs as above.

Exit order:
1. Leg1 (BUY CE) closed
2. Leg2 (SELL CE) closed
3. Leg3 (BUY PE) closed
4. Leg4 (SELL PE) closed

Original insertion order — no sorting.
Important: In the current backtest engine all legs in the same tick are processed within the same data row — there is no simulated intra-tick delay between individual leg exits. The ordering therefore affects PnL attribution sequence and log order in the orderbook, but does not produce different fill prices between the two modes in the current implementation. The ordering will become price-impactful when intra-tick leg-exit delay is introduced in a future build.

Backward Compatibility

Reference

Portfolios created before the Exit Settings tab was introduced will not have an "exit_settings" key in their saved JSON. The engine handles this safely via state.get("exit_settings", {}) — missing keys fall back to defaults that match the UI defaults.

ParameterEngine DefaultUI DefaultEffect on Legacy Portfolios
exit_order_type "MARKET" MARKET No change — engine already used MARKET fills before this parameter existed
exit_sell_first True Checked Legacy portfolios will now exit SELL legs first — consistent with the intended default behaviour
Note on exit_sell_first default: Setting the engine default to True means all legacy portfolios (no exit_settings key) silently adopt the SELL-first ordering. If exact reproduction of a previous backtest run is required, re-run with a portfolio explicitly saved with exit_sell_first = False to restore the original dict-insertion order.
