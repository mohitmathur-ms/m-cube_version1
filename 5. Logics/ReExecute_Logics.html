Overview & Tab Layout

Context

The ReExecute Tab controls what happens when a leg's SL or Target is hit and the leg is configured to re-execute or re-enter. It is divided into two panels:

PanelContents
Left — Checkboxes
(green labels)		 Four global toggle flags that modify re-execution behaviour:
No ReExecute If Moved SL To Cost  ·  No Wait & Trade for ReExecute  ·  No Strike Change for ReExecute  ·  No Reentry/ReExecute after Portfolio End Time
Right panel No ReEntry If Moved SL to Cost — blocks the ReEntry action type when the leg's SL was previously moved to cost (mirrors Plugin 1 for Re-Entry).
ReExecute vs ReEntry — key distinction:
Re-Execute — leg re-enters immediately at the current market price of the same (or original) strike after SL/Target hit.
Re-Entry — leg waits for the market price to return to the original entry price before re-entering (price-based waiting).
01

No ReExecute If Moved SL to Cost

Checkbox Default: OFF

When ticked: if the SL of a leg was previously moved to its cost (entry price) via the Move SL to Cost feature, the engine will not trigger any further Re-Execution for that specific leg — even if the leg's SL is subsequently hit.

The rationale is that once SL is at cost the position is risk-free. Allowing re-execution after such a hit would open a fresh risk exposure that the user may not want.

How the engine tracks this

When MoveSLManager.apply() moves a leg's SL to cost it stamps leg["sl_moved_to_cost"] = True on the leg object. During the action-dispatch loop in process_active_legs(), before queuing a Re-Execute action, the engine checks:

if reexec_cfg["no_reexec_sl_cost"] and leg["sl_moved_to_cost"]: # Block re-execution → log REEXEC_BLOCKED_SL_COST and skip
✅ Ticked — re-execution blocked
Leg 1 entered @ $5.00. SL moved to $5.00 (cost).
Later SL hit → No re-execution fired.
Log: REEXEC_BLOCKED_SL_COST | LegID=1
❌ Unticked (default) — re-execution proceeds
Leg 1 entered @ $5.00. SL moved to $5.00 (cost).
Later SL hit → Re-execution fires normally.
A new position is opened at current market price.
Scope: This flag blocks Re-Execution only. It does not affect Re-Entry. Use No ReEntry If Moved SL to Cost to block Re-Entry for the same scenario.
Config keyEngine flag on legLog messageFile
no_reexec_sl_cost leg["sl_moved_to_cost"] REEXEC_BLOCKED_SL_COST engine/monitoring.py
02

No Wait & Trade for ReExecute

Checkbox Default: OFF

When ticked: the Wait & Trade delay configured on each leg is skipped entirely for re-executions. The re-executed leg enters at the very next available tick after its SL/Target hit, without waiting.

The original leg still waits its configured Wait & Trade delay on first entry. Only subsequent re-executions bypass the wait.

Example

Original leg: Wait & Trade = −10%, OnSL = Re-Execute.

✅ Ticked — re-executed leg skips wait
09:30 — Original leg enters after waiting for −10% Wait & Trade level.
10:15 — SL hit → re-execution queued.
10:15 — Re-executed leg enters immediately (no wait applied).
❌ Unticked (default) — re-executed leg also waits
09:30 — Original leg enters after −10% wait.
10:15 — SL hit → re-execution queued.
Re-executed leg waits for −10% again before entering.

How the engine detects a re-execution

Inside process_idle_leg_entries(), before applying the Wait & Trade delay, the engine checks leg["exec_count"]. A value >= 1 means the leg has been executed at least once, i.e. it is a re-execution:

_is_reexec = leg.get("exec_count", 0) >= 1 if wt_delay > 0 and _is_reexec and reexec_cfg["no_wait_trade_reexec"]: wt_delay = 0 # skip — log WAIT_TRADE_SKIPPED_REEXEC
Config keyDetectionLog messageFile
no_wait_trade_reexec exec_count >= 1 WAIT_TRADE_SKIPPED_REEXEC engine/execution.py
03

No Strike Change for ReExecute

Checkbox Default: OFF

When ticked: the re-executed leg uses the same strike as the original leg instead of resolving the current ATM (or ATM±offset) dynamically at the time of re-execution.

This is particularly useful for strategies involving Opposite Position, calendar spreads, or any case where consistency of strike across multiple executions is important.

Example

📌 Setup
Leg configured as ATM CALL. OnSL = Re-Execute.
10:15 — First entry. ATM = $525. Strike locked to 525 CE.
⏱ 12:10 — SL hit, re-execute triggers
Current ATM has moved to $530.

Ticked: Re-executes at $525 CE (original strike).
Unticked (default): Re-executes at $530 CE (current ATM).

Scope & Constraints

  • Only applies to ATM-based strikes (e.g. atm, atm+2, atm-1). Fixed numeric strikes are always used as-is regardless of this flag.
  • The original resolved strike is stored in leg["resolved_strike"] at entry time and persists through the IDLE state.
  • Also suppresses the Straddle Width Multiplier override for re-executed legs — the locked strike is used as-is and the multiplier formula is not re-applied.
  • Does not interfere with same_strike re-entry mode (which manages its own lock).
# In process_idle_leg_entries() — strike resolution logic strike_key = safe_strike_key(leg["conf"]["Strike"]) # e.g. "atm" _is_atm_based = strike_key.startswith("atm") if reexec_cfg["no_strike_change_reexec"] and _is_reexec and _is_atm_based: strike_key = leg["resolved_strike"] # e.g. "525" — log STRIKE_LOCKED_REEXEC
Config keyApplies toLog messageFile
no_strike_change_reexec ATM-based strikes only STRIKE_LOCKED_REEXEC engine/execution.py
04

No ReEntry / ReExecute after Portfolio End Time

Checkbox Default: OFF (allow till SqOff)

Controls whether Re-Execution and Re-Entry are allowed after the portfolio's End Time has elapsed (but before the SqOff Time).

StateBetween Start → End TimeBetween End Time → SqOff
Unticked (default) ✅ Re-Execution & Re-Entry allowed ✅ Re-Execution & Re-Entry still allowed up to SqOff
Ticked ✅ Re-Execution & Re-Entry allowed ❌ Re-Execution & Re-Entry blocked after End Time
Open positions are never affected by this flag.
Legs that are already ACTIVE continue to run with their SL / Target logic until they are closed by SqOff or their own exit condition. This flag only blocks new entries.

Scenario — Start 09:30 · End 11:30 · SqOff 16:15

✅ Ticked — strict end-time block
11:31 — Leg SL hit → No re-execution.
Leg exits and stays flat.
Open positions continue until SqOff 16:15.
🔄 Unticked (default) — allow till SqOff
11:31 — Leg SL hit → Re-execution fires.
New position opens and runs till SqOff 16:15.
Further hits can re-execute up to SqOff.
Exception — Manual / Cross-portfolio trigger: If a portfolio was started via a Start Other Portfolio cross-portfolio action, the end_blocked gate is bypassed entirely for that trigger cycle regardless of this checkbox. This ensures explicit cross-portfolio triggers always take effect.

05

No ReEntry If Moved SL to Cost

Checkbox Default: ON (ticked)

When ticked (default): if the SL of a leg was moved to its cost (entry price) via Move SL to Cost, the engine will not queue a Re-Entry waiting state for that leg when its SL is subsequently hit.

This mirrors Plugin 1 but applies to the Re-Entry action type (price-based waiting) rather than immediate Re-Execute.

Default is ON — this is the safe, conservative default. Most strategies that move SL to cost intend to fully exit the position at breakeven; allowing a re-entry would re-open exposure which is usually unintended.
✅ Ticked (default) — re-entry blocked
Leg SL moved to cost ($5.00). SL hit at $5.00.
Leg closed at breakeven.
No re-entry waiting state is set.
Log: REENTRY_BLOCKED_SL_COST | LegID=1
❌ Unticked — re-entry proceeds
Leg SL moved to cost ($5.00). SL hit at $5.00.
Leg closed at breakeven.
Re-entry waiting state activated — leg waits for price to return to $5.00 before re-entering.

Interaction & Edge Cases

Engineering Notes

Plugin 1 vs Plugin 5 — same trigger, different action

Both No ReExecute If Moved SL to Cost (P1) and No ReEntry If Moved SL to Cost (P5) react to the same leg["sl_moved_to_cost"] flag. The difference is which action they block:

PluginBlocked ActionLeg state after block
P1 — No ReExecuteRe-Execute action typeLeg remains CLOSED — no new position
P5 — No ReEntryReEntry action typeLeg remains CLOSED — no waiting state

Both can be active simultaneously for full protection across both action types.

Plugin 3 + Straddle Width Multiplier

When No Strike Change for ReExecute (P3) is active for a re-execution, the engine also suppresses the Straddle Width Multiplier override for that leg on that tick. This prevents the multiplier from recalculating a new strike and overriding the locked one. The guard in the engine:

if straddle_mult != 0 and not is_prem_greek \ and mode != "same_strike" \ and not _no_strike_change_active: # ← P3 guard # apply multiplier...

Plugin 4 + Manual / Cross-Portfolio trigger

If a portfolio's execution mode is Manual and it was triggered by a Start Other Portfolio action from another portfolio, the end_blocked gate is bypassed for that trigger. Plugin 4's checkbox has no effect on cross-portfolio triggered entries — the trigger always takes priority.

sl_moved_to_cost flag persistence

The leg["sl_moved_to_cost"] flag is set once and never cleared within a trading day. If a leg somehow re-executes after the SL was moved to cost (e.g. Plugin 1 is off), the flag carries over into the next execution cycle. This means if Plugin 1 is later enabled mid-session (live mode), it will correctly block the next re-execution attempt on that leg.

Execution order within a tick

1
process_active_legs() — SL/Target check, action dispatch.
Plugin 1, 4 (dispatch), 5 gates applied here.
2
process_reentry_waiting() — Check if waiting legs reached their re-entry price.
3
process_idle_leg_entries() — Enter IDLE legs.
Plugin 2, 3, 4 (entry gate) applied here.
4
Portfolio SL/TGT checks — Portfolio-level force_sqoff and re-execution logic.
