1 Overview

The Dynamic Hedge system automatically opens a protective hedge leg whenever a parent leg with "Hedge Required" enters a trade. Two axes of configuration control the behaviour:

Hedge Type — how to find the strike
PremiumBased — scans the OTM side of the parent strike within a distance window, filtered by premium range.
DistanceBased — takes exactly parent_strike ± fixed_dist. No scan, no premium filter.
SqOff Mode — when to exit the hedge
LegSqOff — hedge exits when parent exits.
PortfolioSqOff — hedge stays alive after parent exits; closes only on portfolio-level event or EOD.

4 Core Hedge Rules

These three rules apply to every hedge leg created, regardless of hedge type or sqoff mode.
C/P — Same as Parent
CALL parent → CALL hedge
PUT parent → PUT hedge
BUY/SELL — Opposite of Parent
SELL parent → BUY hedge
BUY parent → SELL hedge
Strike — Always OTM of Parent Strike
CALL parent → hedge strike > parent strike (further OTM for calls)
PUT parent → hedge strike < parent strike (further OTM for puts)
Base reference is always the parent leg's own strike — not ATM.
Examples:
SELL CALL @ 652  →  BUY CALL @ strike > 652  (e.g. 657)
SELL PUT  @ 600  →  BUY PUT  @ strike < 600  (e.g. 595)
BUY  CALL @ 652  →  SELL CALL @ strike > 652

7 Hedge Type Logic

7.1 PremiumBased

find_hedge_strike(rec, parent_stk, leg_type, hedge_params) — engine/execution.py

# Base = parent leg's own strike (not ATM)
if leg_type == CALL:
    candidates = [parent_stk + dist_min  ...  parent_stk + dist_max]  # above = OTM
else:  # PUT
    candidates = [parent_stk - dist_max  ...  parent_stk - dist_min]  # below = OTM

valid = [s for s in candidates if prem_min <= premium(s) <= prem_max]
valid.sort(key=lambda x: (x.prem, x.dist))   # cheapest first
return valid[0].strike  if valid  else  None

7.2 DistanceBased

No scan, no premium filter. Computed strike is snapped to the nearest valid strike_step so the option chain quote is guaranteed to resolve.

# No ATM lookup. No chain scan. No premium filter.
parent_stk  = float(leg.get("resolved_strike", 0))   # resolved numeric strike (not raw DSL string)
fixed_dist  = int(dh_cfg.get("hedge_fixed_min_distance", 0))
strike_step = int(meta.get("general", {}).get("strike_step", 1)) or 1
raw_stk     = parent_stk + fixed_dist  if  l_cp == 0  else  parent_stk - fixed_dist
hedge_stk   = round(raw_stk / strike_step) * strike_step   # snapped to valid chain strike
Example: parent_stk=650, fixed_dist=7, strike_step=5
raw_stk = 657 → round(657/5)*5 = 655 — hedge is placed at 655, not 657.

8 SqOff Mode Logic

8.1 LegSqOff

When the parent exits for any reason, the linked hedge is immediately squared off in the same bar.

Parent exits at T → Hedge exits at T (PARENT EXIT)

8.2 PortfolioSqOff

The hedge exit sync block in monitoring.py is skipped when _dh_sqoff_mode == "PortfolioSqOff". Hedge stays ACTIVE and closes only when:

TriggerHandler
Portfolio SL hitprocess_portfolio_sqoff()
Portfolio Target hitprocess_portfolio_sqoff()
User Max Loss hitprocess_portfolio_sqoff()
User Max Profit hitprocess_portfolio_sqoff()
Sqoff time_close_active_legs()
EOD cleanupprocess_eod_cleanup()
Parent exits at T → Hedge stays ACTIVE Portfolio SL at T+N → Hedge exits (PORTFOLIO_SQOFF)
Note: While ACTIVE the hedge's running PnL contributes to portfolio combined PnL — it can itself help trigger the portfolio SL/Target it's waiting for.

13 End-to-End Examples

13.1 PremiumBased + LegSqOff

09:30 — SELL CALL @ 652 enters (dist=[5,10], prem=[0.05,40]) find_hedge_strike(base=652, CALL, dist=[5,10]) 657→3.50 ✓ 659→2.80 ✓ 661→2.10 ✓ 663→0.02 ✗ sort by prem ASC → selected: 661 @ 2.10 H_1_0: BUY CALL 661 11:00 — Parent SL hit → exits H_1_0 exits immediately (LegSqOff) Orderbook: SELL CALL 652 | _IS_HEDGE=False | EXIT=LEG_SL_HIT BUY CALL 661 | _IS_HEDGE=True | EXIT=PARENT EXIT

13.2 DistanceBased + PortfolioSqOff

09:30 — SELL CALL @ 652 enters (fixed_dist=5) hedge_stk = 652 + 5 = 657 (no premium check) H_1_0: BUY CALL 657 @ market price 11:00 — Parent SL hit → exits H_1_0 stays ACTIVE (PortfolioSqOff — not closed here) 13:15 — Portfolio SL breached H_1_0 exits via process_portfolio_sqoff() Orderbook: SELL CALL 652 | _IS_HEDGE=False | EXIT=LEG_SL_HIT BUY CALL 657 | _IS_HEDGE=True | EXIT=PORTFOLIO_SQOFF

14 Edge Cases & Failure Modes

ScenarioBehavior
DistanceBased: fixed_dist not aligned with strike_step Computed strike is snapped: round(raw / step) * step. Hedge always resolves to a valid chain strike.
DistanceBased: snapped strike still not in chain get_option_quote() returns None → HEDGE_STRIKE_NOT_IN_CHAIN logged at WARNING; hedge skipped for this activation.
PremiumBased: no candidates in window find_hedge_strike() returns None; HEDGE_NO_CANDIDATE logged at TRADING; parent enters without a hedge.
PremiumBased: all candidates fail premium filter Same — HEDGE_NO_CANDIDATE logged; hedge skipped.
Quote has NaN or zero price fields _safe_px() filters NaN/None to falsy; fallback chain ask_open → ask_close → close → 0.0 is walked. Hedge is always created. If price resolves to 0.0, HEDGE_PRICE_UNAVAILABLE is logged at WARNING.
Hedge inherits parent SL / Target Prevented at creation: h_conf is explicitly overridden with "Stoploss Type": "None" and "Target Type": "None". Hedge never self-exits.
max_hedge_executions reached MAX_HEDGE_LIMIT_REACHED logged at WARNING; hedge skipped; parent enters normally.
PortfolioSqOff + hedge outlives the day process_eod_cleanup() closes all remaining ACTIVE legs including any orphaned hedge legs.
enabled = false Entire dynamic hedge system bypassed — no hedge legs created.
