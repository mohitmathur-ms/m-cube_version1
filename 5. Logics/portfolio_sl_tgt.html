Overview

Portfolio-level SL/Target operates on the combined PnL of all active legs in the portfolio, checked every tick. It is completely independent of per-leg SL/TGT. When a condition fires the engine dispatches an action (sqoff, re-execute, cross-portfolio trigger, etc.) optionally after a delay guard.

There are six sub-systems:

Sub-system Tab / Group Config key prefix
Portfolio Fixed Stop Loss Stoploss → Stoploss Settings stoploss.*
Portfolio Trailing Stop Loss Stoploss → Trailing SL Settings stoploss.trailing.*
Move SL to Cost Stoploss → Move SL to Cost stoploss.move_sl_to_cost.*
Portfolio Fixed Target Target → Target Settings target.*
Portfolio Trailing Target Target → Trailing Settings target.trailing.*
Global User Limits User Settings (separate) UserSettings.*

All checks run inside check_portfolio_sl_tgt() every tick, in evaluation order:
1. Trailing SL ratchet
2. Fixed SL hit check
3. Trailing Target activation / ratchet
4. Fixed Target hit check

PART 1 — STOPLOSS TAB

1. Stoploss Settings

1.1 Enable Stoploss

Field stoploss.enabled Type: bool
Default false

Master switch for the entire portfolio stoploss system. When false, no SL type is evaluated, no SL delay is checked, and no SL action fires. All other fields in the Stoploss Settings group are disabled in the UI when this is off.

1.2 Stoploss Type

Field stoploss.type Type: str
Default "Combined Loss"

Determines what condition triggers the portfolio stoploss. Five active types, two placeholders.

Combined Loss (default)

Trigger:  curr_pnl  <=  -(stoploss.value)

Fires when the portfolio's combined unrealised + realised PnL drops to or below the configured loss threshold. The threshold is expressed as a positive number representing the maximum allowed loss (e.g. value = 300 means the SL fires when PnL ≤ −300).

  • Most commonly used SL type.
  • Works with all actions including cross-portfolio and re-execute.
  • Compatible with Trailing SL, Delay, and both Leg Filters.

Combined Premium

threshold  =  -(stoploss.value  ×  QTY)
Trigger:   state["pnl"]["combined"]  <=  threshold

Converts the entered premium value into a PnL-based loss threshold by multiplying it by the hardcoded per-lot quantity (QTY = 100, defined in backtester.py). The stoploss fires when the portfolio's raw combined PnL falls to or below the computed negative threshold.

  • Requires at least one leg to have filled (p_comb_prem_captured = True).
  • Value entered is a per-lot premium amount, not a raw PnL figure. The engine scales it to a full PnL threshold internally.
  • Uses state["pnl"]["combined"] directly — bypasses per-lot scaling even when tgt_sl_perlot is enabled.
  • Log: Combined Premium hit - portfolio stoploss hit | PnL=X | Limit=Y | Premium_SL=Z | Qty=100
Example: stoploss.value = 100, QTY = 100 → threshold = −(100 × 100) = −₹10,000. SL fires when PnL ≤ −₹10,000.

Absolute Combined Premium

Trigger:  curr_combined_premium  >=  value

Unconditional — fires when the combined premium exceeds an absolute fixed level regardless of the initial premium. Directional only upward.

  • Use when you know the exact premium level at which you want to stop.
  • Does not compare to initial; fires purely on the raw current premium value.

Underlying Movement (UI label: "Underlying Movemnet")

Trigger:  index price crosses value (in either direction)
  (prev_tick_idx <= value <= curr_tick_idx)
  OR
  (prev_tick_idx >= value >= curr_tick_idx)

Fires when the underlying index (e.g. SPY) crosses a specified price level from either side. State variable last_idx_p carries the previous tick's index price to detect the crossing.

  • Triggers once on the first crossing tick.
  • The value field is the price level to watch, not a loss amount.
  • Leg Filters (Selective SqOff) are automatically disabled for this type.

Loss and Underlying Range

Trigger:  curr_pnl <= -(stoploss.value)
      AND (curr_idx <= underlying_below  OR  curr_idx >= underlying_above)

Dual condition — both must be true simultaneously. The portfolio SL fires only when the PnL loss threshold is breached and the underlying is outside the defined price band.

  • Useful for range-bound strategies: tolerate losses while the index stays inside the band, exit if the index breaks out AND the portfolio is in loss.
  • Requires the two additional fields: underlying_below and underlying_above.
  • If only one direction is relevant, set the other to 0 (disabled).
  • Leg Filters are automatically disabled for this type.

Delta / Theta (Not Enabled)

Placeholder options — not implemented. Selecting these disables SL evaluation.

1.3 Stoploss Value

Field stoploss.value Type: float
Default 0.0 Range: 0 – 10,000,000

The numeric threshold for the selected SL type:

Type Meaning of Value
Combined Loss Maximum allowed loss in ₹ (positive number; SL fires at PnL ≤ −value)
Combined Premium Per-lot premium amount (₹). Engine computes SL threshold = -(value × QTY) and compares against portfolio PnL.
Absolute Combined Premium Absolute combined premium ceiling (₹ or points) — compared directly against live premium price, no PnL math
Underlying Movement Index price level to watch for crossover
Loss and Underlying Range Maximum allowed loss in ₹ (same as Combined Loss)

1.4 On SL Action

Field stoploss.action Type: str
Default "SqOff"

What the engine does when the SL condition fires (after any configured delay). Seven options:

SqOff (default)

Closes all active legs in this portfolio immediately. Sets force_sqoff = True on this portfolio. Log: PORTFOLIO_SQOFF | Reason=STOPLOSS.

ReExecute

Closes all legs and immediately re-opens them (re-enters the strategy). Each re-entry increments reexec_curr. Stops re-entering once reexec_count is reached (0 = unlimited). Log: PORTFOLIO_REEXECUTE | Reason=STOPLOSS | Count=N.

SqOff Other Portfolio

Closes this portfolio and force-sqoffs the named target portfolio. Both portfolios close on the same tick. Requires Target Portfolio name to be set. Log: PORTFOLIO_SQOFF | Reason=STOPLOSS on both portfolios.

Execute Other Portfolio

Closes this portfolio and triggers execution (sets execute_trigger = True) on all IDLE legs of the named target portfolio. Used to activate a hedge or alternate strategy when SL hits. Requires Target Portfolio name.

ReExecute at Entry Price

Closes all legs and waits to re-enter each leg at its original entry price (STRIKED LOCKED). The engine monitors each tick until the option price returns to entry price before re-entering. Log: STRIKED LOCKED {strike} {CP} AT PRICE {price}.

ReExecute SameStrike at EntryPrice

Same as above but re-enters at the exact same strike as the original entry (does not allow strike drift). Log: REEXECUTING AS STRIKE {X} GOT {price} AGAIN when re-entry condition is met.

Start Other Portfolio

Closes this portfolio and activates the named target portfolio's execution mode (sets its exec_mode to active). Used to chain portfolio execution. Requires Target Portfolio name. Log: START_OTHER_PORTFOLIO | Triggering={src} | Target={dst}.

1.5 Target Portfolio

Field stoploss.portfolio_name Type: str
Default "" Visible only for cross-portfolio actions

Name of the portfolio to act upon when the action is SqOff Other Portfolio, Execute Other Portfolio, or Start Other Portfolio. Hidden in the UI for all other actions. Must exactly match the portfolio_name of the target portfolio in the same session.

1.6 Delay (sec)

Field stoploss.delay Type: int
Default 0 Range: 0 – 9999

Confirmation delay in seconds before the SL action fires. When > 0, the engine does not immediately act on the first tick the SL condition is met. Instead:

  1. First hit: sets p_delay_pending = True, records p_delay_start_time. Log: PORTFOLIO_DELAY_PENDING | Reason=STOPLOSS | Delay=Ns.
  2. Each subsequent tick: checks elapsed = (curr_dt - start_time).total_seconds().
  3. If elapsed < delay: waits, no action.
  4. If elapsed >= delay AND condition still holds: fires the SL action.
  5. If the condition clears before the delay expires (e.g. PnL recovers): p_delay_pending = False. Log: PORTFOLIO_DELAY_CLEARED | Condition persistent=False. The delay counter resets completely — next time the condition is met, a fresh delay window starts.

Oscillating behaviour: If the condition repeatedly hits and clears within the delay window, each new hit restarts the delay timer. The SL only fires when the condition holds continuously for the full delay duration.

Interaction with Trailing SL: Delay and Trailing SL work together — the trailing SL keeps ratcheting during the delay window, so the SL threshold may move during the wait period.

1.7 ReExecute Count

Field stoploss.reexecute_count Type: int
Default 0 (= unlimited) Range: 0 – 1,000,000

Maximum number of times the portfolio can re-execute after an SL hit when the action is ReExecute, ReExecute at Entry Price, or ReExecute SameStrike at EntryPrice.

  • 0 means unlimited re-executions — the strategy will re-enter every time the SL fires.
  • Once reexec_curr >= reexec_count, the next SL hit performs a final SqOff instead of re-executing.
  • The counter reexec_curr increments on every re-execution and resets each trading day.
  • Disabled in the UI when the action is not a ReExecute variant.

1.8 Underlying Below / Underlying Above

Fields stoploss.underlying_below, stoploss.underlying_above Type: float
Default 0.0 Visible only when type = "Loss and Underlying Range"

Define the price band for the Loss and Underlying Range SL type:

  • underlying_below: SL contributes if curr_index <= this value. Set to 0 to disable the lower bound.
  • underlying_above: SL contributes if curr_index >= this value. Set to 0 to disable the upper bound.

Combined condition: PnL <= -value AND (index <= below OR index >= above).

Example: value=50, below=0, above=595 → SL fires only when the portfolio is down ≥₹50 AND the index has broken above 595.

1.9 On SL SqOff Only Loss Making Legs

Field stoploss.sqoff_only_loss_legs Type: bool
Default false Enabled only: Combined SL types + SqOff action

When ticked, the SL does not close the entire portfolio. Instead it only closes legs whose current PnL is negative at the moment the SL fires. Legs that are in profit are left ACTIVE and the portfolio continues running.

Post-partial-close state:
- force_sqoff is reset to False — portfolio is NOT fully closed
- trail_state is cleared — trailing SL recalculates from new combined PnL
- Portfolio keeps monitoring target and individual leg SLs on remaining legs
- SL will not immediately re-fire on the same tick
- Log: PORTFOLIO_PARTIAL_SQOFF | Reason=STOPLOSS | Filter=LOSS_LEGS_ONLY

Example — 3-leg portfolio, SL fires:

Leg Leg PnL Action
Leg 1 SELL C −₹45 Closed
Leg 2 SELL P +₹20 Kept ACTIVE
Leg 3 SELL C −₹12 Closed

Leg 2 continues running. The portfolio will exit Leg 2 when its own leg SL/TGT fires or when the portfolio target is reached.

Constraint: Mutually exclusive with sqoff_only_profit_legs — enabling one disables the other. Only available for Combined Loss, Combined Premium, and Absolute Combined Premium types with SqOff action.

1.10 On SL SqOff Only Profit Making Legs

Field stoploss.sqoff_only_profit_legs Type: bool
Default false Enabled only: Combined SL types + SqOff action

Inverse of the above. When the SL fires, only legs currently in profit are closed. Loss-making legs are left running.

Use case: Lock in profitable legs when the portfolio hits a loss threshold, while allowing the losing legs to recover (or their individual SLs to handle exit).

Post-partial-close state: Same as sqoff_only_loss_legs — portfolio continues, trail resets, SL does not re-fire on same tick. Log: PORTFOLIO_PARTIAL_SQOFF | Reason=STOPLOSS | Filter=PROFIT_LEGS_ONLY.

Constraint: Mutually exclusive with sqoff_only_loss_legs.

2. Trailing SL Settings

2.1 Enable Trailing SL

Field stoploss.trailing.enabled Type: bool
Default false

Activates the dynamic trailing stop loss. When enabled, the effective SL threshold tightens automatically as portfolio PnL improves, locking in progressively more profit protection. The trailing SL only moves in one direction (tighter — it never relaxes back).

2.2 For Every Profit of (₹)

Field stoploss.trailing.trail_every Type: float
Default 0.0

Step size in ₹ PnL improvement that triggers one trailing ratchet. Each time the portfolio PnL improves by this amount from the last anchor point, the SL threshold tightens by trail_by.

Example: trail_every = 50 → every ₹50 gain in PnL triggers a step.

2.3 Tighten SL By (₹)

Field stoploss.trailing.trail_by Type: float
Default 0.0

Amount by which the SL threshold is reduced per trailing step. Reducing the SL threshold means the maximum allowed loss shrinks — the portfolio is protected at a tighter level.

Example: trail_by = 20 → each step reduces the SL threshold by ₹20 (e.g. SL moves from ₹300 loss tolerance → ₹280 → ₹260 …).

Full trailing SL computation (per tick):

gain  = curr_pnl - anchor_pnl
if gain >= trail_every:
    steps               = int(gain / trail_every)
    current_sl_val     -= steps * trail_by          ← SL tightens
    anchor_pnl         += steps * trail_every       ← anchor advances

current_sl_val starts at stoploss.value and can only decrease. If trail_by is too large relative to trail_every, the SL can reach 0, meaning any loss at all would trigger it.

Log: TRAIL_SL_UPDATED | Steps=N | Combined_SL=X | PnL=Y

Interaction with Delay: Trailing ratcheting continues during a delay window. So if the SL fires, a delay is pending, and PnL improves during the wait, the SL threshold will have tightened further when/if the delay eventually dispatches.

Interaction with Move SL to Cost trail_after_move_sl: When that flag is enabled, trailing SL only activates after the Move SL to Cost feature has executed at least once (see Section 3.6).

3. Move SL to Cost Settings

3.1 Enable Move SL to Cost

Field stoploss.move_sl_to_cost.enabled Type: bool
Default false

Master switch for the Move SL to Cost feature. When active, whenever any leg in the portfolio hits its individual SL or Target (configurable), the engine repositions the SL of all other active legs to their entry price. This locks each remaining leg's cost basis, preventing a winning leg from turning into a loser after another leg has exited.

3.2 Safety Seconds

Field stoploss.move_sl_to_cost.safety_seconds Type: int
Default 0 Range: 0 – 9999

Minimum number of seconds that must have elapsed since portfolio start before the Move SL to Cost feature can trigger. Prevents premature SL moves immediately at open when prices are volatile and fills may not have settled.

Setting to 0 means it can trigger at any time after portfolio start.

3.3 Move SL Action

Field stoploss.move_sl_to_cost.action Type: str
Default "Move Only for Profitable Legs"

Controls which legs have their SL repositioned when the trigger fires:

Move Only for Profitable Legs

For each active leg:
  if leg_pnl > 0:
      leg["sl"] = leg["entry_price"]

Only moves SL to cost for legs that are currently in profit. Loss-making legs are left with their original SL unchanged.

Move SL for All Legs Despite Loss / Profit

For each active leg (regardless of PnL):
    leg["sl"] = leg["entry_price"]

Moves SL to entry price for every active leg unconditionally — even legs that are currently in loss. This means a loss-making leg's SL moves to entry price (which is actually above its current price if it's a loss, so the leg will immediately close if it's already below entry).

Move SL to LTP + Buffer for Loss Making Legs

For each active leg:
  if leg_pnl > 0:
      leg["sl"] = leg["entry_price"]               ← profitable legs → cost
  else:
      leg["sl"] = current_ltp ± 0.05% buffer       ← loss legs → LTP with buffer

Profitable legs get SL moved to entry price. Loss-making legs get their SL moved to their current Last Traded Price plus a small buffer (0.05%), giving them a tight but realistic stop at the current price level.

Log: MOVE_SL_TO_COST | Action={mode} | LegID={id}

Dynamic hedge legs (is_dynamic_hedge = True) are always skipped regardless of the action selected.

3.4 Trail SL only after Move SL to Cost

Field stoploss.move_sl_to_cost.trail_after_move_sl Type: bool
Default false

When enabled, the Trailing SL (Section 2) is suppressed until Move SL to Cost has executed at least once. This prevents the trailing mechanism from moving the SL before legs have their cost protected.

Typical use case: You want the SL to stay at the original value until the first leg exits, then start trailing from that point onward as the remaining legs are already at cost.

3.5 No Move SL for BUY Legs

Field stoploss.move_sl_to_cost.no_move_buy_legs Type: bool
Default false

When ticked, BUY legs are completely excluded from the Move SL to Cost logic regardless of the selected action. Only SELL legs will have their SL repositioned. Useful for strategies where BUY legs act as hedges and should not have their SL altered.

3.6 Re-apply on every Leg SL hit

Field stoploss.move_sl_to_cost.hit_on_leg Type: bool
Default false

By default, Move SL to Cost triggers once on the first leg exit. When this is checked, it re-applies every time any subsequent leg hits its individual SL. This ensures that each successive leg exit re-locks the remaining active legs to cost.

3.7 Re-apply on every Leg Target hit ✱

Field stoploss.move_sl_to_cost.tgt_hit_on_leg Type: bool
Default false

Same as above but triggered on individual leg Target hits rather than SL hits. When a leg closes at its target price, the SLs of all remaining active legs are repositioned per the selected action.

The ✱ marker in the UI indicates that this option interacts with leg-level target logic and should be used carefully when legs have aggressive target values.

PART 2 — TARGET TAB

4. Target Settings

4.1 Enable Target

Field target.enabled Type: bool
Default false

Master switch for the portfolio target system. When false, no target condition is evaluated. All fields in the Target Settings group are disabled in the UI when this is off.

4.2 Target Type

Field target.type Type: str
Default "Combined Profit"

Determines what condition triggers the portfolio target exit. Four active types, two placeholders.

Combined Profit (default)

Trigger:  curr_pnl  >=  target.value

Fires when the portfolio's combined unrealised + realised PnL reaches or exceeds the configured profit target. Value is expressed as a positive ₹ amount.

Combined Premium

threshold  =  target.value  ×  QTY
Trigger:   state["pnl"]["combined"]  >=  threshold

Converts the entered premium value into a PnL-based profit threshold by multiplying it by the hardcoded per-lot quantity (QTY = 100, defined in backtester.py). The target fires when the portfolio's raw combined PnL reaches or exceeds the computed threshold.

  • Requires at least one leg to have filled (p_comb_prem_captured = True).
  • Value entered is a per-lot premium amount, not a raw PnL figure. The engine scales it to a full PnL threshold internally.
  • Uses state["pnl"]["combined"] directly — bypasses per-lot scaling even when tgt_sl_perlot is enabled.
  • Log: Combined Premium hit - portfolio profit hit | PnL=X | Target=Y | Premium_Target=Z | Qty=100
Example: target.value = 100, QTY = 100 → threshold = 100 × 100 = ₹10,000. Target fires when PnL ≥ ₹10,000.

Absolute Combined Premium

Trigger:  curr_combined_premium  <=  value

Fires when combined premium decays to or below an absolute fixed level. Directional only downward. Suitable for premium-decay targets in short-option strategies.

Underlying Movement (UI label: "Underlying Movemnet")

Trigger:  index price crosses value (in either direction)
  (prev_tick_idx <= value <= curr_tick_idx)
  OR
  (prev_tick_idx >= value >= curr_tick_idx)

Fires when the underlying index crosses a price level. Independent of PnL — purely price-based. The value field is an index price level, not a profit amount. State variable last_idx_p_tgt is separate from the SL's last_idx_p.

Delta / Theta (Not Enabled)

Placeholder options — not implemented.

4.3 Target Value

Field target.value Type: float
Default 0.0 Range: 0 – 10,000,000

The numeric threshold for the selected target type:

Type Meaning of Value
Combined Profit Profit to capture in ₹
Combined Premium Per-lot premium amount (₹). Engine computes target threshold = value × QTY and compares against portfolio PnL.
Absolute Combined Premium Absolute combined premium floor (₹ or points) — compared directly against live premium price, no PnL math
Underlying Movement Index price level to watch for crossover

4.4 On Target Action

Field target.action Type: str
Default "SqOff"

What happens when the target fires. Identical set of options as the SL action:

Action Behaviour
SqOff Close all active legs. Log: PORTFOLIO_SQOFF \| Reason=TARGET
ReExecute Close all legs and re-enter. Log: PORTFOLIO_REEXECUTE \| Reason=TARGET \| Count=N
SqOff Other Portfolio Close self + named target portfolio
Execute Other Portfolio Close self + trigger IDLE legs on named target portfolio
ReExecute at Entry Price Close and wait to re-enter at each leg's original entry price
ReExecute SameStrike at EntryPrice Close and wait to re-enter at the exact same strike at entry price
Start Other Portfolio Close self + activate named target portfolio

Trailing Target hits always use SqOff regardless of this setting.

4.5 Target Portfolio

Field target.portfolio_name Type: str
Default "" Visible only for cross-portfolio actions

Name of the portfolio to act upon when the target action is SqOff Other Portfolio, Execute Other Portfolio, or Start Other Portfolio. Must exactly match the target portfolio's portfolio_name.

4.6 Delay (sec)

Field target.delay Type: int
Default 0 Range: 0 – 9999

Confirmation delay before the target action fires — identical mechanics to SL Delay (Section 1.6). When the target condition is met, the engine waits for delay seconds of continuous confirmation before acting.

  • p_delay_pending is shared between SL and TGT (only one can be pending at a time).
  • Log: PORTFOLIO_DELAY_PENDING | Reason=TARGET | Delay=Ns
  • If the condition clears before delay expires: PORTFOLIO_DELAY_CLEARED | Condition persistent=False

4.7 ReExecute Count

Field target.reexecute_count Type: int
Default 0 (= unlimited) Range: 0 – 1,000,000

Maximum re-executions after a target hit. Same logic as SL ReExecute Count (Section 1.7) but applied to target re-entries. Counter p_tgt_reexec_curr increments per re-entry; resets each day. Once the count is reached, the next target hit performs a final SqOff.

5. Trailing Target Settings

5.1 Enable Trailing

Field target.trailing.enabled Type: bool
Default false

Activates the trailing target / profit lock mechanism. Instead of exiting at a fixed profit level, the engine:
1. Waits for PnL to reach an activation threshold
2. Locks a minimum profit floor
3. Raises the floor progressively as PnL continues improving
4. Exits only when PnL falls back to the locked floor

This allows the portfolio to capture more profit when a trend continues while still protecting a defined minimum gain.

5.2 Lock Minimum Profit

Field target.trailing.lock_min_profit Type: float
Default 0.0

The initial profit floor that is locked the moment the trailing target activates. Once the PnL reaches when_profit_reach, the locked floor is immediately set to lock_min_profit. The portfolio will not exit for less than this amount.

Example: lock_min_profit = 80 → as soon as trailing activates, the portfolio will close if PnL drops back to ₹80 or below.

5.3 When Profit Reach

Field target.trailing.when_profit_reach Type: float
Default 0.0

The PnL level at which the trailing target activates. The portfolio is allowed to trade freely (without a trailing exit) until PnL first reaches this value. Once crossed, the trailing mechanism activates and lock_min_profit is set immediately.

Example: when_profit_reach = 150 → trailing does nothing until PnL reaches ₹150, then the floor is locked at lock_min_profit.

Relationship: when_profit_reach should be ≥ lock_min_profit — you can't lock a profit floor higher than the activation threshold.

5.4 For Every Increase By

Field target.trailing.trail_every Type: float
Default 0.0

Step size in ₹ after which the locked floor is raised. Once the trailing target is active and PnL continues rising, each time PnL increases by trail_every above the current anchor, the floor rises by trail_by.

Example: trail_every = 30 → every ₹30 increase after activation raises the floor.

5.5 Trail Profit By

Field target.trailing.trail_by Type: float
Default 0.0

How much the locked profit floor is raised per step. Works together with trail_every.

Example: trail_by = 20 → each step raises the floor by ₹20.

Full trailing target computation (per tick):

Phase 1 — Activation (first time PnL >= when_profit_reach):
    active        = True
    current_stop  = lock_min_profit
    anchor_pnl    = when_profit_reach
    → Log: TRAIL_TARGET_ACTIVATED

Phase 2 — Exit check (every tick while active):
    if curr_pnl <= current_stop:
        → SqOff  (Log: PORTFOLIO_SQOFF | Reason=TARGET_TRAIL)

Phase 3 — Ratchet up (every tick while active and pnl > stop):
    gain = curr_pnl - anchor_pnl
    if gain >= trail_every:
        steps         = int(gain / trail_every)
        current_stop += steps * trail_by
        anchor_pnl   += steps * trail_every
        → Log: TRAIL_TARGET_UPDATED | current_stop=X

Note: Trailing target sqoff always uses local SqOff regardless of the On Target Action setting — no cross-portfolio actions are triggered on trailing target exits.

        "initialized":       False,  # runtime
    },

    # ── Combined Premium tracking ──────────────────────────────
    "p_comb_prem_initial":    float,   # captured on first full-fill tick
    "p_comb_prem_captured":   bool,
    "p_comb_prem_legs_added": set,     # leg IDs already included in initial sum
    "p_lot_qty":              int,     # = QTY (backtester.py L45); used to scale
                                        #   Combined Premium SL/TGT value → PnL threshold
}

7. Trail State (Per Portfolio, Mutable Each Tick)

Initialised by init_trail_state(state, risk) on the first tick any leg is active:

state["trail_state"] = {
    "sl": {
        "current_val": risk["p_sl_val"],   # effective SL threshold (decreases as profit grows)
        "anchor_pnl":  0.0,
        "active":      True,
    },
    "target": {
        "active":       False,             # True once PnL >= when_profit_reach
        "current_stop": 0.0,               # locked profit floor (increases)
        "anchor_pnl":   0.0,
    },
}

8. Evaluation Order (Per-Tick)

For each tick:
  1. check_global_user_limits()            ← cross-portfolio user-level limits
  2. For each portfolio:
     a. init_trail_state()                 ← first tick only
     b. check_portfolio_sl_tgt()
          i.   Trailing SL ratchet         ← tighten SL if PnL improved
          ii.  Fixed SL condition check    ← Combined Loss / Premium / Underlying / Range
          iii. Trailing Target activation  ← lock floor when PnL reaches threshold
          iv.  Trailing Target ratchet     ← raise floor as PnL keeps improving
          v.   Trailing Target exit check  ← fire if PnL drops to floor
          vi.  Fixed Target condition check
     c. handle_portfolio_delay_and_action()
          - If p_delay_pending: check elapsed vs configured delay
          - If elapsed: dispatch action (force_sqoff / cross-portfolio flags)
     d. process_portfolio_sqoff()          ← close legs (full or filtered) if force_sqoff
     e. MoveSLManager tick                 ← reposition individual leg SLs if triggered

10. Global User Limits (separate from portfolio SL/TGT)

Checked via check_global_user_limits() across all portfolios per user per tick.

Parameter Format Behaviour
MaxLoss Numeric string Sqoff all user portfolios when combined PnL ≤ −MaxLoss
MaxProfit Numeric string Sqoff all user portfolios when combined PnL ≥ MaxProfit
ProfitLock "reach~lock~every~trail" Trailing profit lock across all user portfolios

Profit Lock logic:

Phase 1 — Activate when combined_pnl >= reach
    current_lock = lock;  anchor = reach
Phase 2 — Exit when combined_pnl <= current_lock
    sqoff all user portfolios → GLOBAL_TRAIL_STOP
Phase 3 — Ratchet when gain = combined_pnl - anchor >= every
    current_lock += steps * trail;  anchor advances
