What is RBO

The Range Breakout (RBO) subsystem replaces standard start-time entry with a two-phase market-structure entry: the engine first builds a price range from the underlying index, then fires entries when the price breaks above or below that range.

RBO is configured per portfolio via the Range BreakOut Settings tab. When enabled, all entry timing is controlled by the RBOManager state machine — the individual legs' own Start Time fields are used only when a leg's Idle=True flag explicitly excludes it from RBO.

State Machine

Every RBO-enabled portfolio runs an independent 4-phase state machine, reset fresh each trading day:

PhaseEntry ConditionWhat HappensExit Condition
IDLE Always at day start Nothing — waiting for monitoring window to open current_time ≥ range_monitoring_start
MONITORING Auto-entered from IDLE Track rolling high = max(bar.high) and low = min(bar.low) every tick current_time ≥ range_monitoring_end
ENTRY Auto-entered from MONITORING Range frozen; detect HIGH/LOW breakouts; set rbo_trigger on eligible legs current_time > effective_entry_end or cancel_other fired
DONE From ENTRY (timeout or cancel_other) No further breakout checks; re-executions still allowed via execute_trigger End of day (state reset next morning)
The range is built from underlying index OHLC data (monitoring = "Underlying"). Once range_monitoring_end is reached the high/low snapshot is frozen — intraday moves after that point do not expand the range.

Parameters

P1 rbo_enabled Checkbox
Default: false JSON key: rbo_enabled GUI: "Enable RangeBreakOut" checkbox

Master toggle. When false, the entire RBO state machine is skipped — all legs enter via their own Start Time (standard time-based entry). When true, the portfolio hands entry control to RBOManager.

If rbo_enabled = true but monitoring times are missing or monitoring ≠ "Underlying", an error is logged and the portfolio falls back to time-based entry for that day.
P2 range_monitoring_start Time HH:mm:ss
Default: 09:30:00 JSON key: range_monitoring_start

The time at which the state machine transitions IDLE → MONITORING and begins tracking the daily range. From this point the engine records max(bar.high) and min(bar.low) from every OHLC tick until range_monitoring_end.

Winter time adjustment: When the trade date falls in the US winter calendar, all RBO times are automatically shifted +1 hour before RBOManager is created. A configured value of 09:30:00 becomes effective 10:30:00, keeping range logic aligned with actual data timestamps.
P3 range_monitoring_end Time HH:mm:ss
Default: 10:30:00 JSON key: range_monitoring_end

The time at which range building stops and the state machine transitions MONITORING → ENTRY. At this tick the high and low values are frozen permanently for the day. No further OHLC data changes the range boundaries.

GUI sync: Changing range_monitoring_end in the GUI automatically updates entry_start to match, enforcing the invariant that breakout detection can only begin after the range is fully built.

P4 entry_start Time HH:mm:ss
Default: Auto-synced to range_monitoring_end JSON key: entry_start

The earliest time within the ENTRY phase at which breakout detection is active. If the index breaches the range before entry_start, the breakout is silently ignored. This allows a deliberate quiet gap between range lock and entry eligibility.

In practice entry_start is usually set equal to range_monitoring_end (no gap), which is why the GUI auto-syncs them. Set it later only if you want to ignore early breakouts.

P5 entry_end Time HH:mm:ss
Default: 16:15:00 JSON key: entry_end

The base deadline for new breakout entries. Once current_time > effective_entry_end with no breakout having fired, the state machine moves to DONE.

entry_end is the base value. The actual effective deadline may be extended by range_buffer before the first side fires, and collapses back to entry_end after the first side fires.

Re-executions (legs re-entering after SL/Target via execute_trigger) are not gated by entry_end. They continue up to end_time (portfolio-level 16:15 default).
P6 range_buffer Integer (minutes)
Default: 0 (disabled) JSON key: range_buffer GUI label: "Range Buffer (mins)"

Extends the breakout entry window beyond entry_end — but only until the first side fires.

Buffer Calculation

effective_entry_end = entry_end + range_buffer  ← while triggered_sides is empty
effective_entry_end = entry_end                  ← once any side has fired

Buffer Collapse

The moment the first breakout fires (HIGH or LOW), triggered_sides becomes non-empty and effective_entry_end collapses instantly back to the original entry_end. The second direction can only enter if it also breaks out before entry_end — which may already have passed.

Example — buffer=300 min

TimeEventeffective_end
10:30Range locked16:00 (11:00 + 300 min)
12:30HIGH breakout fires → trades execute11:00 (collapses!)
12:45LOW breakout detected12:45 > 11:00 → BLOCKED
Buffer collapse confirmed live in RBO_TEST001 on 06-Jan, 08-Jan, 24-Jan via rbo5. After HIGH fired in the buffer window, LOW breakout was dispatched to rbo1/rbo4 but silently blocked for rbo5.
P7 entry_at Dropdown (5 options)
Default: Any JSON key: entry_at

Controls which legs execute on which breakout direction. The rule is evaluated per-leg inside RBOManager.get_legs_to_execute() after each breakout is detected. Only legs with Idle=False and status=IDLE are considered.

Any
All non-Idle legs fire on both HIGH and LOW breakouts. Both directions are fully independent.
RangeHigh
Legs only fire on a HIGH breakout. LOW breakouts are ignored for this portfolio entirely.
RangeLow
Legs only fire on a LOW breakout. HIGH breakouts are ignored.
C_OnHigh_P_OnLow
Call legs enter on HIGH breakout; Put legs enter on LOW breakout. Directional: buy bullish when market breaks up, bearish when it breaks down.
P_OnHigh_C_OnLow
Put legs enter on HIGH breakout; Call legs enter on LOW breakout. Counter-directional: sell volatility in the breakout direction.

Leg Type Detection

The C/P field on each leg is normalized: "C", "CE", "CALL", "0" → Call. "P", "PE", "PUT", "1" → Put.

P8 monitoring Dropdown (2 options)
Default: Underlying JSON key: monitoring

Determines the data source used to build and track the intraday range.

ValueBehaviour
Underlying Range tracked against the SPY (or configured symbol) index OHLC data loaded from parquet. This is the primary, validated mode.
Other strategies Reserved — not yet implemented. Setting this value causes error = "RBO Monitoring must be set to 'Underlying'" and RBO is disabled for the portfolio.
Validation: if monitoring ≠ "Underlying", the engine logs an error and the portfolio falls back to standard time-based entry for the entire day.
P9 cancel_other Checkbox
Default: false JSON key: cancel_other GUI label: "Cancel Other Side if One Executes"

Controls whether executing one breakout side permanently blocks the other side for the rest of the day.

ValueBehaviour
true After the first breakout executes, RBOManager.mark_executed() sets current_phase = "DONE". No further breakouts are checked. Maximum 1 entry cluster per portfolio per day.
false Both HIGH and LOW sides remain independently eligible. Each side can trigger at most once (enforced by triggered_sides), but both can fire on a dual-breakout day.
Interaction with entry_at: cancel_other = true with entry_at = "Any" means the first direction to break out wins and the portfolio is done. With entry_at = "RangeHigh" and cancel_other = true, the behaviour is functionally identical to cancel_other = false since only one direction can ever trigger.

Leg Idle Flag — RBO Mode Inversion

When RBO is active, the Idle checkbox on individual legs has inverted semantics compared to non-RBO mode:

RBO StateIdle = falseIdle = true
RBO enabled ✅ RBO leg — enters on breakout signal (rbo_trigger) ⛔ Excluded from RBO — enters at its own Start Time or via execute_trigger
RBO disabled ✅ Normal time-based entry at Start Time ⏳ Waits for external execute_trigger only (legacy waiting leg) 
# execution.py — Idle flag inversion logic
_rbo_active = state["rbo"].enabled and not state["rbo"].error
if _rbo_active:
    is_rbo_leg = not leg.get("Idle", False)   # Idle=False → RBO leg
else:
    is_rbo_leg = leg.get("Idle", False)        # Idle=True  → waiting leg (legacy)

Use case (rbo6): Legs 1+2 have Idle=False — they enter on breakout. Leg 3 has Idle=True — it enters unconditionally at portfolio start time (09:30 / 10:30 winter). This creates a mixed portfolio: breakout-triggered volatility trade + always-on position.

Re-Entry After SL/Target (execute_trigger)

When a leg exits via SL or Target and has a re-execute action configured, the replacement leg activates with execute_trigger = True. This flag bypasses the RBO gate entirely — no rbo_trigger is required. Re-entries are not limited by entry_end and continue up to end_time (portfolio-level 16:15 default).

# execution.py — execute_trigger takes priority over all RBO gates
if leg.get("execute_trigger"):
    can_execute = True
    leg["rbo_trigger"] = False    # execute_trigger overrides rbo gate

This design allows re-executions to happen deep into the afternoon even when new breakout entries closed at 11:00. The RBO phase (DONE) does not interfere with re-entries.

End-to-End Flow (Per Day, Per Portfolio)

Day Start
│
├─ RBOManager initialised with winter-adjusted times
│
├─ IDLE phase
│   └─ Tick by tick: current_time < range_monitoring_start → nothing
│
├─ MONITORING phase  (current_time >= range_monitoring_start)
│   ├─ Each tick: high = max(high, bar.high)
│   │             low  = min(low,  bar.low)
│   └─ At range_monitoring_end: range frozen → transition to ENTRY
│
├─ ENTRY phase
│   ├─ If current_time < entry_start → skip tick
│   │
│   ├─ Compute effective deadline:
│   │   triggered_sides empty?  →  effective_end = entry_end + buffer
│   │   triggered_sides non-empty? →  effective_end = entry_end
│   │
│   ├─ If current_time > effective_end → DONE, return
│   │
│   ├─ bar.high > range.high  AND  "HIGH" ∉ triggered_sides?
│   │   └─ triggered_sides.add("HIGH")
│   │   └─ log: RBO_BREAKOUT | Type=HIGH | Range=[low, high]
│   │   └─ get_legs_to_execute("HIGH")
│   │       └─ filter: status=IDLE, Idle=False, entry_at rule matches leg C/P
│   │   └─ set rbo_trigger=True + rbo_break_type="HIGH" on eligible legs
│   │   └─ mark_executed("HIGH")  →  if cancel_other: phase = DONE
│   │
│   └─ bar.low  < range.low   AND  "LOW"  ∉ triggered_sides?
│       └─ (same logic, Type=LOW)
│
├─ Leg entry via process_idle_leg_entries()
│   └─ rbo_trigger=True → can_execute=True
│   └─ entry_reason_str set to "Entry on HIGH/LOW breakout"
│   └─ rbo_trigger cleared after leg activates
│
└─ DONE / End of Day
    └─ EOD forced exit of any remaining active legs
