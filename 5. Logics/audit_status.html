12. Gaps & Fixes Applied

The original audit found 5 fields where the UI accepted a value that the backend silently ignored. Each is now visually grayed (.pf-live-only) so users can see at a glance that the value will not affect a backtest. Plus 2 partials (max_loss/max_profit) which got tooltips clarifying their post-hoc-only behavior.

Gap 1 — trail_wait_trade

Why a no-op: Spec is for options-leg "Wait & Trade" sequencing. FX/crypto slots are independent and have no inter-leg trade-wait coordination.

Backend evidence: backtest_runner.py:630 — logs warning, discards value.

Fix applied: UI row reclassed to pf-live-only.

Gap 2 — pf_sl_reexecute_count / pf_tgt_reexecute_count

Why a no-op: v1 portfolio-clip is post-hoc — trades after the SL/TP boundary are dropped from results, but each per-slot engine has already been disposed and cannot be replayed in-place.

Backend evidence: backtest_runner.py:760, 822 — values stored on the run config but no replay loop reads them.

Fix applied: Both number inputs reclassed to pf-live-only. A future replay-capable design can reactivate them.

Gap 3 — move_sl_hit_on_leg_sl / move_sl_hit_on_leg_target

Why a no-op: These are cross-slot triggers (any leg's SL/Target hit moves SL on others). Our forex/crypto runs use a per-slot BacktestEngine with no cross-slot event bus, so the trigger has nowhere to fire from.

Backend evidence: core/models.py:213-214 — fields exist; core/managed_strategy.py never reads them.

Fix applied: Both checkboxes reclassed to pf-live-only.

Partial 1 — max_loss / max_profit

Why partial: Implemented as post-hoc boundary flags (max_loss_hit / max_profit_hit in result), not as in-run hard stops. A backtest that exceeds max_loss mid-run will continue and only flag the breach at the end.

Backend evidence: backtest_runner.py:3074-3075.

Fix applied: Tooltip on the input clarifies post-hoc behavior. Hard-stop conversion would require an in-run equity callback inside each per-slot engine plus cross-slot aggregation; out of scope for this round.

Confirmed already-correct (no code change needed)

  • exit_sell_first — already pf-live-only.
  • on_portfolio_complete cross-portfolio actions — already pf-live-only.
  • Dynamic Hedge entire section — already pf-ui-only (red badge).
  • All 6 monitoring fields — already pf-live-only.
  • Execution-Parameter live fields (Product, Strategy Tag, Execution Mode, Based On, etc.) — already pf-live-only.

Side bug observed (Test 5)

When pf_sl clip fires across multiple slots simultaneously, the post-clip aggregation pipeline crashes with ValueError: could not convert string to float: '-8.26 USD-8.26 USD-7.47 USD-9.04 USD-9.04 USD-9.61 USD'. Several leg-PnL Money strings are concatenated without a separator and then passed to float(). The clip itself worked — this is a downstream serialization bug in the per-clip PnL summing path. Not introduced by this audit; flagging as a follow-up. The audit's functional finding stands: pf_sl_enabled is wired and the clip fires on the configured threshold.

Combined verdict — 16/16 PASS

All four pillars of the .txt design are observably correct end-to-end:

  • Multiplier scaling — T01-T06 prove order quantities scale exactly by the user's multiplier under realistic complex portfolio settings (RBO, trailing SL/TP, weekday filters, move-SL). The admin cap binds last (T06).
  • Allowlist enforcement — T07-T09 prove rejection happens on all four entry points before any worker spawns (save, portfolio backtest, standalone stream, standalone sync).
  • Per-user isolation — T10/T15 prove portfolios and reports live in separate <user_id>/ subdirectories with no cross-user leakage.
  • Live admin edits — T16 proves users.json is consulted on every request; admin changes apply on the very next call without server restart.

What is the multiplier for?

A per-user position-size scalar applied to every order quantity. The same slot config (lots, lot_size, exit rules) produces different fills for different users. Concrete uses: capital-allocation tiers (senior trader ×3.0, junior ×0.5), risk-tiered onboarding (test users at ×0.01), and admin-side rebalancing without slot edits. Critical property: the multiplier is applied before the per-symbol trade_size cap, so admin authority is final — a runaway multiplier (T06) cannot bust through.
