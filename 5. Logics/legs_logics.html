Legs Parameter Tab – Logic Documentation

tabs/portfolio.py engine/execution.py engine/monitoring.py engine/target_sl.py engine/move_sl.py dsl/portfolios.json → legs[] Generated: 2026-04-15 · Reviewed: 2026-04-16
The Legs section is the core configuration table inside the Portfolio form. Every portfolio must have at least one leg. Each leg represents a single options contract to be traded (Buy or Sell, Call or Put, at a specific strike and expiry). Parameters control entry timing, exit logic (target/stoploss), and post-exit actions.

Each leg row is stored as a flat dict in portfolios.json under the legs key. The engine loads this config into runtime state (state["legs"]) and processes it on every market tick.

ID
"ID"
PropertyDetail
WidgetQTableWidgetItem (read-only)
TypeInteger string (auto-assigned)
DefaultAuto-incremented row number

Auto-assigned sequential integer when a leg row is added (add_leg_row()). After any deletion, all IDs are reassigned top-to-bottom in delete_leg_row() to keep them contiguous.

The engine uses ID as the canonical leg identifier for execution tracking, log messages, order book entries, and cross-leg references (e.g. Execute action pointing to another leg by its ID).

2
Idle
"Idle"
PropertyDetail
WidgetQCheckBox
TypeBoolean
DefaultFalse (unchecked)

When Idle = True, the leg does not auto-execute at portfolio start time. It sits in status = "IDLE" waiting for one of two triggers:

  • RBO Trigger: A Range Breakout event (rbo_trigger = True) arms this leg for immediate entry.
  • Cross-Portfolio Trigger: A "Start Other Portfolio" action from another portfolio sets execute_trigger = True, bypassing the Idle gate even in Manual mode.
  • Manual Mode: Cross-portfolio triggered Idle legs still respect their individual Start Time before entering.

Engine location: process_idle_leg_entries() → is_rbo_leg = leg.get("Idle", False).

Validation: save_portfolio() validates that any ReEntry action referencing a leg_id must point to a leg that has Idle = True.
3
BUY/SELL
"BUY/SELL"
PropertyDetail
WidgetQComboBox
OptionsBUY, SELL
DefaultSELL

Sets the trade direction for the leg.

DirectionSL triggerTarget triggerExit fillEntry price
SELL bid_high >= sl_val bid_low <= tgt_val max(ask_vwap, hit_price) bid_open → bid_close → close
BUY ask_low <= sl_val ask_high >= tgt_val max(bid_vwap, hit_price) ask_open → ask_close → close
Fill price detail: SL exits — SELL: hit_price = bid_high; BUY: hit_price = ask_low. Target exits — SELL: hit_price = bid_low; BUY: hit_price = ask_high. The max(vwap, hit_price) formula ensures the fill is never worse than the level that triggered the exit.
4
C/P (Call / Put)
"C/P"
PropertyDetail
WidgetQComboBox
OptionsC (Call), P (Put)
DefaultC

Specifies the option type. Normalized by normalize_cp(): C / CALL / CE / 0 → 0, P / PUT / PE / 1 → 1. Used to look up the correct option chain quote from the data layer (get_option_quote()). Also determines the hedge leg's option type when Hedge Required is enabled.

5
Lots
"Lots"
PropertyDetail
WidgetQSpinBox
Range1 – 999
Default1

User-side multiplier (number of contracts). Order quantity is the per-symbol contract size (admin-configured) multiplied by lots, capped by an admin-set per-symbol max:

order_qty = min(instrument.lot_size × slot.lots, instrument.trade_size_cap) └────── admin (venue_config) ──────┘ └─ user (slot) ─┘ └─── admin cap ───┘

Both lot_size and the trade_size cap live in the venue's adapter config under instruments.<symbol> (see adapter_admin/adapters_config/<venue>.json). Example FX shape: { "EURUSD": { "lot_size": 100000, "trade_size": 10000000 } }. Multi-leg strategies (Range Breakout) further multiply the resolved per-trade quantity by legN_lots.

Used in PnL calculation:

pnl = (entry_price - exit_price) × order_qty # SELL pnl = (exit_price - entry_price) × order_qty # BUY

The realized_per_lot metric divides total PnL by lots for normalized reporting.

Migration note: the legacy trade_size slot field has been renamed to lots. Old portfolios are migrated automatically on load — for an FX slot with trade_size: 1000 and venue lot_size: 100000, this becomes lots: 0.01 (preserving the original 1000-unit order quantity).

Days-to-expiry of the target option contract. Parsed by normalize_dte() (strips "DTE" suffix → integer). Stored as target_dte on the leg runtime state. Used by get_option_quote() to filter the option chain to contracts expiring in exactly N days.

7
Strike
"Strike"
PropertyDetail
WidgetQComboBox
OptionsATM+10 … ATM … ATM-10
DefaultATM

Strike selection relative to ATM at the time of entry.

  • ATM resolves to the nearest strike to the underlying spot/futures price.
  • ATM+N / ATM-N offsets by N strike steps (configured via portfolio-level Strike Step).
  • The Strike Selection mode (Normal or Relative) affects how the offset is applied.
  • On Re-Execute, reentry_mode = "same_strike" re-enters at the previously resolved numeric strike instead of re-resolving ATM.

Engine location: process_idle_leg_entries() → safe_strike_key(leg["conf"]["Strike"]).

8
Hedge Required
"Hedge Required"
PropertyDetail
WidgetQCheckBox
TypeBoolean
DefaultFalse

When checked, a dynamic hedge leg is automatically placed immediately after this leg's entry:

  1. Looks up ATM strike via get_atm_strike_fast().
  2. Calls find_hedge_strike() using the Dynamic Hedge tab config.
  3. Creates a system-managed hedge leg with ID H_{parent_lid}_{exec_count}.
  4. Direction is opposite to parent (BUY→SELL hedge, SELL→BUY hedge), same option type. Hedge Required: False prevents recursive hedging.
  5. When the parent exits, the linked hedge leg is simultaneously closed at market price (PARENT EXIT).

Portfolio-level dynamic_hedge["enabled"] is auto-set to True if any leg has Hedge Required = True. Max hedge count enforced by state["max_hedge_executions"].

9
Wait & Trade
"Wait & Trade"
PropertyDetail
WidgetQSpinBox
Range0 – 9999 seconds
Default0 (disabled)

Deliberate delay between entry condition being met and the actual order placement:

  1. wt_pending = True, wt_trigger_time = curr_dt recorded. WAIT_AND_TRADE_PENDING logged.
  2. Leg skips entry while elapsed < wt_delay.
  3. Once elapsed >= wt_delay, wt_pending = False and entry proceeds normally.

Setting Wait & Trade = 0 disables the delay entirely. Engine location: process_idle_leg_entries() lines 937–955.

10
Target Type
"Target Type"
PropertyDetail
WidgetQComboBox
OptionsNone, Premium, AbsolutePremium, Underlying
DefaultNone

Determines how tgt_pending_val is computed at entry time:

ModeComputation
NoneNo target. tgt_pending_val = None. Leg exits on SL, time, or portfolio SqOff only.
PremiumWith %: tgt = entry × (1 - pct/100) for SELL, entry × (1 + pct/100) for BUY. Plain number: tgt = entry - value for SELL, entry + value for BUY.
AbsolutePremiumFixed absolute price level. tgt_pending_val = parsed_value.
Underlying⚠️ Not implemented at leg level. Engine has no handler — tgt_pending_val remains None. Use portfolio-level target instead.
Engine code path (execution.py): AbsolutePremium → fixed value; Premium / PremiumBased → entry-price offset or %; any type containing "Percentage" → percentage-of-entry catch-all; "Underlying" → falls through, no value set.

Target evaluated each tick on bid_low (SELL) or ask_high (BUY). Exit fill = max(ask_vwap, bid_low) for SELL or max(bid_vwap, ask_high) for BUY.

11
Target Value
"Target Value"
PropertyDetail
WidgetQTableWidgetItem (editable text)
TypeNumeric string, supports % suffix
Default""

Numeric parameter for the selected Target Type. Parsed by parse_val() which strips % and converts to float.

  • "20%" with Premium → exit when premium drops 20% from entry (SELL).
  • "5.00" with AbsolutePremium → exit when premium reaches $5.00.
  • Empty or 0 with non-None type → tgt_pending_val remains None (no target armed).
12
Stoploss Type
"Stoploss Type"
PropertyDetail
WidgetQComboBox
OptionsNone, Premium, AbsolutePremium, Underlying
DefaultNone

Mirrors Target Type for the stop-loss direction. sl_pending_val computed at entry:

ModeComputation
NoneNo SL. Leg exits on target, time, or portfolio SqOff only.
PremiumWith %: sl = entry × (1 + pct/100) for SELL, entry × (1 - pct/100) for BUY. Plain number: sl = entry + value for SELL, entry - value for BUY.
AbsolutePremiumFixed absolute price level. sl_pending_val = parsed_value.
Underlying⚠️ Not implemented at leg level. sl_pending_val remains None. Use portfolio-level SL instead.
Alias: Engine also accepts "PremiumBased" as equivalent to "Premium" (backward compatibility). Any type containing "Percentage" is handled as a percentage-of-entry computation.

SL evaluated each tick on bid_high (SELL) or ask_low (BUY). Exit fill = max(ask_vwap, bid_high) for SELL or max(bid_vwap, ask_low) for BUY.

Interaction with Move SL / Trail SL: sl_pending_val is dynamically modified at runtime by (a) MoveSLManager (Move SL to Cost) and (b) the portfolio-level Trailing SL engine, without touching leg["conf"]. If trail_after_move_sl = True, a per-leg trailing SL is activated after the move-to-cost event (tracked via leg["move_sl_trail_anchor"]).

14
On Target Action
"On Target Actions"
PropertyDetail
WidgetLegActionWidget (double-click to open dialog)
TypeList of action dicts
Default[{"type": "SqOff"}]
Storage key"On Target Actions" (list) · "On Target Action" (legacy string)

One or more actions executed when the leg's target is hit. Available actions:

SqOff
Default. Leg is closed (status → CLOSED). PnL realized and recorded.
KeepLegRunning
Hit level cleared (tgt_pending_val = None) but leg stays ACTIVE. Ignored if end_blocked = True — leg closes normally past end_time.
Re-Execute
Leg closed and reset to IDLE for re-entry next tick. Count cap: count=N → up to N re-executions (N+1 total). Suppressed when exec_count >= (1 + N).
Execute
Sets execute_trigger = True on a specific other leg (by Leg ID), making it enter from IDLE on the next tick. Logs a warning if the target leg is not IDLE.
ReEntry
Leg set to REENTRY_WAITING. Waits for premium to return to original entry_price at the same locked numeric strike. Optional count limit (same semantics as Re-Execute).
Stacking rules:
• Multiple actions evaluated in order. SqOff in the list is skipped — leg is already closed before the action loop.
• KeepLegRunning + Re-Execute/ReEntry: engine cannot reuse the ACTIVE leg. A new concurrent leg slot is injected into state["legs"] (keys like "1_re2" or "1_rentry2"), running in parallel with the original.
• SqOff takes precedence when global max_leg_executions is hit or end_blocked = True.

Engine location: engine/monitoring.py → action dispatch loop lines 651–780.

On SL Action
"On SL Actions"
PropertyDetail
WidgetLegActionWidget (double-click to open dialog)
TypeList of action dicts
Default[{"type": "SqOff"}]
Storage key"On SL Actions" (list) · "On SL Action" (legacy string)

Identical structure and action options as On Target Action, but triggered on LEG_SL_HIT events. The is_sl flag passed to get_leg_actions() selects the correct action list.

After action dispatch for a SL hit, MoveSLManager.apply() is called if enabled and conditions match:

  • On SL hit: move_sl_performed == False OR hit_on_leg == True.
  • On Target hit: tgt_hit_on_leg == True.
Same concurrent leg slot behavior applies: KeepLegRunning + Re-Execute/ReEntry spawns new parallel leg slots instead of reusing the ACTIVE leg.
16
SL Wait (sec)
"SL Wait (sec)"
PropertyDetail
WidgetQTableWidgetItem (editable text)
TypeInteger string (seconds)
Default"" → treated as 0

Confirmation delay before executing an SL or Target exit, preventing premature exits on brief wicks:

  1. First tick SL or Target triggers: delay_pending = True, delay_start_time = curr_dt. DELAY_PENDING logged.
  2. Leg not exited while elapsed_seconds < sl_wait_sec.
  3. If condition no longer met (price recovers): delay_pending = False, DELAY_CLEARED logged — exit cancelled.
  4. If trigger persists for full duration, exit proceeds at current price.
Important nuances:
• Applies to both SL and Target exits (shared delay mechanism).
• delay_trigger_reason is locked at step 1 and not re-evaluated during the wait. If condition switches mid-delay (SL → Target or vice versa), exit proceeds under the original trigger reason.
• Setting to 0 or empty disables the delay.

Engine location: engine/monitoring.py → exit delay handling, lines 454–492.

Start Time
"Start Time"
PropertyDetail
WidgetQTimeEdit
FormatHH:mm:ss
Default09:30:00

Individual leg entry time. For non-Idle legs in Start time execution mode, the engine resolves leg_start from this field (parsed via parse_time_str()). Entry only proceeds when curr_time >= leg_start.

Winter Time Adjustment: If timing["is_winter"] = True, the raw start time is shifted forward by 1 hour via add_one_hour() to align data timestamps with the UI-configured time.

Validation: save_portfolio() enforces: Portfolio Start Time ≤ Leg Start Time ≤ Portfolio End Time.

Interaction with Idle legs: Idle legs in Manual execution mode that are cross-portfolio triggered also respect this field — the trigger does not bypass the leg's own start time.

Engine Implementation Notes

⚠️  Underlying Type at Leg Level — Not Implemented

The GUI shows Underlying as an option for both Target Type and Stoploss Type. However, the engine (execution.py) has no handler for this value at the leg level.

The SL/Target computation block only handles: AbsolutePremium, Premium / PremiumBased, and types containing "Percentage". A plain "Underlying" value falls through all conditions, leaving sl_pending_val / tgt_pending_val as None — effectively no SL or Target is armed.

Underlying-based exits are only implemented at the portfolio level via check_portfolio_sl_tgt() in engine/target_sl.py.

Concurrent Leg Slots (KeepLegRunning + Re-Execute / ReEntry)

When KeepLegRunning is active alongside Re-Execute or ReEntry, the engine cannot reuse the original leg (still ACTIVE). It instead injects a new leg dict into state["legs"] with a generated composite key:

Re-Execute concurrent slot key: "{lid}_re{exec_count+1}" ReEntry concurrent slot key: "{lid}_rentry{exec_count+1}"

These new slots are picked up in the same tick's idle/reentry processing loops and behave identically to a normal re-execution or re-entry. This allows a leg to keep running AND spawn a fresh position simultaneously.

Re-Execute / ReEntry Count Semantics

The count parameter means number of allowed re-executions beyond the initial entry. The engine check is:

if limit > 0 and leg["exec_count"] >= (1 + limit): # suppressed — log REEXEC_LIMIT_REACHED

count=2 allows: initial entry (exec_count=1) + 2 re-executions (exec_count=2, 3). At exec_count=3, further re-executions are blocked.

Leg-Level Trailing SL After Move SL to Cost

If MoveSLManager is enabled with trail_after_move_sl = True, after moving a leg's SL to cost (entry price), the engine sets leg["move_sl_trail_anchor"] = ltp.

On subsequent ticks, the engine checks this anchor and reduces sl_pending_val by trail_by for every trail_every gain in leg PnL (using the portfolio-level trailing config). This runs per-leg independently of the portfolio-level trailing SL system.

PremiumBased Alias for Premium

The engine accepts "PremiumBased" as a synonym for "Premium" for both SL and Target types:

elif sl_type in ("Premium", "PremiumBased"):

Backward-compatibility alias. The GUI only shows "Premium" but older saved portfolios may contain "PremiumBased".

Parameter Validation Rules (enforced on Save)

  • 1Portfolio Name must not be empty.
  • 2At least 1 leg must be present.
  • 3Portfolio timings: Start Time ≤ End Time ≤ SqOff Time.
  • 4Each Leg Start Time: must satisfy Portfolio Start ≤ Leg Start ≤ Portfolio End.
  • 5ReEntry action: target Leg ID must exist in the portfolio AND that leg must have Idle = True.
  • 6Range Breakout timings (if enabled): Portfolio Start ≤ RBO Range Start ≤ RBO Range End ≤ Entry Start ≤ Entry End ≤ Portfolio End ≤ SqOff.
  • 7Execution Mode + Based On: if mode is not Start time or Manual and Based On = None, Entry Price must be non-zero.
  • 8Portfolio Target/Stoploss cross-portfolio actions: referenced portfolio name must exist.
