01

Straddle Width Multiplier

Numeric — negative allowed 0.00 = Disabled

Overrides the strike selected for ATM-based legs by using the combined ATM premium (CE + PE) at portfolio start as a width measure, then shifting each leg's strike by combined × multiplier.

⚠ Important constraints:
  • 0.00 = Feature disabled (default). When set to 0.00, the multiplier block is skipped entirely — no strike override is applied and the leg uses its configured strike as normal (ATM, ATM+1, ATM−2, etc.).
  • Overrides the leg's configured strike (e.g. ATM, ATM+1). Use carefully — it can significantly change which strikes are traded.
  • Not applicable to Premium / Greek-based leg portfolios (Premium Greek flag checked in General settings).
  • Also not applied when a leg is in same_strike re-entry mode (the locked numeric strike is used as-is).
  • Negative values are fully supported and produce ITM selections.

Formula

# At portfolio start — captured once, reset on each re-execution combined_premium = ATM_CE_premium + ATM_PE_premium # For CALL (CE) legs: CE_strike = ATM + (combined_premium × multiplier) # For PUT (PE) legs: PE_strike = ATM − (combined_premium × multiplier) # Rounded to nearest strike_step (SPY default = $1) final_strike = round(raw_strike / strike_step) × strike_step

SPY Examples

Assume SPY spot = $525.30, ATM = $525.
ATM CE premium = $3.10, ATM PE premium = $2.90.
Combined = $6.00. Strike step = $1.

✅ Multiplier = +0.50 (OTM selection)
CE leg:
525 + (6.00 × 0.50) = 525 + 3.00 = $528 CE

PE leg:
525 − (6.00 × 0.50) = 525 − 3.00 = $522 PE

Result: OTM strangle — both legs are out-of-the-money.
🔄 Multiplier = −0.50 (ITM selection)
CE leg:
525 + (6.00 × −0.50) = 525 − 3.00 = $522 CE

PE leg:
525 − (6.00 × −0.50) = 525 + 3.00 = $528 PE

Result: ITM strangle — both legs are in-the-money.
✅ Multiplier = +1.00 (full width OTM)
CE leg: 525 + 6.00 = $531 CE
PE leg: 525 − 6.00 = $519 PE

Full combined-premium width strangle.
⛔ Multiplier = 0.00 — Feature Disabled
The multiplier block is skipped entirely. No strike override is applied.

Each leg falls back to its own configured strike (ATM, ATM+1, ATM−2, etc.) as if this setting did not exist.

0.00 is the default — set any non-zero value to activate.

Premium Capture Timing

1
Portfolio reaches its Start Time — first leg becomes eligible to enter.
2
Engine fetches the ATM CE bid/ask mid and ATM PE bid/ask mid from the current tick's option chain.
3
combined_premium is stored in portfolio state. All subsequent legs in this cycle use the same stored value.
R
On re-execution (SL/TGT hit → ReExecute), the cached premium is cleared. The next execution tick re-captures it at the current ATM level.

02

Delay Between Legs in Sec.

Integer seconds · default 0

A post-re-execution delay. After a portfolio SL or Target is hit and the action is ReExecute, this timer must fully elapse before any IDLE leg is allowed to re-enter the market.

Scope: This setting applies only to re-executions triggered by portfolio-level SL / Target actions. It does not apply to the initial (first-time) leg entry — use the per-leg Wait & Trade field for that.

Execution Timeline

# Example: Delay Between Legs = 5 seconds 10:30:35 — Portfolio Target hit, all legs squared off 10:30:35 — REEXEC_DELAY_STARTED logged, countdown begins 10:30:36 — tick: delay not elapsed, no leg entry 10:30:37 — tick: delay not elapsed, no leg entry 10:30:38 — tick: delay not elapsed, no leg entry 10:30:40 — tick: 5s elapsed → REEXEC_DELAY_ELAPSED logged → IDLE legs now allowed to enter

Difference vs. Wait & Trade

Triggers on Initial leg entry (first execution) Re-execution after SL/TGT hit

Default 0 (no delay) 0 (no delay)

03

On Target Action On

Dropdown

When the portfolio has both a fixed Target and a Trailing Target enabled, this setting controls which type of hit triggers the configured Target Action (e.g. SqOff, ReExecute, Execute Other Portfolio, etc.).

Why this matters: Without this filter, both the fixed Target and the Trailing Target would fire the same action. This caused issues where users wanted only the trailing target to close and re-enter, but the fixed target was also triggering the re-execution (or vice-versa).
OnTarget_N_Trailing_Both (default)
Target Action fires on any target hit — whether the fixed profit target is reached or the trailing lock-profit target is hit.

Use when the same action should happen regardless of how the target triggered.
OnTarget_Only
Target Action fires only when the fixed portfolio profit target is hit.

A trailing target hit is silently ignored — no action is taken, the portfolio continues running. A suppression log is emitted.
OnTarget_Trailing_Only
Target Action fires only when the trailing lock-profit target is hit.

A fixed target hit is silently ignored — no action is taken, the portfolio continues running. A suppression log is emitted.

Decision Flow

# Internal reason codes produced by check_portfolio_sl_tgt() Fixed target hit → p_reason = "TARGET" Trailing hit → p_reason = "TARGET_TRAIL" # Filter applied in handle_portfolio_delay_and_action() if on_target_action_on == "OnTarget_Only" and p_reason == "TARGET_TRAIL": → SUPPRESS (log ON_TARGET_ACTION_SUPPRESSED, continue) if on_target_action_on == "OnTarget_Trailing_Only" and p_reason == "TARGET": → SUPPRESS (log ON_TARGET_ACTION_SUPPRESSED, continue) else: → DISPATCH ACTION (SqOff / ReExecute / Execute Other Portfolio / etc.)
Suppression clears pending delay: If the target fired while a portfolio delay (p_tgt_delay) was pending, suppression also cancels that pending delay. This prevents stale delays from firing if the target condition re-appears later in a different form.

Log Messages

Log KeyWhen
ON_TARGET_ACTION_SUPPRESSEDHit was suppressed by this filter — includes Reason and Filter fields
04

On SL Action On

Dropdown

When the portfolio has both a fixed Stoploss and a Trailing Stoploss enabled, this setting controls which type of SL hit triggers the configured SL Action (SqOff, ReExecute, etc.).

How trailing vs. fixed SL is identified: The engine tracks whether the portfolio's SL value has ever been moved by the trailing logic (was_trailed flag). If the SL has been trailed and then the PnL falls below that trailed level, the hit is classified as STOPLOSS_TRAIL. If no trailing has occurred, it is classified as STOPLOSS.
OnSL_N_Trailing_Both (default)
SL Action fires on any SL hit — whether the configured fixed SL is hit directly or the trailing SL has moved and then been crossed.

Equivalent to the previous default behaviour.
OnSL_Only
SL Action fires only on a fixed SL hit (the SL value was never trailed).

A trailing-SL hit is silently ignored — no action taken, monitoring continues.
OnSL_Trailing_Only
SL Action fires only when the trailing SL has moved and then been crossed.

A fixed SL hit (before any trailing has occurred) is silently ignored.

Decision Flow

# Internal reason codes produced by check_portfolio_sl_tgt() Fixed SL hit (trailing never activated) → p_reason = "STOPLOSS" Trailing SL hit (SL was moved, then crossed)→ p_reason = "STOPLOSS_TRAIL" # Filter applied in handle_portfolio_delay_and_action() if on_sl_action_on == "OnSL_Only" and p_reason == "STOPLOSS_TRAIL": → SUPPRESS (log ON_SL_ACTION_SUPPRESSED, continue) if on_sl_action_on == "OnSL_Trailing_Only" and p_reason == "STOPLOSS": → SUPPRESS (log ON_SL_ACTION_SUPPRESSED, continue) else: → DISPATCH ACTION (read p_sl_action from portfolio SL config)

Trailing SL — Step-by-Step Example

Portfolio SL = $500 loss limit. Trailing SL: trail every $200, trail by $100. On SL Action On = OnSL_Trailing_Only.

PnL = +$0 → SL at -$500. No trailing yet. was_trailed = False PnL = +$200 → Trailing step: SL moves to -$400. was_trailed = True PnL = +$400 → Trailing step: SL moves to -$300. was_trailed = True PnL = -$300 → PnL fell to trailed SL → reason = STOPLOSS_TRAIL OnSL_Trailing_Only → ACTION FIRES ✅ # If instead PnL dropped to -$500 before trailing ever moved: PnL = -$500 → reason = STOPLOSS OnSL_Trailing_Only → SUPPRESS — portfolio keeps running

Parameter Interactions & Edge Cases

Reference

Straddle Width Multiplier + Re-Execution

When the portfolio re-executes (SL or Target hit → ReExecute action), the stored combined_premium is reset. The next time a leg enters, the engine re-fetches the ATM CE and PE premiums at the current tick, which may differ significantly from the original entry. This ensures the strike selection always reflects the current market width, not a stale value from the start of the day.

On Target / On SL Action On + Delay

The action-type filter runs before the portfolio-level delay guard. This means:

  • If a suppressed hit type fires while a delay is counting down, the delay is cancelled (not just paused). The portfolio resumes normal monitoring.
  • If the unsuppressed hit type fires later, a fresh delay countdown begins.

Delay Between Legs + End Time

If the portfolio's End Time is reached while a re-execution delay is still counting down, the delay is ignored and no new leg entries are allowed (the end_blocked flag prevents entries regardless of delay state). This is consistent with existing end-time behaviour.
