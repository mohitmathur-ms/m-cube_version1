Trigger Conditions

Triggers use raw bid/ask side high/low — not mid-price — sourced from 1-second bid/ask OHLC data.

Leg DirectionSL TriggerTARGET Trigger
SELL bid_high >= sl_val bid_low <= tgt_val
BUY ask_low <= sl_val ask_high >= tgt_val
Rationale: For a SELL position, risk is the option price rising — the bid side going high is the earliest directional signal. For a BUY position, risk is the price falling — the ask side going low signals the breach. Using raw bid/ask avoids mid-price smoothing which can delay triggers by up to half a spread.

SL & Target Fill Price Formula

The same VWAP-based formula applies to both SL and Target exits:

-- Universal formula --
exit_price = vwap if vwap > hit_price else hit_price
EventDirectionHit PriceVWAP Used
SLSELLbid_highask_vwap (buying back at ask)
SLBUYask_lowbid_vwap (selling at bid)
TARGETSELLbid_lowask_vwap (buying back at ask)
TARGETBUYask_highbid_vwap (selling at bid)
 
# SL   — SELL:  exit = ask_vwap if ask_vwap > bid_high  else bid_high
# SL   — BUY:   exit = bid_vwap if bid_vwap > ask_low   else ask_low
# TGT  — SELL:  exit = ask_vwap if ask_vwap > bid_low   else bid_low
# TGT  — BUY:   exit = bid_vwap if bid_vwap > ask_high  else ask_high
Conservative fill model: If the second's VWAP is higher than the trigger-level price (bid/ask H or L that fired the condition), the VWAP is used. Otherwise the trigger-level price itself is the fill. This gives the worst-case, realistic fill for both SL and Target exits.

SL Wait / Delay Logic

On trigger second:
  delay_pending    = True
  delay_start_time = current_dt

Each subsequent second:
  if condition reverses → delay_pending = False  # DELAY_CLEARED
  elif elapsed < SL_Wait_sec → skip (hold)
  else → execute exit

B. Trail After Move SL

When trail_after_move_sl = True, once SL is moved to cost the leg enters step-based tightening using the portfolio's sl_trail config:

# Set once when Move SL fires:
leg["move_sl_trail_anchor"] = ltp

# Each second thereafter:
leg_gain = curr_leg_pnl - move_sl_trail_anchor
if leg_gain >= sl_trail["every"]:
    steps = int(leg_gain / sl_trail["every"])
    new_sl = sl_pending_val ± (steps × sl_trail["by"])   # ± depends on SELL/BUY
    move_sl_trail_anchor += steps × sl_trail["every"]
    # log: LEG_TRAIL_SL_UPDATED

Leg Actions (5 Types)

SqOff
"type": "SqOff"
Default behavior. Leg closes at fill price. No re-entry. This is the fallback when KeepLegRunning is absent — including SqOff in the list is optional.
Re-Execute
"type": "Re-Execute"
"count": N
Leg re-enters at market. If KeepLegRunning is also active: spawns a new concurrent slot. Otherwise: current leg → IDLE for immediate market re-entry. Count-limited.
Execute
"type": "Execute"
"leg_id": "X"
Triggers an IDLE leg (by leg_id) to execute immediately, regardless of current leg status. Requires target leg to be in IDLE state.
ReEntry
"type": "ReEntry"
"count": N
Leg closes, then waits at a specific price for re-entry. Count-limited.
KeepLegRunning
"type": "KeepLegRunning"
Clears the SL or Target condition. Leg stays ACTIVE and keeps running. Disabled automatically if market is closing (end_blocked).
SqOff note: SqOff in the actions list is skipped in the dispatch loop — closure happens by default when KeepLegRunning is absent. Including it alongside other actions is valid for readability but adds no behavior.

Valid Action Combinations (max 3)

Single Actions

ActionValidEffective Behavior
SqOff✓ VALIDLeg closed, no re-entry
Re-Execute✓ VALIDLeg closed → IDLE → re-enters at market (count-limited)
Execute✓ VALIDLeg closed (default), another IDLE leg triggered by leg_id
ReEntry✓ VALIDLeg closed, waits at price for re-entry (count-limited)
KeepLegRunning✓ VALIDSL/Target cleared, leg continues running

Two-Action Combinations

CombinationValidEffective Behavior
Re-Execute + Execute✓ VALIDLeg closed → re-enters and another IDLE leg is triggered simultaneously
Re-Execute + KeepLegRunning✓ VALIDCurrent leg stays running + a new concurrent slot is spawned (no closure of current)
Execute + KeepLegRunning✓ VALIDSL/Target cleared (leg keeps running) + another IDLE leg triggered
Execute + ReEntry✓ VALIDLeg closed, another IDLE leg triggered, current leg waits to re-enter at price
SqOff + Re-Execute✓ VALIDSame as Re-Execute alone (SqOff is skipped in dispatch loop)
SqOff + Execute✓ VALIDSame as Execute alone
Re-Execute + ReEntry✗ INVALIDBoth attempt leg re-entry via different mechanisms; Re-Execute sets leg to IDLE for immediate market entry, conflicting with ReEntry's price-wait logic
KeepLegRunning + ReEntry✗ INVALIDKeepLegRunning prevents closure; ReEntry requires a closed leg to set up its price-wait. Contradictory.
SqOff + KeepLegRunning✗ INVALIDClosure (default) directly contradicts KeepLegRunning. KeepLegRunning wins but intent is nonsensical.

Three-Action Combinations

CombinationValidEffective Behavior
Re-Execute + Execute + KeepLegRunning✓ VALIDKeep current leg running + spawn concurrent new slot + trigger another IDLE leg
Re-Execute + KeepLegRunning + Execute✓ VALIDSame as above (order irrelevant)
SqOff + Re-Execute + Execute✓ VALIDEquivalent to Re-Execute + Execute (SqOff skipped)
Re-Execute + Execute + ReEntry✗ INVALIDRe-Execute and ReEntry conflict (see two-action rule)
Execute + ReEntry + KeepLegRunning✗ INVALIDKeepLegRunning + ReEntry conflict (see two-action rule)
SqOff + Re-Execute + KeepLegRunning✗ INVALIDSqOff + KeepLegRunning conflict

Impossible Combinations — Summary

RuleReason
KeepLegRunning + ReEntryReEntry requires the leg to be closed first; KeepLegRunning prevents closure
KeepLegRunning + SqOffMutually exclusive exit intents; KeepLegRunning wins but the combination is logically inconsistent
Re-Execute + ReEntryTwo competing re-entry mechanisms on the same leg; Re-Execute wins (IDLE state) but ReEntry's price-wait never activates — undefined state
More than 3 actionsNot hard-blocked in code but unsupported and unvalidated; behavior is undefined beyond 3
Execute without valid leg_idIf leg_id is absent or the target leg is not in IDLE state, Execute silently does nothing
Re-Execute with count = 0Zero count means unlimited re-executions; leg will re-execute every SL hit until market close — only valid if intentional

TRAILING Portfolio Trailing SL

Configured in stoploss.trailing block. The SL tightens in discrete steps as portfolio profit grows.

ParameterKeyDescription
EnabledenabledToggle
Trail Everytrail_everyProfit gain increment that tightens the SL by one step
Trail Bytrail_byAmount by which the SL limit is reduced per step
 
# Initial state at portfolio activation:
trail_state["sl"]["current_val"] = p_sl_val
anchor_pnl = 0

# Each second:
gain = curr_pnl - anchor_pnl
if gain >= trail_every:
    steps = int(gain / trail_every)
    current_sl -= steps × trail_by     # SL tightens
    anchor_pnl += steps × trail_every
    # log: TRAIL_SL_UPDATED

# Then check SL hit using current_sl (not original p_sl_val)
Note: The trailing SL only ever tightens — it never loosens. Once steps are taken they are permanent for that day.

TRAILING Portfolio Trailing Target (Profit Lock)

Configured in target.trailing block. A profit-locking ratchet stop — locks minimum profit and tightens the lock as profit grows.

ParameterKeyDescription
EnabledenabledToggle
When Profit Reachwhen_profit_reachActivation threshold — trailing activates when PnL first crosses this
Lock Min Profitlock_min_profitMinimum profit locked at activation (floor of the exit)
Trail Everytrail_everyProfit gain increment that raises the lock by one step
Trail Bytrail_byAmount by which the lock floor is raised per step
 
# Activation:
if curr_pnl >= when_profit_reach:
    active = True
    current_stop = lock_min_profit
    anchor_pnl   = when_profit_reach
    # log: TRAIL_TARGET_ACTIVATED

# Active phase (each second):
if curr_pnl <= current_stop:
    → p_hit = True  # exits portfolio  | TRAIL_TARGET
else:
    gain = curr_pnl - anchor_pnl
    if gain >= trail_every:
        steps = int(gain / trail_every)
        current_stop += steps × trail_by    # lock ratchets up
        anchor_pnl   += steps × trail_every
        # log: TRAIL_TARGET_UPDATED
Lock is permanent: The floor only ever rises. Once activated it overrides the fixed target; the fixed target check is guarded by and not p_hit so it cannot fire while trailing target is active.

Portfolio Actions on SL / Target Hit

ActionBehavior
SqOffClose all ACTIVE legs in this portfolio. Standard exit.
SqOff Other PortfolioClose this portfolio + force-close all ACTIVE legs in target_portfolio
Execute Other PortfolioClose this portfolio + set execute_trigger = True on all IDLE legs in target_portfolio
Start Other PortfolioClose this portfolio + resume execution mode of target_portfolio (sets to ACTIVE or WAITING_FOR_ENTRY depending on its entry mode)

ReExecute SameStrike at EntryPriceClose this portfolio + re-enter at original strikes and entry prices
Trailing actions (TARGET_TRAIL) always use SqOff regardless of what action is configured — cross-portfolio actions are not dispatched from trailing exits.

Portfolio Delay Logic

# On trigger second:
p_delay_pending    = True
p_delay_start_time = current_dt

# Each subsequent second:
if condition reverses:
    p_delay_pending = False    # PORTFOLIO_DELAY_CLEARED
elif elapsed < p_delay:
    skip (continue holding)
else:
    dispatch action

User Level
Configured in user_settings.json under risk_limits · Aggregates PnL across all portfolios owned by that user · Evaluated in target_sl_bt.check_global_user_limits()

ParameterKeyDescription

Max Profitmax_profitPositive value. If aggregate PnL reaches or exceeds this, all user portfolios are force-squared immediately.
Trail Reachtrail_cfg[0]Profit level at which user-level trailing activates.

User-Level Logic

# Aggregate PnL for this user across all portfolios
combined_pnl = Σ state["pnl"]["combined"]  for state owned by uid

# ── Max Loss ──────────────────────────────────────────────
if max_loss != 0 and combined_pnl <= max_loss:
    force_sqoff all user portfolios
    gst["hit"] = True   # no further checks today | GLOBAL_MAX_LOSS_HIT

# ── Max Profit ────────────────────────────────────────────
elif max_profit != 0 and combined_pnl >= max_profit:
    force_sqoff all user portfolios
    gst["hit"] = True                       # GLOBAL_MAX_PROFIT_HIT

# ── Profit Trailing ───────────────────────────────────────
elif reach > 0:

    # Activate
    if not trail_active and combined_pnl >= reach:
        trail_active  = True
        current_lock  = lock
        anchor_pnl    = reach              # GLOBAL_TRAIL_ACTIVATED

    # Active phase
    if trail_active:
        if combined_pnl <= current_lock:
            force_sqoff all user portfolios
            gst["hit"] = True             # GLOBAL_TRAIL_STOP_HIT
        elif every > 0:
            gain = combined_pnl - anchor_pnl
            if gain >= every:
                steps = int(gain / every)
                current_lock += steps × trail     # GLOBAL_TRAIL_UPDATED
                anchor_pnl   += steps × every

User-Level Constraints

ConditionBehavior
max_loss = 0Max loss check disabled entirely
max_profit = 0Max profit check disabled entirely
reach = 0Profit trailing disabled
gst["hit"] = TrueNo further user-level checks for the rest of the trading day
Max Loss + Max Profit both setLoss checked first, then profit (if / elif). Only one fires per second.
Max Loss/Profit + Profit Trailing simultaneouslyMax Loss/Profit take precedence (if/elif guards). Trailing only runs in the else branch and cannot fire the same second.
trail_cfg = [0, 0, 0, 0]All trailing fields zero — trailing disabled; every = 0 guard prevents division issues
