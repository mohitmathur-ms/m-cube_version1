Part 1: Data Types (The "Nouns" of the System)

All raw market data is converted into these core Nautilus types, defined in Cython for performance.

Data TypeWhat It RepresentsKey Fields
QuoteTickBest bid/ask snapshotbid_price, ask_price, bid_size, ask_size, instrument_id, ts_event
TradeTickA single trade on the exchangeprice, size, aggressor_side, trade_id, instrument_id, ts_event

OrderBookDeltasBatch of book deltaslist[OrderBookDelta]
OrderBookDepth10Top-10 level snapshotbids[10], asks[10], bid_counts, ask_counts
CustomDataAnything else (vendor-specific)data_type, payload
InstrumentStatusIs trading halted/open?status, ts_event
MarkPriceUpdateMark price (derivatives)mark_price, index_price
FundingRateUpdateFunding rate (futures)funding_rate, next_funding_time
Key insight: Both backtest and live use the exact same types. Your strategy code doesn't know if it's running on historical or live data.

Part 2: Data Ingestion (How Raw Data Enters)

2A. Live Data — Adapter Pattern

Directory: nautilus_trader/adapters/

Each exchange has an adapter folder (binance, bybit, databento, interactive_brokers, etc.) with a consistent structure:

adapters/binance/
├── config.py          ← BinanceDataClientConfig (API keys, endpoints)
├── data.py            ← BinanceCommonDataClient (extends LiveMarketDataClient)
├── factories.py       ← BinanceLiveDataClientFactory
├── common/
│   └── schemas/       ← msgspec JSON decoders for raw API messages
└── ...

Flow (using Binance as example):

Binance WebSocket sends raw JSON bytes
  → _handle_ws_message(raw: bytes)
    → msgspec.Decoder parses JSON into schema objects
      → Route by stream name to handler:
          _handle_trade()       → TradeTick
          _handle_book_ticker() → QuoteTick
          _handle_kline()       → Bar
          _handle_book_diff()   → OrderBookDeltas
        → _handle_data(data)
          → msgbus.send("DataEngine.process", data)

Key files:

  • nautilus_trader/adapters/binance/data.py — Binance data client
  • nautilus_trader/live/data_client.py — LiveMarketDataClient base class (async)
  • nautilus_trader/data/client.pyx — MarketDataClient base class (Cython)

Part 3: Data Processing (The DataEngine)

The DataEngine is the central hub. All data passes through it.

File: nautilus_trader/data/engine.pyx (Cython, ~170KB)

Config: nautilus_trader/data/config.py — DataEngineConfig

What the DataEngine does:

  1. Receives data via message bus endpoint "DataEngine.process"
  2. Dispatches by type to specific handlers:
_handle_data(data)
  ├── QuoteTick       → _handle_quote_tick()
  ├── TradeTick       → _handle_trade_tick()
  ├── Bar             → _handle_bar()
  ├── OrderBookDelta  → _handle_order_book_delta()
  ├── MarkPriceUpdate → _handle_mark_price()
  └── CustomData      → _handle_custom_data()
  1. Each handler does 4 things:
    • Updates the Cache (cache.add_quote_tick(tick))
    • Updates Synthetic Instruments (if any)
    • Publishes to Message Bus with a topic like "data.quotes.BINANCE.ETHUSDT"
    • Writes to Catalog for persistence (if configured)

Bar Aggregation (Ticks → Bars)

File: nautilus_trader/data/aggregation.pyx

When a strategy subscribes to bars, the DataEngine creates an aggregator:

Raw Ticks arrive
  → BarAggregator.handle_trade_tick() / handle_quote_tick()
    → BarBuilder updates OHLCV
      → When threshold reached (time/volume/tick count):
        → build() creates a Bar object
          → Published to message bus

Aggregator types:

  • TimeBarAggregator — e.g., 1-minute, 5-minute bars
  • VolumeBarAggregator — bar closes after X volume
  • TickBarAggregator — bar closes after N ticks
  • ValueBarAggregator — bar closes after X notional value
  • RenkoBarAggregator — bar closes after fixed price movement

Order Book Delta Buffering

if config.buffer_deltas:
    # Collect deltas until F_LAST flag
    buffer.append(delta)
    if delta.flags & RecordFlag.F_LAST:
        publish(OrderBookDeltas(buffer))
        buffer.clear()

Part 4: Message Bus (The Nervous System)

File: nautilus_trader/common/component.pyx (line 2178+)

Topics: nautilus_trader/common/data_topics.pyx

The MessageBus is a publish/subscribe system that routes data between components.

Topic naming convention:

data.quotes.BINANCE.ETHUSDT              ← QuoteTick for ETH/USDT on Binance
data.trades.BINANCE.ETHUSDT              ← TradeTick
data.bars.BINANCE.ETHUSDT-1-MINUTE-LAST  ← 1-min Bar
data.deltas.BINANCE.BTCUSDT              ← OrderBookDeltas
data.status.BINANCE.ETHUSDT              ← InstrumentStatus

How it works:

  1. DataEngine publishes data to a topic
  2. Strategies/actors that subscribed to that topic get called automatically
  3. Wildcard subscriptions supported (* = any chars)

Part 5: Cache (In-Memory State)

File: nautilus_trader/cache/cache.pyx (~193KB)

The Cache stores the latest state of everything:

Cache
├── Instruments     dict[InstrumentId, Instrument]
├── Latest Quotes   dict[InstrumentId, QuoteTick]
├── Latest Trades   dict[InstrumentId, TradeTick]
├── Bar History     dict[BarType, deque[Bar]]  (configurable depth)
├── Order Books     dict[InstrumentId, OrderBook]
├── Mark Prices     dict[InstrumentId, MarkPriceUpdate]
├── Funding Rates   dict[InstrumentId, FundingRateUpdate]
├── Orders          dict[ClientOrderId, Order]
└── Positions       dict[PositionId, Position]

Strategies access cache via: self.cache.quote_tick(instrument_id), self.cache.bar(bar_type), etc.

Part 6: Data Consumption (How Strategies Get Data)

Step 1: Strategy subscribes in on_start()

File: nautilus_trader/common/actor.pyx (subscription methods)

class MyStrategy(Strategy):
    def on_start(self):
        self.subscribe_bars(self.bar_type)          # Subscribe to bars
        self.subscribe_quote_ticks(instrument_id)    # Subscribe to quotes
        self.subscribe_trade_ticks(instrument_id)    # Subscribe to trades

        # Register indicators to auto-update
        self.register_indicator_for_bars(self.bar_type, self.ema_fast)
        self.register_indicator_for_bars(self.bar_type, self.ema_slow)

Step 2: Data arrives via handler chain

DataEngine publishes Bar to message bus
  → Actor.handle_bar(bar)                    # Internal handler
    → Update all registered indicators       # EMA, RSI, etc. auto-updated
    → Call self.on_bar(bar)                   # YOUR strategy code runs

Step 3: Strategy implements on_bar(), on_quote_tick(), etc.

def on_bar(self, bar: Bar):
    if self.ema_fast.value > self.ema_slow.value:
        self.buy(bar.bar_type.instrument_id, quantity)

Example strategy: nautilus_trader/examples/strategies/ema_cross.py

Indicators

Directory: nautilus_trader/indicators/

  • base.pyx — Indicator base class
  • averages.pyx — EMA, SMA, etc.
  • momentum.pyx — RSI, MACD, etc.
  • trend.pyx — ADX, Aroon, etc.
  • volatility.pyx — ATR, Bollinger Bands, etc.
How it works: Indicators have handle_bar(), handle_quote_tick(), handle_trade_tick() methods. When registered via register_indicator_for_bars(), they update automatically before on_bar() is called.

Part 9: Complete End-to-End Example

Scenario: A live strategy receiving 1-minute bars from Binance

1. CONFIG
   TradingNodeConfig specifies BinanceDataClientConfig with API keys

2. STARTUP
   TradingNode → NautilusKernel → LiveDataEngine
   BinanceLiveDataClientFactory creates BinanceCommonDataClient

3. SUBSCRIPTION
   Strategy.on_start() calls self.subscribe_bars(BarType("ETHUSDT.BINANCE-1-MINUTE-LAST"))
   → Actor registers handle_bar on message bus topic
   → SubscribeBars command sent to DataEngine
   → DataEngine creates TimeBarAggregator
   → Client subscribes to Binance WebSocket kline stream

4. DATA ARRIVES
   Binance WebSocket sends kline JSON
   → BinanceCommonDataClient._handle_ws_message(bytes)
   → msgspec decodes → _handle_kline() → creates Bar
   → _handle_data(bar) → msgbus.send("DataEngine.process", bar)

5. PROCESSING
   DataEngine._handle_bar(bar)
   → cache.add_bar(bar)
   → msgbus.publish("data.bars.BINANCE.ETHUSDT-1-MINUTE-LAST", bar)
   → catalog.write_data([bar])  (if persistence configured)

6. CONSUMPTION
   Actor.handle_bar(bar) triggered by message bus
   → Updates registered indicators (EMA, RSI, etc.)
   → Calls Strategy.on_bar(bar)
   → Strategy logic runs, potentially submitting orders
